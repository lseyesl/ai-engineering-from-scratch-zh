# 多模态智能体与 Computer Use

> VLM 可以看和说。智能体 VLM（Agentic VLM）可以行动。Computer Use（Anthropic，2024 年 10 月）展示了当 VLM 经过训练后可以操作计算机——移动光标、点击按钮、填写表单、在终端中输入命令——时会怎样。这需要 VLM 不仅理解屏幕截图，还要预测下一个操作（动作），模拟用户输入（键盘/鼠标），并在无结构的 GUI 中自主导航。本课程阅读计算机智能体 VLM 的设计——屏幕截图作为输入、动作预测作为输出、状态机交互作为循环——并构建一个为计算机智能体行动的玩具 VLM。

**类型：** 构建
**语言：** Python（标准库，基于网格的计算机智能体）
**前置要求：** Phase 12 · 21（VLA），Phase 20（智能体）
**时间：** ~180 分钟

## 学习目标

- 将计算机交互问题帧化为 VLM 的观察-行动循环：screen → VLM → action → environment → next screen。
- 为计算机智能体实现动作空间：鼠标点击（坐标）、键盘输入（文本命令）、滚动、等待。
- 构建使用状态机持续采取行动的智能体 VLM：屏幕截图、解析、生成动作、执行、检测指令完成状态。
- 比较 Computer Use 的方法与 UI-TARS（字节跳动，2025 年 1 月）和 CogAgent（智谱，2024 年）的方法。

## 问题

VLM 可以理解屏幕截图。它们可以读取按钮、查找图标和解析文本。但它们如何将理解转化为点击、输入和滚动等动作呢？

Automated GUI 操作面临着来自软件测试的标准限制：

- 屏幕作为状态：一个 Web 页面是一个复杂的状态机，其中交互产生新屏幕。智能体必须推断下一步应该发生什么。
- 动作空间：点击（x，y）、键盘输入（文本）、滚动（方向）、等待。动作必须精确（100 像素的错位会点击错误的按钮）。
- 任务描述：用户以自然语言描述他们想要的内容："预订一张从纽约到伦敦的机票"。智能体必须将其分解为动作序列。

## 概念

### 屏幕截图到动作

在任何时刻 t，环境是一个 GUI 屏幕截图。VLM 将其编码为 patch token。文本 token（任务描述、屏幕历史、动作历史）与 patch token 连接。VLM 输出一个动作 token。

动作空间（Computer Use v1，2024 年 10 月）：

- 鼠标点击（坐标 x，y）。坐标被 token 化为整数，通常相对于屏幕尺寸（例如 1280x1024 尺度下的 x，y）。
- 键盘输入（文本字符串）。文本被 BPE token 化。模型输出应输入的字符串。
- 滚动（方向："向上"、"向下"、"向左"、"向右"）。
- 等待（步数）。当模型不确定时，它应该等待并重新捕获屏幕截图。
- 终止（停止信号）。任务完成。

### 计算机智能体作为一个状态机

计算机智能体循环：

1. 观察：捕获屏幕截图（S_t）。
2. 编码：VLM 将 S_t 编码为 patch token，与任务描述和之前的动作历史连接。
3. 动作预测：VLM 预测动作 a_t = f(S_t，task，history)。
4. 执行：动作在计算环境中播放（点击此处、在此输入、向下滚动）。
5. 下一个状态：环境转换到 S_{t+1}。重复，直到模型输出终止。

循环是自我纠正的：如果动作无效（点击无操作区域），智能体应推断出发生了错误并重试。

### 计算机视觉的挑战

- 分辨率：现代显示器（1920x1080、3840x2160 等）在 VLM token 中产生巨大的表示。（1920x1080）/（14px patch 大小）= 139x77 = 10,703 个 patch token。这比典型的 VLM 上下文（4096 个 token）还要多。需要对屏幕截图进行下采样。

Computer Use 使用一种将屏幕截图无损压缩为 patch 的双编码器：一个高分辨率编码器（用于 OCR）和一个低分辨率编码器（用于布局）。

- 延迟：每个动作周期都需要 VLM 推理（通常约 500ms）加上屏幕截图捕获/动作执行（通常约 200ms）。一个 10 个步骤的任务约需 7 秒。在 UI-TARS 中，这通过模型蒸馏和改进的注意力掩码进一步优化。

- 屏幕变化检测：屏幕可能根本不变（模型点击空区域并等待）。智能体必须预测变化的存在。

- 错误恢复：如果一个动作抛出错误（弹出"无效输入"），智能体必须解释错误消息并调整动作。

### UI-TARS（字节跳动，2025 年 1 月）

UI-TARS 于 2025 年 1 月发布。它是计算机任务上最好的开源智能体 VLM。关键设计决策：

- 屏幕截图补丁：以固定分辨率（通常是 1280x1024）将屏幕截图分为 14x14 像素的补丁。每个补丁是 1 个 token。将 (x, y) 映射到一个补丁索引。
- 预测动作：模型输出动作类型（点击、输入、滚动、等待）+ 动作参数（补丁索引、文本）。所以它是在预测 (action_type，patch_index，[text]) 三元组。
- 训练：在 GUI 操作数据上进行监督训练（例如，来自 Mind2Web 的 3 万个 Web 任务 + 2 万个合成任务）。通过拒绝采样和强化学习进一步微调。

在 Design2Code 和 Screen2Words 上，UI-TARS 比 Computer Use 更好，因为 UI-TARS 是针对未见的 Web 页面进行训练和评估的。

### CogAgent（智谱，2024 年 12 月）

CogAgent 的架构类似，但具有视觉编码器中屏幕截图的双编码器：一个用于文本 OCR 的高分辨率交叉注意力模块和一个用于布局的低分辨率 ViT。CogAgent 是用于智能体 VLM 的编码器。它有一个带有所见和机器状态之间显式交叉注意力的 VLM。

## 使用它

`code/main.py` 构建了一个基于网格的计算机智能体：

- 环境：一个 5x5 网格。每个补丁是屏幕上像素颜色的表示。
- 动作：点击（网格行，列）、输入（文本字符串）、滚动（方向）、等待（N 步）。
- VLM 智能体：ViT（将网格编码为 5x5=25 个 token）+ 文本编码 + 多模态 Transformer → 动作 token。
- 任务："点击左上角，然后输入'你好'，然后点击右下角。" 使用状态机：开始 → 等待到达左上角 → 点击 → 输入 → 等待 → 点击右下角 → 完成。
- 训练：使用 20 个任务合成数据集上的行为克隆。

该智能体通过逐步推理正确执行任务。

## 练习

1. Computer Use 使用 1920x1080 的屏幕截图。每 14px 补丁是一个 token。一张屏幕截图变成多少个 token？有 32K 的上下文限制，这张屏幕截图是否适用？计算 token 压缩率以达到 4K 上下文。

2. UI-TARS 将 (x，y) 映射到一个补丁索引。如果按钮位于像素 (100，250) 并且补丁大小为 14px，补丁索引是什么？给定 1280x1024 的屏幕分辨率。

3. WebArena 需要 20 个步骤来完成"预订机票"任务。在 500ms 的推理时间下，端到端的响应时间是多少？讨论延迟瓶颈。

4. 该智能体陷入了一个循环：它点击一个模式窗口，但错误消息仍然存在。设计一个状态机来检测和恢复错误。

5. Computer Use 在推理时需要动作历史。在 50 个步骤的任务中，动作历史是多长？随着历史增长，模型如何记忆动作历史？描述一个在 200 个步骤后缓解遗忘的架构。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 计算机智能体 | "GUI 代理" | 理解屏幕截图并预测鼠标/键盘动作以在软件界面上导航的 VLM |
| 动作空间 | "鼠标/键盘元组" | 一组动作：点击（坐标）、输入（文本）、滚动、等待、终止 |
| 观察-行动循环 | "截图 → 预测 → 执行" | 智能体捕获屏幕截图、使用 VLM 预测动作、将其应用于环境的循环 |
| 补丁索引 | "UI-TARS 风格" | 将屏幕截图划分为一个网格；动作坐标映射到一个网格索引 |
| 错误恢复 | "异常处理" | 模型检测到动作无效并尝试不同动作的状态机循环 |
| 双编码器 | "高分辨率 + 低分辨率" | 两个编码器：一个用于 OCR 的高分辨率（文本检测），一个用于布局的低分辨率 |
