# 多模态 RAG：跨模态检索与生成

> RAG 使 LLM 能够从文本语料库中检索信息。但当该语料库包含文本、图像、表格和图表，且用户的问题是跨模态时（"显示价格高于 x 的表格"、"描述图 3 的趋势"），会发生什么？多模态 RAG（Multi-modal RAG）系统结合了文本检索器（稀疏、密集）和视觉检索器（CLIP、ColPali）来获取相关片段，然后将它们输入到多模态 LLM 进行生成。本课程阅读多模态 RAG 系统的架构——多模态分块、混合检索、重排序、生成——并构建一个端到端的多模态 RAG 流水线。

**类型：** 构建
**语言：** Python（标准库，多模态 RAG 流水线）
**前置要求：** Phase 12 · 23（ColPali），Phase 10（RAG）
**时间：** ~180 分钟

## 学习目标

- 设计多模态分块策略：给定一个包含文本、图像和表格的文档，定义如何将其划分为片段，使每个片段是自包含的多模态单元。
- 实现混合检索：文本检索器（BM25、密集通道）和视觉检索器（CLIP、ColPali）通过加权融合分数。
- 实现多模态重排序：使用交叉编码器对检索结果进行重排序，并实现文本和视觉模型分数的融合。
- 为多模态 RAG 输出实现归因（引用）。

## 问题

标准 RAG 系统（Phase 10）将文档分块、嵌入文本，并将查询映射到文本。但在多模态 RAG 中，用户询问的是视觉信息。一个好的 RAG 系统必须：

1. 检索包含图像的片段，即使查询只是文本（"显示相关图表"）。
2. 检索仅对多模态信息有意义的片段（"按价格排序的表格"）。
3. 向 LLM 呈现多模态输入：不仅是文本片段，还有图像片段，VLM 可以执行推理。

## 概念

### 多模态分块

标准文本分块（按 token 数量、按句子或按段落）在多模态文档中失败。段落可以从一个图表突然跳到一段文本。多模态分块必须尊重文档的视觉和语义边界。

多模态分块的典型方法：

1. Layout 分块（mPLUG-DocOwl 风格）：根据布局结构划分页面 → 标题块、正文块、表格块、图像块。
2. 语义分块：检测语义边界（章节边界、主题转换），并将相关图像分组到同一块中。
3. 重叠分块：在更细的粒度上进行分块，并允许重叠。
4. 多模态 token 流（ColPali 风格）：不进行分块。将完整页面保留为图像 patch。对于检索，使用后期交互从 patch 中获取答案。这是最有效的方法，但需要 VLM 生成器。

对于纯文本分块，每个片段是文本。对于多模态分块，每个片段是文本和图像的元组：`(text_chunk, [image1, image2, table1])`。每个片段以文本形式描述图像内容，并包含图像的 token 化版本。

### 混合检索

混合检索结合了文本检索和视觉检索的分数。

+ 文本检索器：BM25（稀疏）或 dense passage（DPR、BGE）。给定一个查询 q，文本检索器对文档片段进行排名：`score_text(d) = BM25(q, d)`。
+ 视觉检索器：CLIP（对比语言-图像预训练）。给定一个查询 q，视觉检索器对与 q 最相关的图像进行排名：`score_vis(q, images(d)) = avg_{i in images(d)} CLIP(q, i)`。
+ 混合分数：`score_hybrid(d) = α · score_text(d) + (1-α) · score_vis(d)`，其中 α 通过验证集上的网格搜索进行调整。

近年来，ColPali 风格的检索开始取代视觉路径中的 CLIP；ColPali 是专门针对文档检索进行微调的。

### 多模态重排序

重排序器使用交叉编码器对检索结果重新排序。在多模态设置中，重排序器是：

+ 交叉编码器文本：text encoder (q, text_chunk) → 相关性分数。
+ 交叉编码器视觉：VLM (q, text_chunk, image) → 相关性分数；可能是描述图像文本的相关性。
+ 融合重排序：`rerank_score(d) = β · rerank_text(d) + (1-β) · rerank_vis(d)`。

多模态重排序的计算量很大——VLM 重排序器对于每个查询-文档对都是 O(L) 的。它通常只在检索之后对 top-k 个片段使用。

多模态 RAG 中归因很重要：答案应该引用生成答案的图像或文本片段。对于 VLM，这意味着生成类似“根据 2023 年的表格（图 2），收入增长了 20%”的文本。通过来自检索步骤的分块边界来实现归因。

### 多模态 RAG 的流水线

端到端的多模态 RAG 流水线：

1. 文档：解析并分块 → 片段（`{text_chunk, images, tables}`）。
2. 索引：文本检索器索引文本通道。CLIP / ColPali 索引视觉通道。
3. 检索：给定查询 q，计算文本检索器分数和视觉检索器分数。融合 → top-k 片段。
4. 重排序：在 top-k 片段上运行交叉编码器（文本 + 视觉）。重新排名 top-k。
5. 生成：将 top-k 片段和查询输入到 VLM（LLaVA-NeXT、Qwen-VL）。生成带有归因的答案。

### 评估

多模态 RAG 的标准基线：

- **MMQA**（多模态问答）：文本 + 图像的问题。指标：准确率（精确匹配）。
- **MuSiQue**（多步多模态 QA）：需要多个推理步骤。指标：F1。
- **WebQA**：来自网络的文本 + 图像。指标：准确率。
- **MultiModal RAG Benchmark**（2024，Sardar 等人）：为测试多模态检索 + 归因而设计的新基准。指标：Retrieval Recall@k、Generation Accuracy、Attribution Precision。

## 使用它

`code/main.py` 构建了一个端到端的多模态 RAG 流水线：

- 文档库：3 个文档，每个包含文本和表格（合成数据）。
- 文本检索器：BM25。
- 视觉检索器：使用"图像 caption"嵌入的简单 CLIP 模拟。为每个片段计算余弦相似度。
- 融合：α=0.5。
- 重排序：使用文本交叉编码器（模拟，但数学正确）。
- 生成：top-1 片段 + 查询 → 玩具 VLM 输出。
- 输出：带有归因的结果和正确的片段引用。

2. 实现混合检索。给定一个查询"2023 年收入最高的公司"，使用 BM25（文本检索器）返回文档 D1、D2、D3、D4（分数：[0.8, 0.6, 0.4, 0.2]），而视觉检索器返回分数 [0.3, 0.1, 0.9, 0.5]。混合因子 α=0.5，新的排名是怎样的？

3. 多模态重排序使用交叉编码器对 top-k 片段重新排序。如果 k=10（检索器输出 10 个结果），重排序的延迟是多少？与用延迟换取归因质量相比，有什么理由？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 多模态分块 | "布局感知分块" | 将文档划分为片段，使每个片段是自包含的多模态单元（文本 + 图像） |
| 混合检索 | "文本 + 视觉分数融合" | 使用 α 权重将来自 BM25（文本）和 CLIP（视觉）的检索分数组合起来 |
| 多模态重排序 | "视觉 + 文本重排序" | 使用文本和视觉交叉编码器对 top-k 检索结果重新排名 |
| 归因 | "引用图像" | 生成器输出引用生成答案所使用的图像或表格 |
| 多模态 token | "patch token" | 文档片段的视觉表示保留为 token，与文本 token 一起送入 VLM |
| 片段 | "多模态单元" | (文本_chunk，[images]，[tables]) 的元组，代表文档中的一个语义单元 |
