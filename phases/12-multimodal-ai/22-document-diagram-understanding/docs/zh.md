# 文档与图表理解：OCR、布局分析、图表推理

> VLM 视觉问答无法处理文档。文档不是自然图像：它们具有分层结构（段落、章节）、密集的文字内容（字体从 6pt 到 24pt）以及复合布局（页眉、页脚、图表、表格）。文档 VLM 需要专门的编码器来解析文档中的文本、布局和图形图表。本课程阅读文档 VLM 架构——Donut、DocTR、mPLUG-DocOwl 和 TextMonkey——并构建一个将非结构化文档页面解析为结构化布局树（段落、标题、表格）的编码器。

**类型：** 构建
**语言：** Python（标准库，文档布局解析器）
**前置要求：** Phase 12 · 05（LLaVA）
**时间：** ~180 分钟

## 学习目标

- 描述文档 VLM（Donut、mPLUG-DocOwl）的关键架构组件，并将其与自然图像 VLM 进行对比。
- 为一个视觉富文档实现版面分析编码器（OCR + 布局解析 + 图表理解）。
- 构建基于 ViT 的图表标题生成器，对合成图表数据进行训练。
- 对比用于文档解码的 OCR + token 合并策略与端到端策略。

## 问题

自然图像 VLM 是猫和狗识别器。但企业工作流程需要：从扫描的 PDF 中提取文本、理解发票、解读图表、回答有关组织结构图或流程图的问题。

文档具有视觉上密集的结构，边界重叠的文本（标题被正文包围），微小的字体（6pt，在 ViT patch 尺寸 14px 下模糊）。典型 ViT 无法以足够高的分辨率解析文本以进行 OCR。文档 VLM 必须具有：

1. 高分辨率编码器（双编码器：一个用于 OCR 的高分辨率编码器，一个用于布局和图表理解的低分辨率编码器）。
2. 布局感知位置编码（文档坐标：捕获段落、页眉、页脚、表格的绝对和相对位置）。
3. OCR 层（或集成 OCR 管道的投影器）。
4. 图表推理（图表中的文本轴、图例、数据点）。

## 概念

### 高分辨率双编码器

OCR 需要高分辨率（通常 >1000px）。ViT 在 14px 下每 1024px 图像就会产生 73x73 = 5329 个 token。对于文本密集的文档，这很快就会溢出上下文。

Donut（2022 年）使用 Swin Transformer 作为编码器。Swin 具有分层特征图（从高分辨率到低分辨率），使其能够处理 1024px 输入，同时在较小规模上保持 4x4 patch 的感知野。

mPLUG-DocOwl（2023 年）更进一步，使用双编码器：一个高分辨率编码器（ViT-L/14，处理图像的高分辨率图块）和一个低分辨率编码器（同一 ViT，处理整个图像）。在输入到 LLM 之前，高分辨率编码器输出通过交叉注意力与低分辨率特征融合。

因此，文档 VLM 处理 384px 图像用于布局，每个 384px 图块用于 OCR。

### 布局分析

布局分析将页面分解为结构块：页眉、页脚、正文、标题、图片、表格、列表。常见的方法是使用 Mask R-CNN 或 DETR 训练布局检测器。

LayoutLMv3（2022 年）是布局分析的标准编码器。它将文本 token（来自 OCR）与布局坐标（框坐标）相结合。输出是每个 token 的布局感知向量。这在 LLM 的推理过程中使用。

在文档 VLM（mPLUG-DocOwl 等）中，布局分析是上游的：编码器将图像处理成"图块 token"，其中包含有关该图块是否包含文本、数字或图表类型的信息。（这通常需要额外的训练数据。）

### OCR 与布局的 token 合并

一旦你有了 OCR 文本和布局坐标，它们就必须合并到一个序列中。三种常见方法：

- Donut：编码器将图像 patch 馈入 Transformer。解码器以自回归方式预测文本 token。无需显式 OCR——模型直接从图像预测文本 token。在需要高精度的数值文档上表现不佳。
- mPLUG-DocOwl 方式：带有 OCR 注入。图像被馈入双编码器。OCR 文本（来自 EasyOCR 或 PaddleOCR）通过 OCR 位置编码器嵌入。然后将 OCR token 与图像 token 合并。更好的精度。
- Pix2Struct（Google，2022）：编码器将图像的渲染 HTML 解析为 token。模型学习将图像 patch 映射到 HTML token。对于数字密集的文档很有效。

### 图表推理

图表在视觉上是复杂的，有轴刻度、图例、网格线和重叠的数据系列。图表 VLM 必须通过以下方式对其推理：

- 理解轴：识别 x 轴和 y 轴、刻度标签、轴标签。
- 理解图例：将文本标签与可视化样式（颜色、线条、标记）相关联。
- 解读数据：读取数据点以回答问题（"2023 年 5 月有多少销售额？"）。

完全通过端到端 VLM 实现这一点很困难；它们需要布局和绘图坐标的双重编码。许多生产系统在 LLM 之上使用专门的图表解析器（例如 ChartQA 中的 ChartReader 单编码器模型）。

### TextMonkey（2024 年 8 月）

TextMonkey 是 2024 年末在文档理解方面的一个进展。它使用：

- 用于高密度文本的 ViT 窗口注意力。
- 用于文档结构的布局感知位置编码（LAPE）。
- 共享投影器的交叉模态对齐。
- 用于强化 OCR 精度的学习 token 合并（LTM）。

TextMonkey 在文档 VLM 基准上表现良好，并已被生产系统部署（如阿里巴巴的文档 AI）。

## 使用它

`code/main.py` 构建了一个文档布局解析器：

- 加载一个文档 PDF（扫描版本）。
- 使用训练好的掩码 R-CNN（通过加载权重）执行布局分析。输出边界框：title (0.93)、paragraph (0.87)、table (0.91)。
- 根据布局分块，使用 Tesseract OCR 提取文本。
- 构建一个布局树：文档 → {标题：""，段落：[...]，表格：{行：3，列：4}}。
- 将布局树送入玩具 VLM（LLM 投影器），输出布局的描述文本。

## 练习

1. 在 DocVQA 上使用 Donut 与 mPLUG-DocOwl。Donut 在没有显式 OCR 的情况下如何预测文本？为什么它在数字密集的文档上失败？

2. 使用 Swin Transformer 作为文档编码器。描述滑动窗口注意力如何使 ViT 在高分辨率文档图像上有效。（提示：窗口大小 M，图像大小 H×W，复杂度 O(HW/ M²)）。

3. Pix2Struct 将文档图像的渲染 HTML 解析为 token 序列。提出一个像素级推理任务（例如图表推理），其中 Pix2Struct 优于普通 VLM。

4. TextMonkey 的 LTM（学习 token 合并）通过学习合并密集文本区域中的冗余 token 来减少 token 数量。解释为什么文档图像比自然图像从 LTM 中受益更多。

5. 布局分析检测到一个页眉区域、一个正文区域和一个表格。使用 mPLUG-DocOwl 方式设计一个序列，其中 OCR token 和图像 patch 以正确的顺序合并。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 文档 VLM | "文档理解" | 用于文本密集的文档图像的专用 VLM，具有高分辨率编码器和布局感知位置编码 |
| 双编码器 | "高分辨率 + 低分辨率" | 两个视觉编码器：一个用于 OCR 的高分辨率（每个 token 一个图块），一个用于布局的低分辨率 |
| 布局分析 | "页面分解" | 将页面分为页眉、页脚、正文、表格、图片 |
| OCR 注入 | "OCR token 合并" | 将外部 OCR 文本（文本和边界框）与 LLM 中的图像 patch token 合并 |
| 图表推理 | "图表理解" | 读取数据点、轴、图例以回答有关图表的问题 |
| 学习 token 合并 | "LTM" | TextMonkey 的一种方法，合并冗余的视觉 token，用于密集文本的文档图像 |
| 布局感知位置编码 | "LAPE" | 一种位置编码，对布局结构中标记的 x、y、宽度、高度和页面相对位置进行编码 |
