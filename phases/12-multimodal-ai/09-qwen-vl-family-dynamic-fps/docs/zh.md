# Qwen-VL 家族与动态 FPS 视频

> Qwen-VL 家族——Qwen-VL（2023 年）、Qwen2-VL（2024 年）、Qwen2.5-VL（2025 年）、Qwen3-VL（2025 年）——是 2026 年最有影响力的开源视觉语言模型谱系。每一代都做了一个决定性的架构选择，其余的开源生态系统在 12 个月内就复制了：通过 M-RoPE 实现原生动态分辨率、带绝对时间对齐的动态 FPS 采样、ViT 中的窗口注意力以及结构化代理输出格式。到 Qwen3-VL，配方已经稳定下来：具有原生宽高比输入的 2D-RoPE-ViT 编码器、将信号投影到大型 Qwen3 语言基础的 MLP 投影器，以及强调 OCR、接地和代理行为作为一等公民的训练阶段。本课程按时间顺序阅读这个家族，以便你理解为什么每个旋钮都在现在的位置上。

**类型：** 学习
**语言：** Python（标准库，M-RoPE 编码器 + 动态 FPS 采样器）
**前置要求：** Phase 12 · 06（patch-n'-pack）
**时间：** ~120 分钟

## 学习目标

- 计算 M-RoPE 的三轴旋转（时间、高度、宽度）并解释为什么三者都是必需的。
- 为视频选择动态 FPS 采样策略，并推理每秒 token 数量与事件检测准确率之间的关系。
- 按顺序说出 Qwen-VL 的四代升级以及每代启用了什么。
- 编写 Qwen2.5-VL 风格的 JSON 代理输出格式，并从 VLM 响应中解析结构化工具调用。

## 问题

Qwen-VL 于 2023 年 8 月发布，直接回应了 LLaVA-1.5 和 BLIP-2。Qwen 团队针对的差距有三方面：分辨率、视频和结构化输出。

分辨率：LLaVA-1.5 运行在 336x336。对照片来说可以，但对于中文发票或密集的电子表格截图无用。Qwen-VL 的第一个创新是 448x448 和基于边界框的输出，让模型可以指向事物。

视频：Video-LLaMA 堆叠了每帧编码器并将它们馈入 LLM。它适用于短视频片段，不适用于时间轴就是信号的多分钟视频。Qwen 团队想要一个理解时间的单一编码器。

结构化输出：LLaVA 输出自由格式文本。代理需要 JSON。Qwen-VL 在显式的 JSON 输出格式上训练，包括边界框坐标作为文本。

每一代 Qwen-VL 都在这三个轴之一上进行扩展。

## 概念

### Qwen-VL（2023 年 8 月）

第一代：OpenCLIP ViT-bigG/14 作为编码器（2.5B 参数），兼容 LLaMA 的 Q-Former（1 步，256 个查询），Qwen-7B 基础。贡献：

- 448x448 分辨率（当时开源 VLM 的 SOTA）。
- 接地：在带有显式坐标 token 输出的图文对上训练。"猫在 <box>(112, 204), (280, 344)</box>"。
- 从一开始就进行中英双语多语言训练。

当时的基准：在英文上与 GPT-4V 竞争，在中文上占主导地位。接地监督才是真正的头条新闻。

### Qwen2-VL（2024 年 9 月）——M-RoPE 和原生分辨率

Qwen2-VL 用原生动态分辨率 ViT 编码器替换了固定分辨率 + Q-Former 堆栈。关键变化：

- 原生动态分辨率。ViT 接受任何能被 28 整除的 HxW（patch 14，2x 空间合并）。一张 1120x672 的图像（40x24 合并 patch）产生 960 个视觉 token。无需调整大小、无平铺、无缩略图。
- M-RoPE（多模态 RoPE）。每个 token 携带一个 3D 位置（t，h，w）而不是 1D。对于图像 t=0，对于视频 t = 帧索引。RoPE 按每个轴的频率旋转 query/key 向量。没有位置嵌入表。
- MLP 投影器。放弃 Q-Former；在合并后的 patch token 上使用 2 层 MLP。
- 视频动态 FPS。默认以 1-2 FPS 采样视频，但模型接受任意帧数。

结果：Qwen2-VL-7B 在几个多模态基准上与 GPT-4o 匹配，并在 DocVQA 上击败了它（94.5 vs 88.4）。架构变化是决定性的举措。

### Qwen2.5-VL（2025 年 2 月）——动态 FPS + 绝对时间

Qwen2.5-VL 的重大转变是视频。动态 FPS 不仅仅是"需要时采样更多帧"。论文形式化了：

- 绝对时间 token。不是位置索引（帧 0，1，2...），而是使用实际时间戳。"在 0:04，猫跳了起来。"模型看到与帧 token 交错的 `<time>0.04</time>` token。
- 动态 FPS。慢速素材以 1 FPS 采样，动作场景以 4+ FPS 采样。用户或训练器选择；M-RoPE 自适应。
- ViT 中的窗口注意力。空间注意力控制在窗口内（局部）以提高吞吐量；每几层有一次全局注意力。
- 显式 JSON 输出格式。在工具调用数据上训练：`{"tool": "click", "coords": [380, 220]}`。开箱即用的代理就绪。
- MRoPE-v2 缩放。位置随最大输入大小缩放，这样 10 分钟的视频不会超出频率范围。

基准：Qwen2.5-VL-72B 在大多数视频基准上击败 GPT-4o，在文档上匹配 Gemini 2.0，并设定了 GUI 接地的开源模型 SOTA（ScreenSpot：84% 准确率 vs GPT-4o 的 38%）。

### Qwen3-VL（2025 年 11 月）

Qwen3-VL 是一个增量升级，巩固而非重新发明：更大的 LLM 骨干网（Qwen3-72B）、扩展的训练数据、改进的 OCR、通过 Qwen3"思考模式"实现更强的推理。ViT 和 M-RoPE 保持不变。论文侧重于数据和训练改进，而非架构。

谱系要点：到 2025 年，Qwen-VL 架构已经稳定。后续世代扩展计算和数据，而非原语。

### M-RoPE 数学

经典的 RoPE 使用配对的坐标，将维度为 `d` 的查询 `q` 按位置 `m` 旋转：

```
q_rot[2i]   = q[2i]   * cos(m * theta_i) - q[2i+1] * sin(m * theta_i)
q_rot[2i+1] = q[2i]   * sin(m * theta_i) + q[2i+1] * cos(m * theta_i)
theta_i     = 10000^(-2i/d)
```

M-RoPE 将隐藏维度分成三个频带。假设 `d = 96`。分配 32 维给时间，32 给高度，32 给宽度。每个频带按其自身的轴位置旋转。位于 (t=5, h=10, w=20) 的 patch 对其三个频带应用旋转 `R_t(5)`、`R_h(10)`、`R_w(20)`。

文本 token 使用 `t = text_index, h = 0, w = 0`（或归一化的选择），保持兼容性。视频帧使用 `t = frame_time, h = row, w = col`。单张图像使用 `t = 0`。

好处：一个位置编码无需分支代码或不同的位置表即可处理文本、图像和视频。

### 动态 FPS 采样逻辑

给定持续时间为 `T` 秒的视频和目标 token 预算 `B`：

1. 计算你能负担的最大 FPS：`fps_max = B / (T * tokens_per_frame)`。
2. 从 `{1, 2, 4, 8}` 中选择满足 `fps <= fps_max` 的目标 FPS。
3. 如果运动大（光流启发式或显式用户请求），选择更高 FPS。如果运动小，选择更低。
4. 以选定的 FPS 均匀采样；在帧之间插入 `<time>t</time>` token。

Qwen2.5-VL 隐式地训练了这种逻辑；在推理时，用户通过 `fps` 参数控制。60 秒的动作序列在 4 FPS 下，每帧 81 个 token = 19440 个 token，在 32k 上下文中可管理。

### 结构化代理输出

Qwen2.5-VL 的代理训练显式地针对结构化工具调用：

```
{
  "tool": "mouse_click",
  "coords": [1024, 512],
  "button": "left",
  "modifier": null
}
```

解析是确定性的：对模型输出进行 JSON.parse。与自由格式的"click at (1024, 512)"（需要正则表达式和歧义处理）相比。这一转变就是为什么 Qwen2.5-VL 的 ScreenSpot 分数从 Qwen2-VL 的 55% 跃升到 84%。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| M-RoPE | "多模态 RoPE" | 3D 旋转位置嵌入，隐藏维度分时间、高度和宽度频带 |
| 动态 FPS | "智能采样" | 根据运动、时长和 token 预算为每个视频选择的帧采样率 |
| 绝对时间 token | "时间戳 token" | 序列中交错的 `<time>t</time>`，让模型看到实际秒数而非帧索引 |
| 窗口注意力 | "局部注意力" | 限制在小型窗口内的空间自注意力以加快速度；周期性添加全局注意力 |
| 结构化代理输出 | "JSON 模式" | 训练数据监督，教 VLM 输出带有坐标和工具名称的可解析 JSON |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 的每请求控制，限制总像素数从而限制 token 数 |
| 接地 | "指向它" | 输出边界框坐标作为文本 token；自 Qwen-VL v1 起使用 |
