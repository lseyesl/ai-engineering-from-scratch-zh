# 全模态模型：思考者与对讲机架构

> 音频 VLM 将 LLM 视为一个接收音频 token 并输出文本的"思考者"。但在实时语音应用中，比如语音助手，我们需要低延迟且易于停止的交互。思考者与对讲机架构（Zhang 等人，2024 年 9 月）将音频 LLM 拆分为两个部分：一个"思考者"（了解上下文但不输出声音）和一个"对讲机"（输出声音但不推进推理）。这避免了让单一 LLM 同时执行推理和发声，这种方式在文本中表现尚可但在语音中会失败。思考者发出隐藏状态；对讲机从思考者的潜在空间中采样并生成语音。本课程学习思考者与对讲机设计，并从头开始构建一个玩具思考者-对讲机。

**类型：** 构建
**语言：** Python（标准库，思考者-对讲机模拟器）
**前置要求：** Phase 12 · 19（音频全模态）
**时间：** ~180 分钟 📈

## 学习目标

- 在给定单模型推理延迟的情况下，推导语音助手响应时间的理论下限，并展示思考者-对讲机拆分如何绕过这一限制。
- 为"对讲机"设计一个离散语音 tokenizer（VQ-VAE，16kHz 音频 → 50 Hz token），对比 Whisper 风格的连续编码器。
- 实现思考者-对讲机编解码器，包括解耦的注意力掩码（思考者因果掩码，对讲机双向掩码）。
- 比较 Emu3 中 VoiceCaps 风格的全模态音频生成与专有全模态系统。

## 问题

语音助手应提供即时反馈。单一语音 LLM 在响应之前必须完成整个推理链，语言模型生成 ~100 个 token 的停顿对我们来说是无法忍受的。

在文本模型中，首个 token 时间（推理到第一个输出 token 之间的时间）用 100ms 是可行的。但在语音中，需要生成 ~50 个音频 token，耗时 ~1 秒。你在语音中等待他们思考，这感觉像 bug。

## 概念

### 思考者-对讲机拆分

+ 思考者是一个标准的 Transformer 语言模型。它接收音频 token（编码的语音）和文本 token（对话上下文）。它输出**高级语义 token**（不是声音，而是意图）。思考者不发送可听音频。

+ 对讲机是一个较小的 Transformer，专门用于将语义 token 转换为音频 token。它以思考者的隐藏状态为条件，并持续解码直到遇到停止 token。它不推进对话上下文——它只是发出声音。

因此，在实际语音中：
1. 用户结束说话。
2. 思考者接收音频并输出第一个语义 token（= "我懂了"）。
3. 对讲机立即开始生成音频 token（= "嗯嗯" + 部分单词），如"嗯"或 um 等填充词，通常由对讲机而非思考者生成。
4. 思考者继续其推理链，并行输出下一个语义 token。
5. 对讲机将思考者的新 token 流式输出为音频。用户开始听到单词。

用户端感知的延迟从"思考者完成 ~2 秒的推理"降低到"对讲机生成 ~200ms 的音频 token"。这对交互体验至关重要。

### 离散语音 tokenizer

VQ-VAE 用于语音，与图像原理相似。一张 16kHz 的 10 秒录音 → 160,000 个采样点。划分成 20ms 帧（50 Hz）。每帧是一个 80-bin mel 频谱图。编码器输出一个 50 Hz 的向量序列（10 秒 → 500 个帧）。码本将这些 500 帧向量量化成整数。与图像类似。

对于语音，常用的码本大小是 1024（K=1024），而非图像的 8192。语音更不那么富于变化，但流畅性比细节更重要。

### 编解码器：从声音到声音

思考者-对讲机架构本质上是语音到语音的。语音 token 通过编解码器循环，该编解码器包含：

- 语音编码器（将语音转换为语义 token）。
- 思考者（对语义 token 进行推理，输出响应语义 token）。
- 对讲机（将响应语义 token 解码为语音 token）。
- 语音解码器（将语音 token 转换为原始音频）。

这与 TTS（文本转语音）的常见做法不同，后者使用单独的模型进行 ASR、语义理解和合成。这里是语音进入→语音输出，全程端到端。

### 全注意力掩码

根据模态的不同，注意力掩码有三种设计：
+ 对思考者：输入文本 token 和输入语音 token 之间的因果交叉注意力。
+ 对讲机：针对思考者的文本隐藏状态的交叉注意力，以及针对输出音频 token 的因果掩码。
+ 整体：因果序列可以通过时间完成，但思考者和对讲机之间的注意力是并行的（思考者无需等待对讲机完成即可推进推理）。

这对多头注意力掩码而言略复杂，但可以用标准的 `torch.Tensor.masked_fill` 实现。

### Emu3 中的 VoiceCaps

Emu3（课程 12.12）还使用了一种称为 VoiceCaps 的语音 token 化方法。与思考者-对讲机不同，Emu3 的语音 token 化是 VQ 编码器，且所有 token 进入一个单一的、统一的 next-token 预测循环。Emu3 不是思考者-对讲机模型；它只是一个单一的因果语言模型。但它确实生成了音频 token。关键的区别在于，Emu3 的音频生成并不早于文本 token，而是与之交错；在纯语音场景中，这可能会导致首个 token 时间延长。

### 全模态 vs 统一模型

统一模型（Chameleon、Emu3、Show-o）对所有模态使用单一的离散 token 词汇表，但并不能很好地处理低延迟，因为解码器将每种模态视为相等。

全模态模型（思考者-对讲机、SpeechGPT）为交互而设计。思考者部分计算意图，对讲机部分处理即时响应。这是一项架构创新，无法被下一代"更好"的离散 token 化替代。

## 使用它

`code/main.py` 构建了一个思考者-对讲机模拟器：

- 编码器是 ViT（带交叉注意力的语音编码器）。输出 16 个语义 token。
- 思考者是一个小型语言模型（8 层，因果）。以编码器语义 token 为条件，输出响应高级语义 token（输出 32 个整数）。
- 对讲机是一个 4 层 Transformer，因果，以思考者的输出为条件。输出离散音频 token（码本大小为 128）。
- 语音解码器将对讲机的 token 映射回原始音频。

模型在 10 对合成语音-文本对上过拟合。

## 练习

1. 思考者每 100 毫秒产生一个语义 token。对讲机在听到思考者的 token 后每 10 毫秒产生一个音频 token。如果音频响应是 100 个音频 token，用户需要等待多少个思考者 token 才能听到第一个词？假设思考者生成 5 个语义 token 后，对讲机才开始生成。

2. 离散语音 tokenizer（码本 1024，帧率 50 Hz，帧长 20 ms）。10 秒的语音输入变成多少个 token？一个 5 秒的语音输出呢？与视频 token 进行对比。

3. 思考者-对讲机注意力掩码。给定 N 个思考者 token（3 个输入，5 个输出）和 M 个对讲机 token（100 个输出），为完整的跨模态注意力写出完整的注意力掩码矩阵。（提示：思考者必须因果地关注之前的思考者 token。对讲机必须关注之前的所有对讲机 token 和所有思考者 token。）

4. 思考者-对讲机模型如何扩展到视频？提出对讲机应输出视频帧 token 而非仅音频 token 的情况。这对延迟有何影响？

5. 比较思考者-对讲机和语音流式传输（SpeechGPT 方法）。为什么前者更受生产级语音助手的青睐？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 思考者 | "推理 LM" | 接收输入 token，输出语义 token（意图）；不直接输出语音 |
| 对讲机 | "语音 LM" | 以思考者的语义 token 为条件，生成离散音频 token |
| 离散语音 tokenizer | "VQ-VAE 语音" | 将原始音频（16 kHz 采样）转换为 50 Hz VQ token，使用码本大小为 1024 |
| 首个 token 时间 | "感知延迟" | 用户结束通信到对讲机开始输出音频之间的时间 |
| 全模态模型 | "端到端语音" | 语音输入到语音输出，无独立 ASR 或 TTS 组件 |
| 语义 token | "思考者 token" | 捕捉输入含义的高级 token；由思考者产生并由对讲机消费 |
