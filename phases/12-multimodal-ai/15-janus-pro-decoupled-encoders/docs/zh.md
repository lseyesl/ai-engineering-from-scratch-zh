# Janus-Pro：用于 Unified 理解和生成的多模态架构设计

> DeepSeek 的 Janus（2024 年 10 月）和 Janus-Pro（2025 年 1 月）是关于一个简单观察的系列成果：如果你使用相同的视觉编码器来理解和生成，你必须做出妥协。理解编码器是去噪的，并且随着细节的增长而更好；生成编码器必须是离散的，以匹配 Transformer 的词汇表。Janus 将它们分开——一个用于理解的 SigLIP 编码器，一个用于生成的 VQ 编码器——然后在 Transformer 的输入处合并两者。结果是：当你解耦为任务选择正确的编码器时，统一模型可以同时做得更好。Janus-Pro 是 DeepSeek 使用的 7B 统一模型的发布；JanusFlow（2024 年 11 月）将生成路径替换为流匹配。本课程阅读解耦架构并主张为什么理解编码器和生成编码器应保持分离。

**类型：** 学习
**语言：** Python（标准库，编码器比较器 + 跨模态注意力数学）
**前置要求：** Phase 12 · 11（Chameleon），Phase 12 · 14（Show-o）
**时间：** ~120 分钟

## 学习目标

- 论证为什么在统一 VLM 中理解编码器和生成编码器应该保持分离。
- 描述 Janus 的解耦架构：文本、视觉理解、视觉生成路径在输入投影处汇合。
- 比较 Janus-Pro（VQ 离散生成）与 JanusFlow（流匹配连续生成）在质量和推理时间上的差异。
- 说出训练阶段：理解预训练、生成预训练、联合微调。

## 问题

Chameleon、Emu3 和 Show-o 都共享一个用于理解和生成的视觉编码器。Image → VQ → token → 统一的 next-token 预测。简单、统一、优雅。

问题是：理解编码器和生成编码器有不同的要求。

- 对于理解（VQA、描述、OCR）：高分辨率细节很重要。在大规模的噪声图像文本对上训练的 ViT（如 SigLIP、CLIP、InternViT）效果最好。特征是连续的。空间分辨率可以细至 14x14 patches。
- 用于生成（文生图、图像修复）：离散 token 很重要。码本的表示容量定义了图像质量。视觉特征被量化为整数索引，匹配 LLM 的词汇表，并与文本 token 交错。

如果你强制一个编码器扮演两个角色，你会在理解性能（因为离散化丢失了理解所需的细节）或生成性能（因为你使用了理解编码器，它没有针对重建或码本多样性进行优化）上做出妥协。

## 概念

### 解耦架构

Janus 在 Transformer 的输入处将路径分开：

- 文本路径：标准 BPE token → 文本嵌入。
- 视觉理解路径：图像 → SigLIP ViT（336px，patch 大小 14）→ 24x24 个 patch 特征 → MLP 投影 → 图像嵌入。
- 视觉生成路径：图像 → VQ tokenizer（下采样 16x）→ 离散索引 → 嵌入查找 → 图像嵌入。

这三个嵌入序列被拼接成一个单一序列，送入仅解码器的 Transformer。Transformer 不知道模态；它看到的是一个 token 序列，带有确定哪种类型接哪种类型的注意力掩码。

在输出方面，Transformer 对于文本 token 使用 LM 头，对于生成 token 使用 VQ 头。理解 token 作为文本 token 被消耗——它们仅仅是提供文本 LLM 上下文的投影图像特征。

### 两个编码器，一个 LLM

Key insight：你可以在 LLM 内部运行时融合理解图像特征和生成图像特征，但在编码时它们是不同的。Janus-Pro 在预训练期间联合训练它们，且 LLM 的权重是共享的。

这种分离的成本很低：
- 每个理解图像的一次额外 SigLIP 前向传播（参数 300M，大致是 LLM 的 5%）。
- 每个生成图像的一次额外 VQ 编码器前向传播（参数更少，约 100M）。

作为回报，你获得：
- 理解准确度比共享编码器的统一模型高出几个点。
- 生成的 FID 分数比共享编码器的统一模型更优。

### Janus-Pro vs JanusFlow

Janus-Pro 对生成使用 VQ 扩散（Show-o 风格）。
JanusFlow（2024 年 11 月）对生成使用流匹配（Transfusion 风格）。

训练是相同的，直到生成头。JanusFlow 将 VQ tokenizer + 离散扩散替换为连续的 VAE 编码 + 流匹配。生成质量略有提高（在 MJHQ-30K FID 上前进一个点），但训练复杂性增加。（双损失权重现在不同了。）

DeepSeek 同时发布了两者，因为切换生成头以匹配部署场景或研究目标是便宜的。

### 训练阶段

Janus 使用三个阶段：

第 1 阶段：理解预训练。冻结 VQ 编码器和 LLM（除了投影器）。训练 SigLIP 投影器 + 图像 patch → LLM 对齐。在高资源语言（英语、中文）上训练。

第 2 阶段：生成预训练。冻结 VQ 编码器和 LLM（除了嵌入和头）。训练生成投影器。在文生图对上训练。

第 3 阶段：联合微调。所有编码器保持冻结。解冻 LLM 并一起训练理解和生成。（仔细权衡损失以避免漂移。）

### 训练数据

Janus-Pro 使用了比 Janus 多两倍的数据：1.07 亿个多模态理解样本（长文本描述、OCR、图表、文档）+ 4000 万个文生图对 + 15 万个视频样本（用于时间理解）。

### 基准测试

Janus-Pro-7B：在 VQAv2（79.4 vs 78.5）上匹配 LLaVA-1.6，在 T2I 生成（FID 9.5 vs 10.1）上优于 DALL-E 2，但不如 SDXL。统一模型不像专用模型那么擅长生成，但现在至少在所有任务上都具有竞争力。

关键胜利：在同一 checkpiont 中，Janus-Pro 同时处理理解 + 生成，质量可接受。这是首个真正能用的 unified 模型。Show-o 也不赖，但 Janus 避免了编码器折衷。

### 下游的发展

Janus-Pro 已经被 Janus-Pro Max（2025 年 7 月，680B 激活参数）和其他大型统一模型超越，但其架构启发了几乎所有后续的开源 unify 模型。

## 练习

1. Janus 使用 384 像素的 SigLIP 和 16x 的下采样 VQ 编码器。一张 384x384 的理解图像产生多少个 token？如果 VQ 编码器下采样 16x，同一图像会变成多少个生成 token？不成对的 token 长度如何处理？

2. 在 Phase 1（理解预训练）期间，LLM 被冻结。为什么？如果 LLM 在训练期间更新，什么会崩溃？

3. Janus-Pro 同时发布了两类生成损失：VQ 扩散和流匹配。如果产品团队告诉你他们需要迭代图像编辑（即：在 VQ 空间很难实现但连续空间支持掩码扩散），你会推荐哪种变体？

4. 解耦编码器增加了两次编码器前向传播而不是一次。在 7B 规模下，计算增加的总百分比是多少？这值得吗？

5. JanusFlow 使用流匹配。阅读第 4 节。描述 VQ 扩散路径和流匹配路径之间损失函数差异。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 解耦编码器 | "分离的视觉塔" | 理解图像通过 SigLIP（连续），生成图像通过 VQ（离散）；LLM 在中间 |
| 理解路径 | "SigLIP 管道" | Image → SigLIP ViT → 投影 → LLM 上下文 token |
| 生成路径 | "VQ 管道" | Image → VQ tokenizer → 离散索引 → 嵌入 → LLM 文本生成 token |
| JanusFlow | "基于流匹配的 Janus" | 将 VQ 生成替换为 DDPM 流匹配；更高的质量天花板 |
| 联合微调 | "解冻一切" | 第 3 阶段训练同时优化理解和生成 LLM 权重，使用多任务损失 |
| 共享编码器折衷 | "一个不适合所有" | 强制一个编码器承担两种任务，降低理解分辨率和生成离散覆盖率 |
