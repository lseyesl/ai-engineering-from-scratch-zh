# MIO：从 Any-to-Any 到原生流式多模态输出

> 迄今为止的每个统一模型（Janus、Show-o、Chameleon）都假设一个固定的序列：文本 token，然后图像 token，然后文本 token。MIO（Meta AI，Zhong 等人，2024 年 12 月）打破了这种模态块排序。MIO 将一个模态内的任何位置与来自不同模态的内容交错。一个对话可以包含文本词，其间散布着图像 token，后面跟着视频帧 token，之间还有文本 token——无论数据中出现什么顺序。为此，MIO 将模态生成帧化为状态机：一个模态生成器产生一定的 token，然后发出一个特殊的 `<next_modal>` token，触发另一个模态。结果是一个以任意序列生成任意模态的单一模型。本课程讲解 MIO 的状态机采样、训练配方以及为什么原生模态切换比模态轮循更好。

**类型：** 学习
**语言：** Python（标准库，MIO 状态机模拟器）
**前置要求：** Phase 12 · 15（Janus-Pro）
**时间：** ~120 分钟

## 学习目标

- 解释为什么 MIO 的状态机设计（模态间的软切换）比模型路由（模态块排序）更具表现力。
- 实现 MIO 的模态循环训练：给定一个跨越 M 种模态的序列，循环 M 种模态损失。
- 描述 MIO 部署原语：推测解码优先级、KV 缓存隔离、模态负载均衡。
- 将 MIO 与竞争性的 any-to-any 模型进行比较。

## 问题

Janus 和 Show-o 可以生成文本和图像，但它们以块为单位。模型产生 50 个文本 token，然后 1024 个图像 token，然后另外的文本 token。没有交错的混合——保持干净，但缺乏灵活性。

自然语言并不简单地遵守模态边界。一个人可以说："这是我去年做的蛋糕[图像]，这是蛋糕在烤箱里的样子[图像]，顺便说一下配方是[文本token]……加热到 350 度。"多模态输出的自然后续遵循相同的模式。

MIO 说：我们可以在生成的 token 空间中实现这一点。不需要模态切换。

## 概念

### 状态机生成

MIO 将生成视为一个状态机，状态代表模态。

正式地，存在一个模态集合 {文本, 图像, 视频, 音频}。每个状态有一个发射器（在该模态中产生 token）和一个转换函数（决定何时切换到新模态）。在采样过程中，发射器产生 token，直到满足一个停止条件（检测到一个特殊的 `<next_modal>` token 或固定的 token 预算），然后转换函数选择目标状态。

在实践中，这两者都由同一个 Transformer 建模——预测下一个 token，无论何种模态。但停止条件和转换函数是分开的，使推理更易于预测。

### 模态循环训练

为了支持任意到任意序列，MIO 使用一种称为模态循环的训练策略。给定一个跨越多个模态的训练序列，模型将其分解为模态片段。每个片段被馈送到适当的模态损失函数。

序列：[文本 token T1..T10] [图像 token I1..I50] [文本 token T11..T15]
训练：
- 片段 1（文本）：对 T1..T10 使用 NTP
- 片段 2（图像）：对 I1..I50 使用 VQ 扩散
- 片段 3（文本）：对 T11..T15 使用 NTP

梯度步骤累积所有片段的损失。训练序列本质上是每个片段不同损失的损失函数。

这与 Janus 的阶段式训练不同，在 Janus 中，模型学习固定块（先全部文本，然后全部图像）。在 MIO 中，模型在交错数据上从头开始训练。

### 跨模态依赖

MIO 的关键优势是跨模态依赖——图像 token 可以依赖于相同的序列中较早的图像 token。注意力是模态不可知的。序列中的图像 token 可以参考早期文本 token 和早期图像 token。

这解锁了多轮图像生成：模型生成一个图像，对话中提到它，然后生成基于第一张图像的第二张图像——都在同一个序列中，因此 KV 缓存保存了较早的图像 token。

### 模态负载均衡

在推理时，MIO 将每种模态分配给不同的计算资源（类似于 Janus 的 DVD）。文本 token 在 GPU 0 上产生；图像 token 被发送到 GPU 1 的 VQ 解码器（或扩散采样器）；视频 token 在 GPU 2 上处理。负载均衡器根据每种模态的预期 token 预算在 GPU 之间分配预生成的序列。

### 推测解码优先级

MIO 为每种模态维护一个独立的推测解码认证器。在生成时，文本 token 具有最高的推测优先级（文本 token 通常被推测得最多）。图像 token 具有较低的优先级，因为它们更昂贵且推测更少。MIO 维护一个包含 pending 推测 token 的多模态缓冲区，并以最高确定性为条件进行认证。

### 与其他 any-to-any 模型的对比

- MIO：原生交错，状态机，多模态模态循环，大批量部署流量优先级。
- Janus-Pro：解耦编码器但块顺序输出，无模态交错。
- Show-o：掩码扩散但仍然是块状（先全部文本，然后全部图像 token）。
- Emu3 / Chameleon：自回归顺序，但仍然是块状。
- Gemini 原生多模态（Google，2023/2024）：专有，但推测使用类似的状态机生成。

MIO 是最早的开源 any-to-any 模型之一，真正支持原生交错（而不仅仅是块状）。这使得它更适合对话——多模态 UX 要求的模态之间来回交替。

## 练习

1. MIO 的状态机维护单独的发射器和转换函数。在一个自回归 Transformer 中，LM 头担任这两个角色。提出一个架构修改，显式地将它们分开。

2. 模态循环训练为不同片段使用不同的损失。除了文本的 NTP 和图像的扩散之外，音频的适当损失是什么？

3. 采用跨模态依赖的 MIO 生成图像，其中一张图是蛋糕，下一张图是蛋糕在烤箱中。图 2 的 token 如何注意到图 1 的 token？数学推导这个注意力模式。

4. 提出一个产品用例，其中任意到任意交错是必需的，而不仅仅是锦上添花。

5. 比较 MIO 状态机与 Gemini 原生多模态方法（Gemini Tech Report，2023 年 12 月）。描述设计上的相似之处与不同之处。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 状态机生成 | "模态 FSM" | 每个模态是一个状态；发射器在该模态中产生 token，转换函数切换到下一个模态 |
| 模态循环 | "交错损失" | 对于每个模态片段，应用适当的损失，梯度跨模态累积 |
| 跨模态依赖 | "注意力穿透模态" | 序列中的图像 token 可以注意到之前的文本和图像 token，解锁多模态上下文 |
| 模态负载均衡 | "GPU per modality" | 将不同模态分配给不同的 GPU，并对输出 KV 缓存进行分区 |
| 推测解码优先级 | "推测优先级" | 模态的推测 token 认证顺序；文本优先，其次是图像，再是视频 |
