# 长视频理解与百万 token 上下文

> 一分钟的视频在 1 FPS 下产生 60 个帧。但以原生分辨率考虑视频理解所需的 token 数：一张 1080p 图像仅作为 ViT token 就包含 3072 (72*42) 个 patch。在 1 FPS 下，一分钟就是 3072 * 60 = 184,320 个 patch。在 24 FPS 下，就是 184k*24 = 4.4M 个 patch。单纯扩大 patch 大小会丢失细节。视频 VLM 生产系统使用复杂的采样策略——帧采样（每 N 帧取 1 帧）、时间压缩（将 8 帧合并为 1 个 token）、token 剪枝和 Mamba 状态层——来适配长视频。本课程阅读长视频 VLM 的架构——从 LLaVA-NeXT-Video 中的帧采样到长上下文视频语言模型中的 Mamba 混合——并构建一个 token 预算管理器，为任何视频长度推荐最佳策略。

**类型：** 构建
**语言：** Python（标准库，视频 token 预算计算器 + 自适应帧采样器）
**前置要求：** Phase 12 · 17（视频理解基础），Phase 7（视觉）
**时间：** ~180 分钟

## 学习目标

- 对于一段给定长度和分辨率的视频，计算在 LLM 上下文窗口内适配所需的 token 压缩率。
- 实现自适应帧采样——一个决定从视频中选择哪些帧以及在何处压缩字节的轻量级调度器。
- 描述 Mamba 状态层如何将紧凑的视觉状态转发到 LLM，减少 KV 缓存。
- 将 VidChapters、Video-MME 和 LongVideoBench 基准进行比较，并解释为什么 LongVideoBench 是更好的长视频评估基准。

## 问题

视频上下文是第一个真正的 LLM 上下文wall。

考虑生产视频 VLM 的 LLM 上下文中的一秒钟视频：

- 4K 视频（3840x2160）÷ 14px ViT patch = 274 * 154 = 42,196 个 patch/帧。
- 24 FPS = 42,196 * 24 = 1,012,704 个 token/秒。
- 60 秒 = 60,000,000 个 token。世界上没有 LLM 可以处理 60M token。
- 即使我们采用每 N 帧取 1 帧：在 0.5 FPS 下，60 秒 = 30 帧 = 30 * 42,196 = 1,265,880 个 token。对于 Gemini 1.5 Pro 或 GPT-4.5 的上下文窗口（约 200K）来说仍然太多了。

所以：你必须压缩。而压缩会丢失信息。关键在于在不丢失太多细节的情况下进行压缩，以便你仍然可以回答视频的后续问题。

## 概念

### 长期记忆 token

长视频 VLM 将视频分割成块。每块由一个视频编码器处理，该编码器将该块压缩成少量（通常 64-256 个）token。

这种压缩具有两种风格：

- 软压缩（基于注意力的池化）：视频编码器为每帧输出一个扁平序列；一个可训练的 MLP 或交叉注意力块将这些 token 压缩成一个固定大小的向量。LLM 看到的是压缩后的向量，而不是原始 token。
- 硬压缩（状态模型）：视频帧通过 Mamba 状态空间模型馈送。Mamba 将帧序列压缩成一个隐藏状态，该状态捕捉时间范围。然后 LLM 将这个隐藏状态视为一个 token。

两者都比简单的帧采样保留更多信息。

### 视频 Mamba 混合

David 等人（2024 年）的 Video Mamba 混合结合了用于视觉 token 压缩的 Mamba 块和用于 token 解码的 Transformer 块。

架构：
1. 视觉主干网（CLIP ViT）：每帧输出 patch token。
2. Mamba 压缩器：将 patch 序列（帧 1..N 的所有 patch）馈送到 Mamba 状态空间模型。Mamba 的隐藏状态（一次性处理整个序列）压缩了因果时空信息。
3. 投影：Mamba 隐藏状态 → 一个小的视觉 token 集（例如，每视频 256 个 token）。
4. LLM 解码器（标准 Transformer）：将视频 token + 文本 token 视为序列。解码答案。

好处：Mamba 的状态是 O(1) 内存，无论视频长度如何。你不需要将每个帧的每个 patch 存储在 KV 缓存中。你只需要存储压缩后的状态。对于 30 分钟的视频，这是唯一可行的路径。

坏处：Mamba 的压缩是有损的——掩蔽帧会丢失信息，但比帧采样丢失的要少。

Mamba → Transformer 的连接是训练稳定的。压缩 token 占据了 LLM 的固定 token 预算（例如，2048 个 token 全部用于视频）。其余上下文是文本 token。

### 自适应帧采样

不是以固定速率（每 N 帧取 1 帧）进行采样，而是使用一个关注度模型来选择信息最丰富的帧。

帧 i 的关注度是通过将帧的 CLIP 嵌入与文本查询的 CLIP 嵌入相比较来计算的。将文本查询编码，然后计算每一帧的余弦相似度。选择 top-K 帧。对于没有查询的设置（密集视频描述），使用差异的关注度——计算帧与前一帧之间的特征差异，选择差异幅度最大的帧。

对于视频问答，后者比固定采样高 3-5 分。与 Mamba 压缩结合时，可以增加 2x 的有效上下文。

### LLaVA-NeXT-Video 和 LLaVA-OneVision 工作

LLaVA-NeXT-Video（2024 年 1 月）是第一个重要的 LLaVA-Video 扩展。它使用 1 FPS，每帧 24 个 patch（通过动态分辨率将鲁棒性推向最大 7680×4320，但文本编码器每帧仅输出固定数量的 576 个 token）。长视频，使用每 N 帧取 1 帧（每秒 1 帧）+ 帧打包（每帧 1 个 token）和一个最大上下文。在 Video-MME 上的性能约为 50%。

LLaVA-OneVision（课程 12.8，2024 年 8 月）使用 S^2 打包和 4 帧（每段视频最多 16 帧，每帧 196 个 patch），辅以帧池化和动态分辨率。在 Video-MME 上达到 60%+。但到 2025 年，长视频模型在 Video-MME 上达到 80%+，因此这个领域正在快速发展。

### 基准测试：Video-MME vs LongVideoBench

Video-MME（2024 年 6 月）使用 YouTube 视频，每个视频 2-30 分钟。问题涵盖感知和时间推理。是一个很好的通用基准。

LongVideoBench（2024 年 7 月）包含 30-60 分钟视频。问题是基于视频中超出短期（<5 分钟）窗口的信息。对于测试"压缩是否足够好"的模型来说，这是一个更好的基准——它需要真正的长视频理解，而不是从 2 分钟剪辑中理解。

截至 2026 年，LongVideoBench 是最具区分度的开放视频基准。好的模型得分超过 70%；差的模型得分低于 40%。

## 使用它

`code/main.py` 实现了一个 token 预算管理器：

- 给定一个视频长度、分辨率、帧率和上下文窗口 → 确定压缩率。
- 自适应帧采样实现：基于查询的 cosine 选择 top-K 帧。
- 模拟 Mamba 压缩：将 K 帧压缩为 N_visual_tokens（可配置）。显示压缩率。
- 报告：给定 LLM 上下文窗口，你可以使用多少帧而不会溢出。为建议的视频工作负载推荐压缩策略。

## 交付物

本课程产生 `outputs/skill-video-token-budget-manager.md`。给定一个视频理解产品（视频长度、质量要求、基础模型上下文窗口、预算），推荐帧采样策略、帧率和压缩方法（池化 vs Mamba vs 注意力压缩），并附定量权衡。

## 练习

1. 对于 1080p 视频，以 14px ViT patch 大小，每秒帧率是多少？如果长期记忆 token 为 256 且 LLM 上下文窗口为 128K，你能在上下文中存放多少秒？视频编码器为每帧输出 576 个 token。使用 1 FPS，你能放多少秒的 1080p 视频而不溢出窗口？那么在 Mamba 压缩（隐藏状态，而非逐帧 token）下呢？

2. 自适应帧采样：给定三个候选查询"狗叫什么"、"汽车的颜色"、"人在做什么"，论证对同一视频的不同帧采样如何产生不同的答案。

3. 选择 video-MME 和 LongVideoBench，并描述每个基准中一个仅自适应帧采样就能提升 5 分的问题。

4. Mamba 的隐藏状态将整个视频压缩为 256 个 token。提出一种 Mamba 无法恢复的"需要精确数硬币"的长视频场景。Mamba 由于有损压缩而丢失了什么？

5. 如果可以每帧分配 196 个 token，但只有 4 帧的上下文，对于 3 分钟的视频，你如何选择帧？（提示：自适应帧采样，或均匀采样。）

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 长期记忆 token | "视频压缩" | 将视频帧序列压缩成少量固定 token 的技术，保存到 LLM 上下文 |
| Mamba 压缩 | "状态模型" | 将视频帧通过 Mamba 时间序列模型馈送以产生一个捕捉整个视频的高维隐藏状态 |
| 自适应帧采样 | "信息帧选择" | 一个基于帧嵌入与查询嵌入之间最大余弦相似度选择帧的调度器 |
| 信息帧 | "差异感知选择" | 通过选择差异幅度最大的帧来捕获视频中最大变化的帧 |
| LLaVA-NeXT-Video | "逐帧，每 N 帧取 1 帧" | LLaVA-1.5 的视频扩展：在 0.5-1 FPS 下对帧进行子采样，帧打包 |
| Video-MME | "中等长度视频基准" | 2-30 分钟；需要时间推理 |
| LongVideoBench | "30-60 分钟视频基准" | 2-60 分钟；专门测试超短期之外的长上下文 |
