# 故障模式——MAST、群体思维、单一文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等人，NeurIPS 2025，arXiv:2503.13657），来自 7 个最先进开源 MAS 的 1642 条执行轨迹，显示 **41–86.7% 的失败率**。三个根类别：**规范问题**（41.77%）——角色模糊、任务定义不清；**协调失败**（36.94%）——通信中断、状态不同步；**验证缺口**（21.30%）——缺少验证、没有质量检查。**群体思维**族（arXiv:2508.05687）增加了：单一文化崩溃（相同基础模型→相关失败）、从众偏差（智能体相互强化错误）、心智理论缺陷、混合动机动态、级联可靠性失败。级联示例：重试风暴——支付失败触发订单重试，触发库存重试，压垮库存服务（数秒内 10 倍负载——需要断路器）。内存中毒：一个智能体的幻觉进入共享内存，下游智能体将其视为事实；准确率逐渐下降，使根本原因诊断痛苦。**STRATUS**（NeurIPS 2025）报告通过专门的检测/诊断/验证智能体实现了 1.5 倍的缓解成功率提升。本课程将故障模式作为一流的工程目标来处理。

**类型：** Learn
**语言：** Python（标准库）
**前置知识：** Phase 16 · 13（共享内存），Phase 16 · 14（共识与 BFT），Phase 16 · 15（投票与辩论拓扑）
**时间：** ~75 分钟

## 问题

多智能体系统在真实任务上失败的概率为 41-86.7%（Cemri 等人 2025 年在 7 个开源 MAS 上测量）。这不是"多加智能体"就能调试的。这些失败有结构性原因。MAST 分类法为你提供了类别。本课程将每个类别映射到具体的检测、诊断和缓解模式，使这些数字不再显得随意。

2026 年的生产实践是将故障模式视为设计输入。你的架构在你能指出每个 MAST 类别并说出你部署的缓解措施之前，不是"足够好"的。

## 概念

### MAST 类别

**规范问题（占失败的 41.77%）。** 智能体的任务定义不够严谨。示例：

- 角色模糊：两个智能体都认为自己是评审者。
- 任务规定不足："总结这个"而用户想要的是特定角度。
- 成功标准隐式：智能体无法判断是否成功。

缓解措施：
- 编写显式的角色契约。每个智能体的提示说明它做什么*以及不做什么*。
- 每个任务的验收测试。在智能体开始之前，定义"完成看起来像 X"。
- 启动前规范检查：一个单独的智能体在分派前审查任务定义。

**协调失败（36.94%）。** 通信或状态故障。

示例：
- 两个智能体在没有同步的情况下更新共享状态。
- 智能体间消息丢失（队列故障、超时）。
- 状态漂移：智能体 A 认为任务完成；智能体 B 仍在执行。

缓解措施：
- 带乐观并发的版本化共享状态。
- 关键消息的显式确认（重试直到确认）。
- 周期性状态同步检查点；及早检测漂移。

**验证缺口（21.30%）。** 输出没有独立检查。

示例：
- 一个智能体声称成功；没有人验证。
- 智能体链，每个都信任前一个的输出。
- 涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立验证器智能体（Lesson 13）。只读、独立来源访问。
- 显式交接契约："A 的输出必须通过检查器 C 后 B 才能开始。"
- 事后分析的结果日志记录。

### 群体思维族（arXiv:2508.05687）

当智能体同质化或模仿彼此时产生的五种相关失败：

**单一文化崩溃。** 相同基础模型或训练数据→相关错误。当三个智能体共享一个 LLM 时，它们共享它的幻觉。

**从众偏差。** 智能体向最响亮或最自信的同伴调整，即使它是错的。

**心智理论缺陷。** 智能体未能建模彼此的信念；协调崩溃（Lesson 18）。

**混合动机动态。** 具有部分对齐激励的智能体向妥协中间点漂移，这满足不了任何人。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例——重试风暴

一个经典的 2026 年事故模式：

```
支付服务 10% 的请求失败
   ↓
订单智能体重试支付（指数退避但幼稚）
   ↓
每次重试都是一个新的订单库存检查
   ↓
库存服务看到 2 倍正常负载
   ↓
库存服务开始超时
   ↓
每个订单重试库存检查
   ↓
库存服务看到 10 倍正常负载
   ↓
集群宕机
```

修复方法是经典的：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。外加每个请求有上限的重试预算。

断路器是少数几个你直接从分布式系统借用的多智能体故障缓解措施之一，无需修改。

### 内存中毒（重温）

来自 Lesson 13：一个智能体的幻觉成为共享内存中的事实；下游智能体在中毒事实上推理。在 MAST 术语中，这是共享内存层的验证缺口。

逐渐的准确率下降是症状。你不会得到崩溃；你得到难以根本原因定位的缓慢漂移。

缓解措施：仅追加日志、来源、不可写入的验证器。在 Lesson 13 中已经介绍。

### STRATUS——用于故障检测的专业智能体

STRATUS（NeurIPS 2025）报告当你部署以下内容时缓解成功率提升 1.5 倍：

- **检测智能体。** 监视症状模式（高分歧、重试尖峰、准确率漂移）。
- **诊断智能体。** 给定症状，从 MAST 分类法推断可能根本原因。
- **验证智能体。** 在缓解措施应用后，检查症状是否消除。

这是 SRE 风格的事件响应，应用于智能体系统。三个角色都可以是具有专业化提示的 LLM 智能体。

### 故障模式审计

2026 年的最佳实践是年度（或每次主要版本）故障模式审计：

1. **轨迹采样。** 收集约 1000 条真实执行轨迹。
2. **分类。** 对于每个轨迹的失败，映射到 MAST + 群体思维类别。
3. **计算按类别失败率。** 哪些类别在你的系统中占主导？
4. **按缓解措施排序。** 哪个修复会消除最多的失败？
5. **选择 2-3 个缓解措施。** 实施；下个季度重新审计。

纪律比具体选择更重要。没有审计，失败会混入噪声，永远不会被系统地处理。

### 系统静默失败时

最危险的失败类别是静默正确性失败。大声失败的系统（崩溃、异常、警报）可以被监控。产生看似合理但错误输出的系统无法通过异常日志检测。这就是为什么验证缺口按次数计算只有 21.30%，但每个失败的成本最高。

投资于：
- 基于抽样的审查。
- 黄金数据集回归测试。
- 对重要输出的跨智能体交叉检查。

### 失败 vs 缓慢失败

有些失败是即时的；有些是缓慢的。即时失败（超时、模式不匹配、认证错误）检测成本低。缓慢失败（内存中毒、单一文化漂移、角色模糊）检测和预防成本高。

2026 年的工程手段：对缓慢失败的代理指标进行 instrumentation，以便你可以在它变成可见错误之前捕获漂移。协议率、重试率、输出长度分布和连续智能体版本之间的编辑距离都是有用的代理指标。

## 交付

生产中的故障模式纪律：

- **每季度 MAST 审计。** 不是每年。随着系统增长，类别会变化。
- **无处不在的断路器。** 每个对任何依赖服务的出站调用。默认断开阈值在 5-10% 错误率。
- **黄金数据集。** 小型、高质量、人工审计。每周针对它们运行回归测试。
- **STRATUS 三重奏。** 检测 + 诊断 + 验证智能体监控生产。先只从检测智能体开始；症状嘈杂时再添加诊断。
- **故障预算。** 按类别失败率的显式 SLO。超过预算会触发停止发布对话。

3. 阅读 Cemri 等人（arXiv:2503.13657）。选择他们 7 个 MAS 系统之一，映射其前 3 个失败类别。它们与 MAST 预测的相比如何？

5. 为你知道的一个特定多智能体系统设计 STRATUS 风格的检测-诊断-验证三重奏。检测监视哪些症状？诊断推荐哪些缓解措施？验证如何确认它们起作用？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|---|---|---|
| MAST | "2026 分类法" | Cemri 2025；3 个根类别 + 14 个亚型的失败。 |
| Specification Problem | "角色模糊" | 任务或角色定义不清；智能体不知道做什么。 |
| Coordination Failure | "状态漂移" | 智能体间的通信或同步故障。 |
| Verification Gap | "没人检查" | 输出在没有独立验证的情况下被接受。 |
| Groupthink family | "同质性失败" | 单一文化、从众、心智理论缺陷、混合动机、级联。 |
| Monoculture collapse | "相同模型，相同幻觉" | 来自共享基础模型或训练数据的相关错误。 |
| Retry storm | "级联错误放大" | 一个失败触发重试，放大了下游负载。 |
| Circuit breaker | "错误率高时快速失败" | 错误率超过阈值时断开；用默认值短路。 |
| STRATUS | "事件响应三重奏" | 检测 + 诊断 + 验证智能体。1.5 倍缓解成功率。 |
| Memory poisoning | "幻觉传播" | 共享内存事实被污染；下游智能体在毒物上推理。 |
