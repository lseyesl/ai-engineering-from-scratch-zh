# 生产级扩展——队列、检查点、持久性

> 将多智能体系统扩展到数千个并发运行需要**持久执行**。LangGraph 的运行时在每次超步（super-step）后写入检查点，键为 `thread_id`（默认使用 Postgres）；工作者崩溃释放租约，另一个工作者恢复。智能体可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）为每个智能体运行了一个生产者-消费者队列，具有三种状态（空闲/处理中/响应）和两层协调（组内聊天 + 组间管理聊天）。**Fiber/async** 在 LLM 流式传输中击败了每任务一线程：线程在等待 token 时 99% 的时间处于空闲状态，fiber 协作式地在 I/O 上让出。反方观点：Ashpreet Bedi 的"扩展智能体软件"主张**FastAPI + Postgres + 除此之外什么都不需要**，直到负载证明其他情况——简单的架构走得比预期更远。本课程构建持久检查点日志、每个智能体的带状态转换的工作队列、一个 async vs 线程演示，并落地"从简单开始"的实用规则。

**类型：** Learn + Build
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置知识：** Phase 16 · 09（并行群体网络），Phase 16 · 13（共享内存）
**时间：** ~75 分钟

## 问题

一个原型多智能体系统在一台笔记本电脑上用三个智能体在内存事件循环中工作。你把它移到生产环境：

- 智能体有时运行数小时（长时间研究、人在环中等待）。
- 工作者进程崩溃。重启丢失状态。
- 峰值负载是平均值的 10 倍；你需要水平扩展。
- 用户按智能体运行付费；你需要精确一次语义来收费。

内存事件循环做不到这些。你需要底层的持久执行层。2026 年的规范选项是：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. 参与者模型框架（MegaAgent 每个智能体的生产者-消费者）。
4. 手工打造的 FastAPI + Postgres（Bedi 的主张）。

本课程构建每个的微型版本。

## 概念

### 持久执行，模式

持久执行引擎在每次"步骤"（LangGraph 术语中的超步）后持久化完整程序状态。崩溃时：

```
工作者在步骤中崩溃
  -> 租约超时
  -> 另一个工作者拾取 thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这需要满足的条件：

- **可序列化的状态。** 所有智能体状态必须是可持久化的。带有实时数据库连接的闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体产生相同的动作（或将 LLM 调用委托给外部确定性预言机）。
- **幂等的副作用。** 外部调用（工具调用、支付）必须是幂等的或使用去重键。

LangGraph 在每个超步后写入检查点；Temporal 在每个活动后写入；Restate 使用事件源日志。三者实现相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化字典；每个超步向检查点表写入一行。恢复时，运行时从最后一个检查点重放，而不是从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化状态并释放工作者。当输入到达时，任何工作者都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每个智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中数千个并发智能体。架构：

```
agent i:
  state ∈ {空闲, 处理中, 响应}
  in_queue   <- 发送给智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调器：
  组内聊天（同一组中的智能体）
  组间管理聊天（高级路由）
```

两层协调让组内对话密集发生，同时组间保持稀疏——这是用于在数千个智能体中保持成本线性的模式。

### Async vs 每任务一线程

LLM 调用是 I/O 密集型的。等待下一个 token 的线程 99% 的时间处于空闲状态。线程每个约消耗 1MB RAM；在 10,000 个并发调用时，仅栈空间就需要 10GB。

Fiber（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作式让出。同样的 10,000 个调用可以舒适地放在一个进程中。在 LLM 智能体规模下，async 不是优化——而是架构。

例外：CPU 密集型的后处理（嵌入、tokenizer 技巧）仍然需要线程或进程。将你的 I/O 层与 CPU 层分离。

### Bedi 的反方观点

"扩展智能体软件"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度设计了。务实的默认做法：

- FastAPI + Postgres。
- 每个智能体运行是一行；状态在原地使用乐观并发更新。
- 通过 `pg_notify` 或简单的 Celery 工作者进行后台任务。
- 应用代码中的重试策略。

对于可管理任务上低于约 100 个并发智能体运行的负载，这通常就是你需要的全部。当你测量到它失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，才采用持久执行框架。过早采用会在不带来回报的仪式上浪费时间。

### 精确一次语义

对于付费的智能体运行，你需要"精确一次有效"（至少一次传递 + 幂等消费者）。工程手段：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用先写入一个表，然后一个单独的进程执行它们。两个步骤都是幂等的。
- **补偿事务。** 当一个副作用成功但其跟踪写入失败时，安排补偿。

这些是数据库工程模式，不是 LLM 特定的。LLM 的代价仅在于 LLM 调用很慢；其他一切都是标准的分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用"彩虹部署"：多个版本的智能体运行时同时运行，这样长时间运行的智能体不必在每次代码部署时被杀死。在流量切片上 canary 新版本；当旧版本的智能体完成时退役它们。

这对长时间运行的有状态系统来说是标准的；2026 年的调整是智能体可以存活数小时，因此部署周期必须适应。

### 规范的生产清单

- 持久状态（检查点、快照、或发件箱 + 可重放日志）。
- 幂等副作用。
- LLM 调用的异步 I/O 层。
- 带去重的至少一次传递。
- 有状态工作负载的彩虹/canary 部署。
- 可观测性：每个智能体的轨迹、超步审计、重试计数器。

## 构建

`code/main.py` 实现：

- `CheckpointStore`——基于 SQLite 的检查点日志，以 thread-id 为键。每个超步追加一行。
- `run_with_checkpoint(agent, thread_id)`——模拟运行中崩溃；第二个工作者从最后一个检查点恢复。
- `AgentQueue`——每个智能体的空闲/处理中/响应状态机，带有小的工作队列。
- `demo_async_vs_threads()`——通过 asyncio 和通过线程运行 500 个并发模拟"LLM 调用"；报告挂钟时间和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：检查点在模拟崩溃后成功恢复；async 版本在 < 1 秒内处理 500 个并发调用；线程版本需要数秒并且每个并发单元使用数量级更多的内存。

## 交付

规范的生产加固：

- **从简单开始（Bedi 的规则）。** FastAPI + Postgres，直到你测量到它失败。
- **在优化之前先做 instrumentation。** 每次运行的延迟直方图、每步时间、重试计数、失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 永远不要在部署时杀死正在进行的智能体运行。
- **当你遇到特定问题时采用持久执行引擎（Temporal / LangGraph / Restate）：** 数小时的人在环中等待、跨区域协调、复杂重试/补偿策略。
- **I/O 层用 async。** 线程仅用于 CPU 密集型的后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复工作；测量 async vs 线程并发差异。
2. 实现**发件箱**表：每个工具调用先写入发件箱，然后一个单独的 goroutine/任务执行。通过运行工具调用两次来验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将一半的新 thread_id 路由到每个；确认旧版本上的正在进行线程不被中断。
4. 阅读 LangGraph 的运行时文档。识别运行时中哪些特性在手工打造的 FastAPI + Postgres 版本中最耗时。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是明确的。草拟如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|---|---|---|
| Durable execution | "持久化程序状态" | 引擎在每个超步后写入状态；崩溃恢复是确定性的。 |
| Super-step | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| Idempotency | "可安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| Outbox pattern | "解耦副作用" | 将意图写入表；单独的执行器执行并标记完成。 |
| At-least-once delivery | "可能重复" | 消息队列语义；去重键使消费者有效一次。 |
| Rainbow deploy | "重叠版本" | 长时间运行工作负载期间的多个并发运行时版本。 |
| Async fiber | "协作式让出" | 用户态并发；对于 I/O 密集负载比线程便宜。 |
| Checkpoint | "状态快照" | 在超步边界处序列化的状态；用于恢复的键。 |
