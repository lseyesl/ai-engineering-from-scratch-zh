# AI 的 SRE — 多智能体事件响应、Runbook、预测性检测

> AI SRE 使用基于基础设施数据（日志、runbook、服务拓扑）通过 RAG 进行 grounding 的 LLM，来自动化调查、文档记录和协调阶段。2026 年的架构模式是多智能体编排 — 由监督者协调专门的智能体（日志、指标、runbook）；AI 提出假设和查询，人类批准判断决策。Datadog Bits AI 和 Azure SRE Agent 以托管产品形式提供此功能。Runbook 正在演进：NeuBird Hawkeye 使用对抗性评估（两个模型分析同一事件；一致 = 置信，不一致 = 不确定性）；运营记忆跨越团队变更持续存在。自动修复保持谨慎：AI 建议，人类批准。完全自主的操作范围狭窄（重启 Pod、回滚特定部署）并配有严格的安全护栏 — 任何推销"一劳永逸"的人都在过度承诺。新兴前沿：事件前预测。MIT 研究报告称，基于历史日志 + GPU 温度 + API 错误模式训练的 LLM 在 10-15 分钟前预测了 89% 的故障。预测：到 2026 年底，95% 的企业 LLM 将拥有自动故障转移。

**类型：** Learn
**语言：** Python（stdlib，玩具级多智能体事件分类模拟器）
**前置要求：** Phase 17 · 13（可观测性），Phase 17 · 24（混沌工程）
**时间：** ~60 分钟

## 学习目标

- 绘制多智能体 AI SRE 架构图：监督者 + 专门智能体（日志、指标、runbook）+ 人类批准门控。
- 解释为什么自动修复范围狭窄（重启 Pod、回滚部署）而非宽泛（重新架构服务）。
- 说出对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 置信；不一致 = 升级。
- 引用 MIT 89% 早期检测结果及其操作约束：没有执行力的预测只是仪表盘。

## 问题

一名值班工程师在凌晨 3 点收到告警。"结账流程错误率高。"他们检查 Datadog、Loki、三个 runbook、部署日志。30 分钟后他们才意识到根本原因是 vLLM 因 KV 缓存峰值导致的 OOM。他们重启 Pod；错误消失。

在 2026 年，调查的前 20 分钟是可以自动化的。按服务分组日志、关联最近的部署、匹配 runbook — 这些都是 RAG + 工具使用。一个有监督的智能体可以在人类打开 Datadog 之前完成第一轮分类并呈现假设。

完全自主的修复是另一个问题。重启 Pod：安全。扩展 GPU 池：如果策略允许则安全。重新架构服务：绝对不行。这门学科就是划定这条狭窄的界限。

## 概念

### 多智能体架构

```
          事件
             │
             ▼
        监督者
        /    |    \
       ▼     ▼     ▼
  日志智能体  指标智能体  Runbook 智能体
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
        人类批准
             │
             ▼
        操作（狭窄集合）
```

监督者将事件分解为子查询。专门智能体拥有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合信息，向人类呈现假设 + 证据。人类批准或重定向。

### 自动修复范围

**安全（狭窄）**：重启 Pod、回滚特定部署、在预先批准的范围内扩展池、启用预先批准的功能标志。

**不安全（宽泛）**：更改服务拓扑、修改资源限制、部署新代码、更改 IAM、修改数据库。

任何推销"一劳永逸"的人都在过度承诺。随着 AI SRE 的成熟，安全集合会增长，但边界是真实存在的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们对根本原因达成一致，置信度就高。如果它们不一致，则将两个假设都呈现给人类进行升级。简单的模式，有效过滤幻觉产生的根本原因。

### 运营记忆

团队人员流动是传统 SRE 的无声杀手 — 隐性知识流失。AI SRE 将 runbook + 事后分析存储在向量数据库中；智能体在每次新事件时检索。当新工程师加入时，AI 拥有完整的历史记录。

### 事件前预测

MIT 2025 年研究：基于历史日志、GPU 温度、API 错误模式训练的 LLM 在测试集上预测了 89% 的故障，提前 10-15 分钟。

现实检验：没有执行力的预测只是仪表盘。操作问题是"当我们预测到时，我们做什么？" 预防性排空？告警？自动扩缩？答案取决于具体策略。

### 2026 年的产品

- **Datadog Bits AI** — Datadog 内部的托管 SRE 副驾驶。
- **Azure SRE Agent** — Azure 原生。
- **NeuBird Hawkeye** — 对抗性评估 + 运营记忆。
- **PagerDuty AIOps** — 分类 + 去重。
- **Incident.io Autopilot** — 事件指挥官 + 协调。

### Runbook 即代码

Runbook 从 Confluence 页面演变为带有结构化章节（症状、假设、验证、行动）的版本化 Markdown。结构化的 runbook 提供更好的 RAG 检索。任何 AI-SRE 推广的第一步都是将非结构化的 runbook 转化为结构化的。

### 你应该记住的数字

- MIT 早期检测：89% 的故障，10-15 分钟提前时间。
- 多智能体分类：监督者 +（日志、指标、runbook）+ 人类。
- 安全自动修复集合：重启 Pod、回滚部署、在范围内扩展。
- 对抗性评估：两个模型独立；一致 = 置信。

## 使用它

`code/main.py` 模拟多智能体分类：日志智能体发现错误，指标智能体发现 CPU 峰值，runbook 智能体匹配已知问题。监督者对假设进行排序。

## 交付物

本课程产出 `outputs/skill-ai-sre-plan.md`。根据当前值班情况、事件量、团队成熟度，设计 AI SRE 推广方案。

## 练习

1. 运行 `code/main.py`。如果日志和指标智能体不一致怎么办？监督者如何解决？
2. 为你的服务定义三个"安全"的自动修复操作。为每个操作提供理由。
3. 编写一个结构化的 runbook 模板：章节、必填字段、验证命令。
4. 预测性检测在提前 12 分钟触发。你的策略是什么 — 告警、预防性排空，还是两者兼有？
5. 论证一个 3 人团队在 2026 年是否应该采用 AI SRE 还是等待。考虑成熟度、事件量、风险。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| AI SRE | "值班智能体" | 基于 LLM 的事件调查 + 协调 |
| 监督者智能体 | "编排器" | 将事件分解为子查询的顶层智能体 |
| 专门智能体 | "领域智能体" | 具有工具访问权限的子智能体（日志、指标、runbook） |
| 自动修复 | "AI 修复它" | 狭窄的预先批准操作；不是广泛的重新架构 |
| 运营记忆 | "向量 runbook" | 向量数据库中的事后分析 + runbook，用于 RAG |
| 对抗性评估 | "双模型检查" | 独立分析；一致 = 置信 |
| NeuBird Hawkeye | "对抗性工具" | 具有对抗性评估 + 记忆模式的产品 |
| Bits AI | "Datadog 的 SRE 智能体" | Datadog 托管的 AI SRE |
| 事件前预测 | "早期检测" | 故障预测提前 10-15 分钟 |
