# LLM 的影子流量、金丝雀发布和渐进式部署

> LLM 发布结合了软件部署中最困难的部分：没有单元测试、故障模式分散、信号延迟。流程是（1）影子模式（shadow mode）——将生产请求复制到候选模型，记录日志，比较，零用户影响；可发现明显的分布问题，但不是质量保证；（2）金丝雀发布（canary rollout）——渐进流量迁移 10% → 25% → 50% → 75% → 100%，每个步骤有门控；跟踪延迟百分位数、每请求成本、错误/拒绝率、输出长度分布、用户反馈率；（3）稳定性确认后对显著不同的备选方案进行 A/B 测试。非确定性（non-determinism）是不可消除的——由于 GPU FP 非结合性加批次大小差异，相同输入的跨次运行准确率差异可达 15%。成本是变量而非常量——一个好 20% 的模型每次调用的成本可能贵 3 倍。回滚速度至关重要：如果回滚需要重新部署，你就太慢了。策略存在于配置/标志中；模型存在于注册表中，带有固定摘要；回滚 = 翻转策略 + 恢复阈值 + 在数秒内固定旧模型。

**Type:** Learn
**Languages:** Python（stdlib，简易金丝雀渐进模拟器）
**Prerequisites:** Phase 17 · 13 (Observability)、Phase 17 · 21 (A/B Testing)
**Time:** ~60 分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（实时流量渐进）和 A/B（稳定性确认后的比较）。
- 列举五个 LLM 特定的金丝雀指标（延迟、每请求成本、错误/拒绝率、输出长度分布、用户反馈）。
- 解释为什么 LLM 非确定性（高达 15%）改变了发布中"稳定"的含义。
- 设计一个以秒计（策略翻转）而非小时计（重新部署）的回滚路径。

## 问题

你发布了一个新模型。离线评估显示准确率提升 3%。你在生产中启用。24 小时内，成本上升 40%，用户点踩增加 8%，三个客户工单报告"奇怪的回答"。你回滚。重新部署需要 3 小时。你的周末毁了。

每一步都是可以避免的。影子模式本可以在任何用户看到之前就发现 40% 的成本飙升。金丝雀发布本可以在用户点踩上升时就在 10% 的阶段停止。策略标志回滚只需 30 秒。纪律性就是填补"离线评估看起来不错"和"真实用户满意"之间差距的东西。

## 概念

### 影子模式

候选模型接收与生产相同的请求；输出被记录，不返回给用户。零用户影响。记录：

- 输出内容（与生产的差异）。
- Token 计数（成本差异）。
- 延迟。
- 拒绝和错误。

能发现：成本激增、长度回归、明显的拒绝变化、硬错误。不能发现：用户能感知的质量差异。影子是冒烟测试，而非质量测试。

### 金丝雀发布

带门控的渐进流量迁移。典型过程：1% → 10% → 25% → 50% → 75% → 100%。每个步骤在 5 个指标上设置门控：

1. **延迟百分位数**——P50、P95、P99。违规：金丝雀 P99 > 基线的 1.5 倍。
2. **每请求成本**——混合均值。违规：超过基线 20%。
3. **错误/拒绝率**——5xx 加显式拒绝。违规：基线的 2 倍。
4. **输出长度分布**——均值 + P99。违规：分布偏移。
5. **用户反馈率**——点踩/工单提交。违规：基线的 1.5 倍。

### 非确定性是新的方差

相同的输入产生不同的输出。原因：

- GPU FP 非结合性（浮点归约顺序因批次而异）。
- 批次大小差异（同一提示在 128 批次 vs 16 批次中）。
- 采样（温度 > 0）。

实测：在相同评估集上，跨次运行准确率差异高达 15%。发布中的"稳定"意味着指标在预期方差范围内，而非与基线完全相同。将门控设置在噪声水平之上。

### 成本是变量

一个好 20% 的模型每次调用可能贵 3 倍。每请求成本是五个门控之一。发布一个"更好"但破坏单位经济学的模型就是回滚理由。

### 回滚是武器

- **策略标志**（功能标志系统）：在配置中翻转百分比；数秒完成。
- **模型固定**（注册表摘要）：固定的模型不会自动升级。
- **回滚** = 恢复标志 + 将固定摘要设置为上一个版本。数秒而非数小时。

如果你的栈需要重新部署来回滚，在发布前修复这个问题。

### 工具

**Argo Rollouts** / **Flagger**——Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由**——服务网格级流量拆分。

**KServe / Seldon Core**——内置金丝雀功能的模型服务。

**功能标志**——LaunchDarkly、Flagsmith、Unleash。策略级翻转，无需重新部署。

### 指标节奏

根据流量量，金丝雀门控每 5-15 分钟检查一次。1% 流量在 10 请求/分钟下，每个窗口有 50-150 个数据点——对延迟足够，但对用户反馈来说噪声太大。10% 提供约 10 倍的数据。每个步骤应暂停足够长的时间以积累足够的样本。

### A/B 步骤是可选的

如果新模型明显不同（不同的行为、不同的成本曲线、不同的语气），在金丝雀通过后在 50% 时进行 A/B 测试。如果只是改进版本，金丝雀门控通过后直接跳到 100%。

### 你应该记住的数字

- 金丝雀过程：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入跨次运行方差高达 15%。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本门控：超过基线 20% 即为违规。
- 回滚：数秒而非数小时。

## 关键术语

| Term | What people say | What it actually means |
|------|----------------|------------------------|
| Shadow mode | "复制到新模型" | 零影响发送到候选模型做日志记录 |
| Canary | "渐进流量" | 逐步暴露给用户的带门控发布 |
| Gates | "发布检查" | 阻止进程的指标阈值 |
| Non-determinism | "LLM 方差" | 不可消除的跨次运行差异 |
| Policy flag | "标志翻转回滚" | 配置级回滚，数秒而非数小时 |
| Model pin | "注册表摘要" | 模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进" | Kubernetes 原生金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 带金丝雀原语的模型服务 |
| Istio weighted | "网格拆分" | 服务网格流量分配器 |
