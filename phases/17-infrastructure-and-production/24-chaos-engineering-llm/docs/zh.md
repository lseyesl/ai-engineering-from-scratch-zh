# LLM 生产环境的混沌工程

> 2026 年，LLM 的混沌工程已成为一门独立学科。在生产环境中运行实验之前的前提条件：定义好的 SLI/SLO、链路追踪+指标+日志可观测性、自动化回滚、runbook、值班人员。架构包含四个平面：控制（实验调度器）、目标（服务、基础设施、数据存储）、安全（防护 + 中止 + 流量过滤器）、可观测性（指标 + 链路追踪 + 日志）、反馈（反馈到 SLO 调整）。安全护栏是强制性的：燃烧率告警在每日错误预算消耗超过预期 2 倍时暂停实验；抑制窗口 + 链路追踪 ID 关联对告警噪声去重。节奏：每周小规模金丝雀 + SLO 评审；每月游戏日 + 事后分析；每季度跨团队韧性审计 + 依赖映射。LLM 特定实验：内存过载、网络故障、供应商故障、畸形提示、KV 缓存驱逐风暴。工具：Harness Chaos Engineering（LLM 驱动的建议、爆炸半径缩小、MCP 工具集成）；LitmusChaos（CNCF）；Chaos Mesh（CNCF Kubernetes 原生）。

**类型：** Learn
**语言：** Python（stdlib，玩具级混沌实验运行器）
**前置要求：** Phase 17 · 23（AI 的 SRE），Phase 17 · 13（可观测性）
**时间：** ~60 分钟

## 学习目标

- 列出五个混沌工程前提条件（SLI/SLO、可观测性、回滚、runbook、值班）并解释为什么跳过任何一个都会破坏实践。
- 绘制四个平面（控制、目标、安全、可观测性）以及反馈到 SLO 的循环。
- 列举五个 LLM 特定实验（内存过载、网络故障、供应商故障、畸形提示、KV 驱逐风暴）。
- 根据技术栈选择工具 — Harness、LitmusChaos、Chaos Mesh。

## 问题

传统技术栈的混沌测试已经成熟。LLM 技术栈增加了新的故障模式。一个包含恶意字符的 4K Token 提示使分词器停滞 12 秒。上游供应商返回 429；你的网关重试；你的服务因重试放大的并发而 OOM。在突发负载下，KV 缓存驱逐风暴导致重新预填充级联，使计算资源饱和。

这些都不会在单元测试中出现。混沌工程是你在用户之前发现它们的方法。

## 概念

### 前提条件

不要在缺少以下条件的情况下在生产环境中运行混沌实验：

1. **SLI/SLO** — 定义好的服务级别指标和目标。
2. **可观测性** — 链路追踪、指标、日志，连接到仪表盘。
3. **自动化回滚** — Phase 17 · 20 策略标志回滚。
4. **Runbook** — 结构化的，Phase 17 · 23。
5. **值班** — 有人响应。

缺少任何一项，混沌就会变成真实事件。

### 四个平面 + 反馈

**控制平面** — 实验调度器（Litmus 工作流、Chaos Mesh 调度、Harness UI）。

**目标平面** — 服务、Pod、节点、负载均衡器、数据存储。

**安全平面** — 终止开关、抑制窗口、爆炸半径限制、错误预算门控。

**可观测性平面** — 正常指标 + 链路追踪 ID 关联，以区分混沌引发的故障和自然故障。

**反馈循环** — 发现结果反馈到 SLO 调整、runbook 更新、代码修复。

### 安全护栏是强制性的

- **燃烧率告警**：如果每日错误预算消耗超过预期的 2 倍，暂停实验。
- **抑制窗口**：在实验期间，静默爆炸半径内的非实验告警。
- **链路追踪 ID 关联**：所有实验引发的错误都携带一个标签，以便值班人员去重。

### 五个 LLM 特定实验

1. **内存过载** — 通过发送高并发的长上下文请求强制 KV 缓存抢占风暴。观察：服务是优雅地降级还是崩溃？

2. **网络故障** — 切断推理网关和供应商之间的连接。观察：故障转移是否在 SLA 内启动？（Phase 17 · 19）

3. **供应商故障模拟** — OpenAI 返回 100% 429。观察：路由是否故障转移到 Anthropic？（Phase 17 · 16、19）

4. **畸形提示** — 注入使分词器停滞的负载（例如，深度嵌套的 Unicode、巨大的 UTF-8 码点）。观察：单个请求是否会锁死一个工作进程？

5. **KV 驱逐风暴** — 通过饱和 vLLM 块预算强制驱逐。观察：LMCache 是否恢复，还是服务降级？

### 节奏

- **每周** — 在预发环境中进行小规模金丝雀实验，可能 5% 的生产流量。
- **每月** — 针对特定场景的预定游戏日；跨团队参与；事后分析。
- **每季度** — 跨团队韧性审计；依赖映射更新。

### 工具

- **Harness Chaos Engineering** — 商业产品；AI 驱动的实验建议；爆炸半径缩小；MCP 工具集成。
- **LitmusChaos** — CNCF 毕业项目；基于 Kubernetes 工作流。
- **Chaos Mesh** — CNCF 沙箱项目；Kubernetes 原生 CRD 风格。
- **Gremlin** — 商业产品；广泛支持。
- **AWS FIS** / **Azure Chaos Studio** — 托管云服务。

### 从小开始

第一个实验：在稳态流量下杀死一个解码副本的 Pod。观察重新路由和恢复。如果这可行且看起来安全，再升级到网络混沌。

第一个 LLM 特定实验：注入一个供应商 429 持续 5 分钟。观察故障转移。大多数团队发现他们的故障转移并未经过充分测试。

### 你应该记住的数字

- 四个平面：控制、目标、安全、可观测性。
- 燃烧率暂停：每日预算消耗超过预期的 2 倍。
- 节奏：每周金丝雀、每月游戏日、每季度审计。
- 五个 LLM 实验：内存、网络、供应商、畸形提示、KV 风暴。
