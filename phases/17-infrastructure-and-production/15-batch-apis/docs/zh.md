# 批量 API（Batch APIs）—— 50% 折扣已成行业标准

> 每家主流提供商都提供异步批量 API，附带 50% 折扣和约 24 小时周转时间。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks batch tier、Together batch）都实现了相同的模式。将批量与提示缓存（prompt caching）叠加，隔夜管道的成本可降至同步非缓存方案的约 10%。规则极其简单：如果不是交互式的，就应该走批量。内容生成管道、文档分类、数据提取、报告生成、批量标注、目录打标——任何能容忍 24 小时延迟的工作负载，如果不迁移到批量，就是在浪费钱。2026 年的生产模式是将每个新的 LLM 工作负载分流到三个通道：交互式（同步 + 缓存）、半交互式（异步队列 + 回退）、批量（隔夜、缓存输入叠加）。那些假装是交互式但能容忍几分钟延迟的工作负载，浪费最大。

**类型：** 学习
**语言：** Python（stdlib，玩具级批量 vs 同步成本模拟器）
**前置要求：** Phase 17 · 14（提示与语义缓存）
**时间：** ~45 分钟

## 学习目标

- 列举三家提供商的批量 API（OpenAI、Anthropic、Google）及其通用的 50% 折扣 + 24 小时周转保证。
- 计算在隔夜分类工作负载上叠加批量 + 缓存输入的成本，并与同步非缓存基线对比。
- 将工作负载分流到交互式/半交互式/批量，并证明分流决策的合理性。
- 指出两个陷阱：部分交互性（用户期望快于 24 小时）和输出模式漂移（批量文件格式因提供商而异）。

## 问题

你的团队负责一个夜间报告生成管道。50,000 份文档，每份需要摘要，然后聚类摘要，最后起草一份执行简报。同步运行需要 4 小时，每晚花费 $2,000。你听说了批量 API。

批量给你 50% 折扣。你还在系统提示（所有 50k 次调用共享）上启用了提示缓存。叠加后，账单降至 $180/晚——约为基线的 9%。同样的管道，三次配置变更。

批量是 LLM 成本工具箱中最便宜但没人用的杠杆。原因主要是组织层面的：团队在 SLA 实际上是"明早之前"时，却想着"实时"。这节课讲的是如何不把 90% 的账单浪费掉。

## 概念

### 三种批量 API

**OpenAI 批量 API**：JSONL 文件上传，包含一系列请求。承诺 24 小时周转（实际通常约 2-8 小时）。输入和输出 token 均享受 50% 折扣。`/v1/batches` 端点。符合缓存条件的输入还可叠加缓存输入定价。

**Anthropic Message Batches**：JSONL 上传。24 小时周转。50% 折扣。支持 `cache_control`——缓存写入是显式的，读取在批量内自动发生。

**Google Vertex AI 批量预测**：BigQuery 或 GCS 输入。Gemini 同样享受约 50% 折扣。与 Vertex 管道集成。

### 本质：异步，而非慢速

批量意味着"我承诺在 24 小时内返回"——而不是"这需要 24 小时"。典型的 P50 是 2-6 小时。提供商会安排你的批量在非高峰时段运行，此时 GPU 库存利用率较低。

### 与缓存叠加

一个 50k 文档摘要任务，使用相同的 4K token 系统提示：

- 同步非缓存：50000 × ($input × 4000 + $output × 200) 按全价计算。
- 同步缓存：系统提示在第一次写入后缓存；剩余 49999 次输入便宜 10 倍。
- 批量缓存：以上所有优惠加上读写均享受 50% 折扣。

叠加效果：批量 + 缓存 = 同步非缓存账单的约 10%。任何隔夜运行且有共享系统提示的工作负载都应使用此方案。

### 工作负载分流

**交互式**——用户等待响应。TTFT 至关重要。使用同步调用加提示缓存。不能走批量。

**半交互式**——用户提交任务，几分钟后回来看结果。异步队列，如果批量不可用则回退到同步。例如中等规模的 RAG 索引。

**批量**——用户期望"明早之前"或"下一小时"得到结果。内容管道、大规模分类、离线分析。始终走批量，始终叠加缓存。

常见错误：因为管道是生产环境就把所有东西归类为交互式。生产环境不是延迟规格——SLA 才是。

### 部分交互性陷阱

有些功能看起来是交互式的，但能容忍 5-10 分钟。例如：一个带有"刷新"按钮的夜间客户健康报告。用户点击刷新；等 10 分钟没问题。团队却以同步方式实现。50 个并发刷新的成本是通过批量处理并通过电子邮件交付的 10 倍。

要问的问题是："24 小时对这个用户意味着什么？"如果答案是"他们不会注意到"，那就走批量。

### 输出模式陷阱

批量文件格式因提供商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式内嵌。
- Vertex：BigQuery 表或带 TFRecord 的 GCS 前缀。

编写"一个批量客户端"跨提供商意味着每个提供商都需要适配代码。那些宣传支持多提供商批量的网关（Portkey、LiteLLM 某些层级）仍然只是对原始格式的薄封装。

### 你应该记住的数字

- 各提供商的批量折扣：输入 + 输出统一 50%。
- 周转 SLA：保证 24 小时，典型 P50 为 2-6 小时。
- 批量 + 缓存输入叠加：同步非缓存成本的约 10%。
- 工作负载分流规则：如果 24 小时延迟可接受，始终走批量。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| Batch API | "异步折扣" | 50% 折扣，24 小时周转 |
| JSONL | "批量格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批量" | Anthropic 的批量 API 产品名称 |
| Batch prediction | "Vertex 批量" | Vertex AI 的批量 API 产品 |
| Turnaround SLA | "24 小时承诺" | 保证值，非典型值；典型值为 2-6 小时 |
| Workload triage | "交互性决策" | 交互式/半交互式/批量路由决策 |
| Output schema | "响应格式" | 每个提供商的 JSONL 布局；不可移植 |
| Stacked discount | "批量 + 缓存" | 两者叠加时约为非缓存同步账单的 10% |
