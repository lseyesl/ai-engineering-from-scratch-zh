# 推测解码与 EAGLE-3

> 第 7 阶段·第 16 课证明了数学原理：Leviathan 拒绝规则精确地保持了验证器的分布。本课程是 2026 年生产级推测解码的训练栈视角。EAGLE-3 将草稿模型从一个廉价的近似变成了一个在验证器自身隐藏状态上训练的、专门构建的微型网络，然后添加了一个训练时测试循环，使其训练和推理分布对齐。结果：3 到 6.5 倍的端到端加速，聊天场景下每 token 接受率超过 0.9，没有分布权衡。2026 年的每个生产推理栈默认都配备它。

**类型:** 构建
**语言:** Python (stdlib)
**前置要求:** 第 7 阶段·第 16 课（推测解码数学），第 10 阶段·第 12 课（推理优化）
**时间:** ~75 分钟

## 学习目标

- 用一句话陈述 Leviathan 定理，并证明推测循环产生的样本与验证器的分布完全相同。
- 讲述从普通推测解码（Leviathan 2023）到 EAGLE、EAGLE-2 和 EAGLE-3 的两年演进历程，指出每一步消除的具体限制。
- 从接受率 `α` 和草稿与验证器的成本比 `c` 计算预期加速比，并为每种情况选择最优草稿长度 `N`。
- 从头实现完整的推测循环：草稿、验证、从残差分布中拒绝采样、拒绝时回滚 KV 缓存、完全接受时发出奖励 token。

## 问题

在 70B 模型上进行自回归解码，在 H100 上大约每秒生成 35 个 token。GPU 远未饱和。内存带宽是瓶颈：每个 token 从 HBM 加载 70B 的权重，执行一步算术运算，产生一个浮点数。计算单元大部分时间处于空闲状态。

推测解码将其转化为一个你实际上可以解决的吞吐量问题。一个廉价的草稿模型用 N 次小型前向传播提出 N 个 token。验证器在包含所有 N 个草稿的前缀上运行一次。如果验证器在位置 `i` 的分布与草稿一致（在我们将要精确说明的统计意义上），我们就接受；如果不一致，我们就拒绝并从残差分布中采样一个修正值。一次大模型前向传播可以产生最多 `N+1` 个接受 token，而不是一个。

关键的定理是 Leviathan、Kalman、Matias（ICML 2023）提出的：输出分布与直接使用验证器采样产生的分布完全相同。不是近似。完全相同。这就是推测解码在生产中可接受的唯一原因——它是一种纯粹的延迟优化，没有质量权衡。

第 7 阶段·第 16 课给你的是数学。本课程给你的是训练栈。一个好的草稿模型带来的加速是廉价草稿模型的 2 倍以上。EAGLE、EAGLE-2 和 EAGLE-3（Li 等人，2024-2025）将"草稿 = 同一模型的更小版本"变成了一门精确的工程学科。2026 年的生产推理服务器默认使用 EAGLE-3。

## 概念

### 不变式：Leviathan 拒绝采样

设 `p(t)` 是给定某个前缀下草稿模型对下一个 token 的分布，`q(t)` 是验证器的分布。采样一个草稿 token `d ~ p`。以概率 `min(1, q(d) / p(d))` 接受。拒绝时，从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。得到的样本按照 `q` 分布。无论 `p` 有多差，这都是成立的——越差你拒绝的次数越多，但输出仍然是精确的。

使用验证器在 `prefix + d_1 + ... + d_N` 上一次前向传播，堆叠 N 个这样的调用。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右遍历。在位置 `j` 首次拒绝时，从 `residual(q_j, p_j)` 采样并停止。完全接受时，从 `q_{N+1}` 采样一个奖励 token。

### 什么决定了加速

设 `α` 是每个草稿 token 的预期接受率。设 `c = cost(草稿) / cost(验证器)` 是成本比。每次验证器前向传播的预期接受 token 数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个接受 token 的预期总耗时为 `(N * c + 1) / E[accepted]`。对 `N` 最小化即可得到最佳点。对于 `α = 0.8, c = 0.05`：最优 `N` 约为 5-7，加速比为 3.2 倍。对于 `α = 0.95, c = 0.02`：最优 `N` 约为 8-10，加速比可推向 5 倍。

单个最大的杠杆是 `α`。在固定的 `N = 5` 下，从 `α = 0.6`（普通草稿）到 `α = 0.9`（EAGLE-3），每次验证器前向传播的预期接受 token 数从 2.2 增加到 4.1。来自相同验证器的吞吐量提升近 2 倍。

### 两年演进历程

**普通推测（Leviathan, 2023）。** 草稿模型是来自同一家族的独立训练的较小 LLM。易于连接，`α ≈ 0.6`，加速比最多约 2 倍。

**EAGLE-1（Li 等人, 2024）。** 草稿是一个微型 Transformer——通常只有一两层——以验证器的最后一层隐藏状态作为输入，直接预测下一个 token。由于草稿模型看到了验证器的特征表示，其分布更接近验证器。`α` 提升到 0.7-0.8。

**EAGLE-2（Li 等人, 2024）。** 添加了动态草稿树：不提出单一序列的 N 个 token，而是提出一个小的候选树，用一次前向传播（树注意力）对每个候选进行评分，然后走最高概率路径。草稿长度变为每步自适应。每接受路径的 `α` 超过 0.85。

**EAGLE-3（Li 等人, 2025, NeurIPS）。** 还有两个变化。首先，完全丢弃特征预测损失——EAGLE-1/2 训练草稿模型匹配验证器的隐藏状态，这限制了数据的帮助效果。EAGLE-3 直接在 token 预测上训练。其次，训练时测试（TTT）：在草稿训练期间，将草稿模型自身之前的预测作为输入反馈回去，经过多个步骤，就像在推理时操作一样。这使训练和测试分布对齐，并阻止了错误累积。实测加速比：聊天场景高达 6.5 倍，在 H100 上 SGLang 中的 batch 64 下吞吐量提升 38%。

### KV 缓存回滚

验证在一次前向传播中将验证器的 KV 缓存扩展了 N 个条目。如果在位置 `j` 拒绝，则位置 `j-1` 之后的缓存内容现在是错误的。两种常见实现：写入临时缓冲区并在接受时提交（vLLM、TensorRT-LLM），或者维护一个物理 KV 缓存加一个逻辑长度并在拒绝时截断。无论哪种方式，回滚成本是每层每头的字节数，与前向传播成本相比可以忽略不计。

对于 EAGLE-2 树搜索，验证器使用一个尊重树拓扑的非因果掩码运行注意力。工程实现很繁琐，但计算是一个带有自定义掩码的标准 flash-attention 调用。

### 2026 年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练成本 |
|------|----------|-----|--------|----------|
| 普通 | 独立小 LLM | 0.55-0.70 | 1.8-2.3× | 无（复用现有小模型） |
| Medusa | 验证器上的额外 LM 头 | 0.65-0.75 | 2-3× | ~1B SFT token |
| EAGLE-1 | 基于隐藏状态的 1 层 Transformer | 0.70-0.80 | 2.5-3× | ~60B token |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80-0.88 | 3-4× | ~60B token |
| EAGLE-3 | 多层特征融合 + TTT | 0.88-0.92 | 3.5-6.5× | ~60-200B token |
| Lookahead | 无草稿（Jacobi 迭代） | N/A | 1.3-1.6× | 无 |

在 2026 年生产中：vLLM 和 SGLang 默认可用时使用 EAGLE-3，否则使用 EAGLE-2。TensorRT-LLM 为 Meta 和 NVIDIA 公开模型提供了最快的 Medusa 路径。llama.cpp 为 CPU 部署提供普通草稿。

## 构建

参见 `code/main.py`。这是完整的 Leviathan 推测循环，包含所有组成部分：N 个草稿、验证器并行前向传播、逐位置拒绝、残差采样、奖励 token、KV 回滚，以及输出分布与直接 `q` 采样匹配的经验验证。

### 第 1 步：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 第 2 步：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 第 3 步：完整的推测步骤

`spec_step` 函数从 `p` 草稿生成 N 个 token，然后在一次并行的 `q` 评估中验证所有 token。对于每个草稿 token，它应用拒绝规则，在首次拒绝时从残差中采样修正值。如果全部接受，则从 `q_{N+1}` 发出一个奖励 token。

### 第 4 步：KV 回滚记账

模拟器为每个工作线程跟踪一个逻辑 `kv_length`。接受 k 个草稿时，`kv_length += k`。在位置 `j` 拒绝时，缓存已经写入了 `j` 之后的内容，但逻辑长度设置为 `prefix_length + j + 1`——修正 token 之后的一个位置。后续读取截断到逻辑长度。

### 第 5 步：Leviathan 检查

运行 50,000 步推测。统计接受 token 的经验分布。与从 `q` 直接采样的 50,000 个样本进行比较。卡方统计量应远低于临界值。该定理在实践中成立。

### 第 6 步：加速 vs α

通过以不同幅度扰动 `p` 使其偏离 `q`，扫描草稿质量。测量 `α`，然后绘制预期 token 数随 `α` 和 `N` 变化的函数。代码打印一个表格，展示 EAGLE-3 级别的草稿质量（`α ≈ 0.9`）如何实现每次验证器调用 4-5 个 token。

## 使用

生产级 `vllm serve` 配合 EAGLE-3：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

SGLang 配合 EAGLE-3，在 H100 上 batch 64：根据 EAGLE-3 论文，吞吐量大约是 batch-64 普通解码的 1.38 倍。

何时使用推测解码：

- 任何交互式聊天工作负载，其中 p50 延迟比峰值吞吐量更重要。
- 代码生成和结构化输出（JSON、SQL）。`α` 超过 0.9，因为目标分布高度可预测。
- 长文本生成（数千 token）。分摊的加速持续生效。

何时不使用：

- 非常小的模型（< 3B）。草稿模型并不比验证器便宜多少。
- 极小的 batch-1 CPU 部署。草稿模型的内存开销可能不值得。
- 极高温度的创意采样，其中 `α` 会崩溃。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| Leviathan 规则 | "min(1, q over p)" | 伯努利接受/拒绝，概率为 `min(1, q(d)/p(d))`，当拒绝时从残差中采样，精确保持验证器分布 |
| 残差分布 | "(q 减 p) 正值化并归一化" | `(q - p)_+` 在零处截断并重新归一化——拒绝时采样的正确分布 |
| 接受率 α | "草稿正确的频率" | 拒绝规则下每 token 的伯努利成功预期概率；决定所有加速计算 |
| EAGLE-1 | "隐藏状态草稿" | 以验证器最后一层隐藏状态为条件的微型 Transformer 草稿（Li 等人, 2024） |
| EAGLE-2 | "动态草稿树" | EAGLE-1 加上候选延续树，在验证器一次前向传播中用树注意力评分 |
| EAGLE-3 | "训练时测试" | 丢弃特征预测损失，在直接 token 预测上训练，训练时将草稿自身输出反馈给它 |
| 训练时测试 (TTT) | "暴露偏差修复" | 在训练期间自回归运行草稿，使训练和测试的输入分布匹配——计划采样的直接类比 |
| KV 回滚 | "撤销被拒绝的草稿" | 将验证器的 KV 缓存重置到拒绝发生后已接受前缀长度的记账操作 |
| 奖励 token | "免费的那个" | 当所有 N 个草稿都被接受时，从 `q_{N+1}` 额外采样一个，无需额外的验证器开销 |
| 树注意力 | "一次验证多个候选" | 使用一个尊重草稿树拓扑的非因果掩码的注意力；在一次前向传播中计算树中每个节点的 `q_i` |
