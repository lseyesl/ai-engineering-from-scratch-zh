# 原生稀疏注意力 (DeepSeek NSA)

> 在 64k token 的序列长度下，注意力机制消耗了 70-80% 的解码延迟。每个开源模型实验室都有自己的修复方案。DeepSeek 的 NSA（ACL 2025 最佳论文）是真正落地的那个：三个并行注意力分支——压缩的粗粒度 token、选择性保留的细粒度 token、以及用于局部上下文的滑动窗口——通过一个可学习的门控组合。它对硬件友好（kernel 友好），原生可训练（在预训练中工作，而非推理时外挂），在 64k 解码时比 FlashAttention 运行更快，同时匹配或超越全注意力质量。本课程端到端构建三个分支，并解释为什么稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python (stdlib)
**前置课程：** Phase 7 · 12 (KV cache, flash-attention), Phase 7 · 15 (注意力变体), Phase 10 · 16 (差分注意力)
**时长：** ~60 分钟

## 学习目标

- 说出 NSA 的三个注意力分支及其各自捕获的内容。
- 解释为什么 NSA 是"原生可训练"的，而之前的稀疏注意力方法仅用于推理。
- 计算在 64k 上下文下 NSA 相对于全注意力的计算节省量，作为压缩块大小和选择 top-k 的函数。
- 在短合成序列上用 stdlib Python 实现三分支组合，并验证门控权重的行为。

## 问题所在

全注意力在序列长度 N 下的时间复杂度为 `O(N^2)`，每层的 KV cache 为 `O(N)`。在 64k token 时，计算和内存带宽的数字是灾难性的。NSA 论文中的理论估计：在 64k 时，注意力占总解码延迟的 70-80%。下游的一切——TTFT、tokens/sec、每百万 token 的成本——都由注意力成本主导。

稀疏注意力是显而易见的答案。之前的尝试分为两类。固定模式稀疏性（滑动窗口、跨步、块局部）丢弃信息，在长程回忆任务上失败。推理时稀疏性（KV cache 剪枝、H2O、StreamingLLM）应用于在全注意力上预训练的模型，仅恢复潜在加速的一小部分，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等，DeepSeek + PKU + UW，ACL 2025 最佳论文，arXiv:2502.11089）两者兼顾：一个模型在预训练期间学习的稀疏模式，实现为 kernel 对齐的算法，在推理时真正交付计算节省。两年后，NSA 或其直系后代将成为每个前沿长上下文模型的默认注意力。

## 核心概念

### 三个并行分支

对于每个查询，NSA 运行三次注意力，针对 KV cache 的三个不同视图：

1. **压缩分支。** Token 被分组为大小为 `l`（通常为 32 或 64）的块。每个块通过一个小的学习 MLP 压缩为单个摘要 token。查询对这些压缩 token 进行注意力计算，获得整个序列的粗粒度视图。

2. **选择分支。** 使用压缩分支的注意力分数，识别与当前查询最相关的 top-k 块。读取这些块的细粒度（未压缩）token，查询对所有这些 token 进行注意力计算。可以将压缩分支注意力视为选择的路由信号。

3. **滑动窗口分支。** 查询对最近的 `W` 个 token（通常为 512）进行注意力计算以获取局部上下文。此分支捕获其他两个分支可能遗漏的结构密集型短程模式（语法、局部共指）。

三个分支的输出通过一个可学习的逐位置门控组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是查询上的小型 MLP 输出的门控权重。它们不必总和为 1——它们可以独立地加权各分支。

### 为什么这是"原生可训练"的

选择步骤（top-k 块）是离散的。离散操作会中断梯度流。之前的稀疏注意力工作要么跳过通过选择的反向传播（限制了训练），要么使用不能在推理时给出真正稀疏性的连续松弛。

NSA 绕过了这个问题：压缩分支注意力本身就是对整个序列的可微粗粒度注意力。top-k 操作只是复用压缩分支的顶部注意力分数来选择加载哪些细粒度块。梯度通过压缩分支分数流动（这些分数同时影响压缩输出和选择逻辑），并且所选块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图上是空操作——它只控制从内存加载哪些块。

这就是为什么 NSA 可以端到端地用于预训练。模型学习联合地通过三个分支路由信息，产生一个在推理时真正交付承诺加速的稀疏模式。

### 硬件对齐的 kernel

NSA 的 kernel 为现代 GPU 内存层次结构而设计。kernel 按 GQA 组加载查询（外循环），获取每个组对应的稀疏 KV 块（内循环），并在 SRAM 上运行注意力。因为每个查询组看到相同的选择块（选择是按查询组而非按查询头进行的），KV 加载在组内分摊。算术强度保持高位。

论文报告 Triton kernel 在 64k 解码时比 FlashAttention 快 9 倍，加速比随序列长度增长。前向和反向 kernel 均已提供。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数量，`w` 为滑动窗口，`b` 为所选块大小（通常等于 `l`）。

- 压缩分支：每个查询 `O(N/l)` 个 key，总计 `O(N * N / l)`。
- 选择分支：每个查询 `O(k * b)` 个 key，总计 `O(N * k * b)`。
- 滑动分支：每个查询 `O(w)` 个 key，总计 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当 `N = 64k, l = 64, k = 16, b = 64, w = 512` 时：每个查询的代价为 `1000 + 1024 + 512 = 2536 个 key`。全注意力为 `64000 个 key`。25 倍计算缩减。

当 `N = 128k, l = 64, k = 16, b = 64, w = 512` 时：每个查询的代价为 `2000 + 1024 + 512 = 3536 个 key`。全注意力为 `128000 个 key`。36 倍缩减。收益随序列长度增长，这正是全部意义所在。

### 对比

| 方法 | 可微 | 真正的推理加速 | 长程回忆 |
|------|------|--------------|---------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 跨步/块稀疏 | 是 | 是 | 部分 |
| KV 剪枝 (H2O, StreamingLLM) | 不适用（推理时） | 是 | 部分 |
| MoBA (Moonshot) | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k 时 9 倍） | 匹配全注意力 |

MoBA（Moonshot，arXiv:2502.13189）同期发表，采用了类似的"三胜于一"方法，将 MoE 原则应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练需要了解的两种架构。

## 构建它

`code/main.py` 在短合成序列上实现三个分支，并展示：

- 压缩 MLP（为教学清晰性使用了简单的均值池化基线；真正的 NSA 使用学习 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 最后 `w` 个 token 上的滑动窗口注意力。
- 门控组合。
- 与全注意力比较的计算量打印输出。

### 步骤 1：将 token 压缩为块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤 2：压缩分支注意力

对查询运行针对压缩 key 的 softmax 注意力。压缩分支分数同时作为 top-k 选择的信号。

### 步骤 3：top-k 块选择

选取 `k` 个最高分压缩块的索引。从这些块加载原始未压缩 token 并对其运行注意力。

### 步骤 4：滑动窗口注意力

取最后 `w` 个 token 并对其运行标准注意力。

### 步骤 5：门控 + 组合

查询上的小型 MLP 产生三个门控权重。最终输出是三个分支输出的加权和。

### 步骤 6：计算量统计

打印每个查询每个分支参与的 key 数量及总数。与 `N`（全注意力）比较。在 1024-token 合成序列上，`l = 32, k = 4, w = 128`，NSA 每个查询看到 `32 + 128 + 128 = 288` 个 key，而全注意力为 1024——减少 3.5 倍。

## 使用它

NSA 正在 DeepSeek 自身的长上下文预训练流程中部署。截至 2026 年 4 月，公共推理栈中的集成状态：

- **DeepSeek 内部**：原生，发布的权重使用 NSA 或其继任者 DSA（Deepseek Sparse Attention）。
- **vLLM**：实验性 NSA 支持正在开发中，用于 DeepSeek-V3.x 权重。
- **SGLang**：NSA 基准测试已发布；生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；kernel 分解的开销在 CPU 吞吐量下不值得。

何时选择 NSA：

- 目标为 64k 以上上下文且有严肃计算预算的预训练或继续训练运行。
- DeepSeek 自身长上下文检查点的推理。权重是 NSA 原生的。

何时不选：

- 服务现有密集注意力预训练模型。你不能在没有继续训练的情况下改造 NSA。
- 上下文低于 16k。三分支开销主导了节省。
- 批量 1 的交互式聊天。延迟敏感的解码受益，但仅在长上下文时。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| Compressed branch | "粗粒度视图" | 对块平均 key 的注意力，以 O(N/l) 个 key 每查询提供全局上下文 |
| Selected branch | "Top-k 块" | 对具有最高压缩分支分数的 `k` 个块的细粒度注意力 |
| Sliding window | "局部上下文" | 对最后 `W` 个 token 的注意力，用于短程模式 |
| Native trainability | "预训练时开启稀疏性" | 稀疏模式在预训练期间学习，而非推理时外挂 |
| Compression block size l | "粗粒度视图的组大小" | 多少个 token 合并为一个摘要；典型值 32-64 |
| Top-k | "保留的块数" | 读取未压缩 token 的压缩块数量；典型值 16 |
| Sliding window W | "局部注意力半径" | 通常为 512；更短损害局部连贯性，更长浪费计算 |
| Branch gate | "如何混合三者" | 逐位置 MLP 输出，加权三个分支的贡献 |
| Hardware alignment | "Kernel 友好的稀疏性" | 选择稀疏模式使得实际 GPU kernel 达到理论加速 |
| DSA | "NSA 的继任者" | Deepseek Sparse Attention，DeepSeek 系列中继 NSA 之后的架构 |
