# Anthropic 的工作流模式：简单优于复杂

> Schluntz 和 Zhang（Anthropic，2024 年 12 月）区分了工作流（预定义路径）和智能体（动态工具使用）。五种工作流模式涵盖大多数情况。从直接 API 调用开始。仅在步骤无法预测时才添加智能体。

**类型：** Learn + Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）
**时间：** ~60 分钟

## 学习目标

- 说出 Anthropic 的五种工作流模式：提示链、路由、并行化、编排器-工作器、评估器-优化器。
- 解释智能体与工作流的区别以及各自工程成本。
- 识别何时选择工作流而非智能体（反之亦然）。
- 在标准库中针对脚本化 LLM 实现所有五种模式。

## 问题

团队为只需要单个函数调用的问题选择多智能体框架。成本是真实的：框架增加了层，模糊了提示，隐藏了控制流，并引入了过早的复杂性。Schluntz 和 Zhang 2024 年 12 月的帖子是最常被引用的行业反对意见：从简单开始，仅在复杂性赢得其成本时才增加。

## 概念

### 工作流 vs 智能体

- **工作流。** 通过预定义代码路径编排的 LLM 和工具。工程师拥有图。
- **智能体。** LLM 动态指导自己的工具并采取自己的步骤。模型拥有图。

两者都有各自的适用场景。工作流更便宜、更快、更容易调试。智能体解锁了开放式问题，但使失败模式更难推理。

### 增强型 LLM

所有五种模式的基础：一个具有三种内置能力的 LLM——搜索（检索）、工具（行动）、记忆（持久化）。任何 API 调用都可以使用这些。

### 五种模式

1. **提示链。** 调用 1 的输出是调用 2 的输入。当任务有清晰的线性分解时使用。步骤之间可选程序化门控。
2. **路由。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。当分类不同的输入需要不同处理时使用（一级支持 vs 退款 vs Bug vs 销售）。
3. **并行化。** 并发运行 N 个 LLM 调用，聚合结果。两种形态：分区（不同块）和投票（相同提示，N 次运行，多数/综合）。
4. **编排器-工作器。** 一个编排器 LLM 动态决定运行哪些工作器（也是 LLM）并综合它们的输出。类似于智能体循环，但编排器不会无限循环。
5. **评估器-优化器。** 一个 LLM 提出答案，另一个 LLM 评估它。迭代直到评估器通过。这是泛化的 Self-Refine（第 05 课）。

### 工作流何时优于智能体

- **可预测的任务。** 如果你能枚举步骤，就应该这么做。
- **成本受限的任务。** 工作流有有界的步骤数；智能体可能失控。
- **合规受限的任务。** 审计员希望读取图，而不是从轨迹推断它。

### 智能体何时优于工作流

- **开放式研究。** 当下一步取决于上一步返回的结果时。
- **可变长度任务。** 几分钟到几小时的工作，步骤数未知。
- **新领域。** 当你还不知道正确的工作流时——先探索，后固化。

### 上下文工程伴侣

"有效的 AI 智能体上下文工程"（Anthropic 2025）形式化了相邻学科：200k 窗口是一个预算，不是一个容器。包含什么、何时压缩、何时让上下文增长。在本课程中关于上下文压缩的 Phase 14 课程中详细讨论（在本课程重新编号之前的 Phase 14 早期课程 06）。

## 构建

`code/main.py` 针对 `ScriptedLLM` 实现所有五种工作流模式：

- `prompt_chain(input, steps)`——顺序。
- `route(input, classifier, handlers)`——分类 + 分派。
- `parallel_vote(prompt, n, aggregator)`——N 次运行，聚合。
- `orchestrator_workers(task, workers)`——编排器选择工作器。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)`——循环直到通过。

运行：

```
python3 code/main.py
```

每种模式打印其轨迹。每种模式的代码行数约为 10-15 行；框架的成本以数千行计。

## 使用

- 大多数任务使用直接 API 调用。
- 仅当模式真正需要持久状态（LangGraph）、参与者模型并发（AutoGen v0.4）或角色模板（CrewAI）时才使用框架。
- 当你想要 Claude Code 框架形状而不重新构建它时，选择 Claude Agent SDK。

## 交付

`outputs/skill-workflow-picker.md` 为给定的任务描述选择正确的模式，包括决策理由以及在工作流不足时转向智能体的重构路径。

## 练习

1. 使用置信度阈值实现路由。低于阈值 -> 升级到人工。对于一级支持用例，阈值在哪里？
2. 为 `parallel_vote` 添加超时。当一个调用挂起时会发生什么？你如何在缺少投票的情况下聚合？
3. 将 `evaluator_optimizer` 变成赌徒：跨迭代保留 top-2 输出，这样后期的好结果不会被后期的坏结果覆盖。
4. 组合提示链与路由：一个路由器选择三个链之一。测量 token 成本与单个大提示替代方案的对比。
5. 选择你的一个生产功能。绘制工作流图。计数步骤。智能体在这里真的会更好吗？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 工作流 | "预定义流" | 工程师拥有的 LLM 和工具调用图 |
| 智能体 | "自主 AI" | 模型拥有的图；动态工具指导 |
| 增强型 LLM | "带工具的 LLM" | LLM + 搜索 + 工具 + 记忆；原子单元 |
| 提示链 | "顺序调用" | 调用 N 的输出是调用 N+1 的输入 |
| 路由 | "分类器分派" | 选择哪个链/模型处理输入 |
| 并行化 | "扇出" | N 个并发调用；通过分区或投票聚合 |
| 编排器-工作器 | "调度智能体" | 编排器 LLM 动态选择专业 LLM |
| 评估器-优化器 | "提议者 + 裁判" | 迭代直到评估器通过；泛化的 Self-Refine |
