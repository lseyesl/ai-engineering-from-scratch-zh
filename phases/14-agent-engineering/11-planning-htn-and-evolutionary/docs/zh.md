# HTN 规划与进化搜索

> 符号规划处理计划可证明正确的情况。进化代码搜索处理适应度函数可机器检查的情况。ChatHTN（2025）和 AlphaEvolve（2025）展示了各在与 LLM 配对时解锁的能力。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 02（ReWOO 与计划-执行）
**时间：** ~75 分钟

## 学习目标

- 解释层次任务网络：任务、方法、操作符、前提条件、效果。
- 描述 ChatHTN 的混合循环——符号搜索与 LLM 回退分解。
- 解释 AlphaEvolve 的进化循环以及为什么它只适用于编程评估器。
- 在标准库中实现一个玩具 HTN 规划器和一个玩具进化搜索。

## 问题

ReWOO（第 02 课）、计划-执行和 ReAct 涵盖了大多数智能体规划。两种它们无法很好覆盖的情况：

1. **具有可证明正确性的计划。** 调度、飞行路径规划、合规工作流——计划必须通过构造是合理的。流畅的 LLM 计划偶尔幻觉出步骤是不可接受的。
2. **具有机器可检查适应度函数的优化。** 矩阵乘法、调度启发式算法、编译器通行证——目标不是"一个正确的计划"，而是"最好的计划。"

HTN 规划和 AlphaEvolve 解决了这两个不同的问题。两者都将 LLM 用作放大器，而非替代品。

## 概念

### 层次任务网络

HTN 由以下组成：

- **任务**——复合（需分解）和原语（可直接执行）。
- **方法**——将复合任务分解为子任务的方法，带有前提条件。
- **操作符**——带有前提条件和效果的原始行动。
- **状态**——一组事实。

规划：给定一个目标任务和初始状态，找到分解为原语操作符的序列，其前提条件按顺序满足。

HTN 比 LLM 更古老，并且仍然是可证明正确计划的参考。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交错：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："你如何将 `task` 在状态 `s` 下分解？"
3. 将 LLM 响应翻译为候选子任务。
4. 根据操作符模式验证；拒绝无效的分解。
5. 递归。

论文的核心主张：产生的每个计划都是可证明正确的，因为 LLM 建议仅作为候选分解进入，从不作为直接计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续）添加了一个学习器，通过回归泛化 LLM 产生的分解——将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是一个不同的存在：由 Gemini 2.0 Flash/Pro 集成协调的进化代码搜索。

循环：

1. 从种子程序 + 编程评估器（返回适应度分数）开始。
2. LLM 集成提出变异。
3. 通过评估器运行变异。
4. 保留最优者；再次变异。

已发布的成就：

- 56 年来首次改进 4x4 复数矩阵乘法的 Strassen 算法（48 次标量乘法）。
- 通过 Borg 调度启发式算法恢复 0.7% Google 计算资源。
- 前沿工作负载上 32% 的 FlashAttention 加速。

硬约束：适应度函数必须是机器可检查的。对散文答案进行进化搜索不会收敛。

### 何时使用哪个

| 问题类别 | 使用 | 原因 |
|---------------|-----|------|
| 带硬约束的调度 | HTN + ChatHTN | 可证明正确性 |
| 编译器优化 | AlphaEvolve | 机器可检查适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略绑定自动化 | HTN | 前提条件编码策略 |

### 这种模式出错的地方

- **没有操作符的 HTN。** 没有前提条件/效果模式，正确性主张崩溃。ChatHTN 的"LLM 建议分解"需要模式来拒绝无效移动。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性和快速的。
- **过度工程。** 大多数智能体任务不需要两者中的任何一个。首先选择 ReAct 或 ReWOO。

## 构建

`code/main.py` 实现两个玩具：

- 一个标准库 HTN 规划器，带有操作符、方法、前提条件、效果和当没有方法匹配复合任务时触发的 `LLMFallback`。"LLM"是一个脚本化分解器，使规划器离线运行。
- 一个在算术程序上的标准库进化搜索：进化表达式，其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

轨迹显示 HTN 规划器分解复合任务（中间包含 LLM 回退）和进化循环收敛到目标表达式。

## 使用

- **HTN 规划器**——`pyhop`、`SHOP3` 或为特定领域策略强制执行构建你自己的。
- **ChatHTN**——研究代码；模式（符号 + LLM 回退）可干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；模式（集成 + 评估器）是可复现的。OpenEvolve 和类似的开源分支正在出现。
- **智能体框架**——目前没有一个提供一等 HTN 或 AlphaEvolve 支持。作为子智能体或后台工作器构建。

## 交付

`outputs/skill-hybrid-planner.md` 生成一个混合规划器框架（HTN 或进化），LLM 角色明确限定范围。

## 练习

1. 扩展 HTN 规划器以支持回溯：当操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM-方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。在下一次调用时首先检查方法库。
3. 将进化搜索评估器切换为真实测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计说明。为你关心的领域设计评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合：使用 HTN 将复合任务分解为子任务，然后在每个子任务的原语操作符上使用进化搜索。它在哪些场景表现出色，在哪些场景过度工程？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| HTN | "层次规划器" | 带有操作符、前提条件、效果的任务分解 |
| 方法 | "分解规则" | 将复合任务分解为子任务的方法 |
| 操作符 | "原语行动" | 带有前提条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 当没有方法匹配时符号规划器询问 LLM |
| AlphaEvolve | "进化代码搜索" | 集成 LLM 变异代码；确定性评估器选择 |
| 适应度函数 | "评估器" | 输出上的确定性、机器可检查得分 |
| 在线方法学习 | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
