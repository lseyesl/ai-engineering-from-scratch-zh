# 记忆块与睡眠时间计算（Letta）

> MemGPT 在 2024 年成为 Letta。2026 年的演进增加了两个想法：模型可以直接编辑的离散功能记忆块，以及在主智能体空闲时异步合并记忆的睡眠时间智能体。这就是你如何将记忆扩展到一次对话之外。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT）
**时间：** ~75 分钟

## 学习目标

- 说出 Letta 使用的三个记忆层级（核心、回忆、档案）以及每个层级的角色。
- 解释记忆块模式：Human 块、Persona 块和用户定义块作为一等类型化对象。
- 描述什么是睡眠时间计算、为什么它位于关键路径之外、以及为什么它可以运行比主智能体更强的模型。
- 实现一个脚本化的双智能体循环，其中主智能体提供响应，睡眠时间智能体在轮次之间合并块。

## 问题

MemGPT（第 07 课）解决了虚拟记忆控制流。出现了三个生产问题：

1. **延迟。** 每个记忆操作都在关键路径上。如果智能体在用户等待时必须修剪、总结或协调，尾部延迟会暴涨。
2. **记忆腐烂。** 写入累积。矛盾的事实留存。检索被过时内容淹没。
3. **结构丢失。** 平面档案存储无法表达"Human 块始终在提示中；Persona 块始终在提示中；Task 块按会话切换。"

Letta（letta.com）是 2026 年的重写。记忆块使结构显式；睡眠时间计算将合并移出关键路径。

## 概念

### 三个层级

| 层级 | 范围 | 位置 | 写入者 |
|------|-------|----------------|------------|
| 核心 | 始终可见 | 主提示内 | 智能体工具调用 + 睡眠时间重写 |
| 回忆 | 对话历史 | 可检索 | 自动轮次记录 |
| 档案 | 任意事实 | 向量 + KV + 图 | 智能体工具调用 + 睡眠时间摄取 |

核心是 MemGPT 的核心。回忆是带有被驱逐尾部的对话缓冲区。档案是外部存储。这种拆分清理了 MemGPT 两级架构的过载。

### 记忆块

块是核心层中类型化、持久、可编辑的部分。原始 MemGPT 论文定义了两种：

- **Human 块**——关于用户的事实（姓名、角色、偏好、目标）。
- **Persona 块**——智能体的自我概念（身份、语气、约束）。

Letta 泛化为任意用户定义的块：`Task` 块用于当前目标，`Project` 块用于代码库事实，`Safety` 块用于硬约束。每个块有 `id`、`label`、`value`、`limit`（字符上限）、`description`（以便模型知道何时编辑它）。

块通过工具面进行编辑：

- `block_append(label, text)`
- `block_replace(label, old, new)`
- `block_read(label)`
- `block_summarize(label)`——压缩接近上限的块。

### 睡眠时间计算

2025 年的 Letta 新增功能：在后台运行第二个智能体，不在关键路径上。睡眠时间智能体处理对话记录和代码库上下文，将 `learned_context` 写入共享块，并合并或失效档案记录。

由此产生的特性：

- **无延迟成本。** 主响应不等待记忆操作。
- **允许更强模型。** 睡眠时间智能体可以是更昂贵、更慢的模型，因为它不受延迟约束。
- **自然的合并窗口。** 在用户不等待时去重、总结、失效矛盾的事实。

这种形态与人类的工作方式相匹配：你执行任务，你睡一觉在上面，长期记忆在一夜间沉淀。

### Letta V1 与原生推理

Letta V1（`letta_v1_agent`，2026）弃用了 `send_message`/心跳和内联 `Thought:` token，转而使用原生推理。Responses API（OpenAI）和支持扩展思考的 Messages API（Anthropic）在单独信道上发出推理，跨轮次传递（在生产中跨提供商加密）。控制循环仍然是 ReAct。思考轨迹是结构化的，而不是提示形状的。

### 这种模式出错的地方

- **块膨胀。** 无限的 `block_append` 很快达到上限。在超出上限的写入之前连接一个块摘要器。
- **静默漂移。** 睡眠时间智能体重写块，主智能体从未注意到。对块进行版本控制，并在轨迹中显示差异。
- **中毒合并。** 睡眠时间智能体处理攻击者可访问的内容进入核心。第 27 课同样适用于睡眠时间表面。

## 构建

`code/main.py` 实现：

- `Block`——id、label、value、limit、description。
- `BlockStore`——CRUD + `near_limit(label)` 辅助方法。
- 两个脚本化智能体——`PrimaryAgent` 提供服务轮次，`SleepTimeAgent` 在轮次之间合并。
- 一个显示三轮对话及块写入的轨迹，加上一个压缩块并使过时事实失效的睡眠时间传递。

运行：

```
python3 code/main.py
```

记录显示拆分：主轮次快速且产生原始写入；睡眠传递进行压缩和清理。

## 使用

- **Letta**（letta.com）用于参考实现。自托管或托管云。
- **Claude Agent SDK 技能**作为块形状的知识——技能是智能体按需加载的命名、版本化、可检索的指令块。
- **自定义构建**适用于希望控制存储后端的团队。使用 Letta API 合同，以便以后可以迁移。

## 交付

`outputs/skill-memory-blocks.md` 生成一个 Letta 形状的块系统，带有睡眠时间钩子，适用于任何运行时，包括安全规则和引用连接。

## 练习

1. 添加一个 `block_summarize` 工具，当 `near_limit` 返回 true 时，将块值替换为模型生成的摘要。哪个触发阈值能最小化摘要调用次数和块溢出？
2. 在档案上实现睡眠时间去重：文本重叠超过 90% 的两条记录合并为一条。仅在睡眠传递中执行，从不在关键路径上。
3. 对块进行版本控制。每次写入记录旧值和差异。暴露 `block_history(label)` 以便操作员可以调试"为什么智能体忘记了 X。"
4. 将睡眠时间智能体视为不可信写入者。当它们触及 Persona 或 Safety 块时，在提交前需要第二个智能体的审查。
5. 将示例移植到使用 Letta API（`letta_v1_agent`）。块模式有什么变化，原生推理如何改变轨迹形态？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 记忆块 | "可编辑提示部分" | 核心记忆中类型化、持久、LLM 可编辑的部分 |
| Human 块 | "用户记忆" | 关于用户的事实，固定在核心中 |
| Persona 块 | "智能体身份" | 自我概念、语气、约束，固定在核心中 |
| 睡眠时间计算 | "异步记忆工作" | 第二个智能体在关键路径之外进行合并 |
| 核心/回忆/档案 | "层级" | 三层记忆拆分：始终可见/对话/外部 |
| 块上限 | "上限" | 每块字符上限；强制总结 |
| 原生推理 | "思考信道" | 提供商级别的推理输出，而非提示级别的 `Thought:` |
| 学习上下文 | "睡眠输出" | 睡眠时间智能体写入共享块的事实 |
