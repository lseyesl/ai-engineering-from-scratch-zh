# 智能体可观测性：Langfuse、Phoenix、Opik

> 三个开源智能体可观测性平台在 2026 年占据主导。Langfuse（MIT）——每月 600 万+ 安装量，追踪 + 提示管理 + 评估 + 会话回放。Arize Phoenix（Elastic 2.0）——深入的智能体特定评估、RAG 相关性、OpenInference 自动仪表化。Comet Opik（Apache 2.0）——自动提示优化、护栏、LLM 裁判幻觉检测。

**类型：** Learn
**语言：** Python（标准库）
**前置知识：** Phase 14 · 23（OTel GenAI）
**时间：** ~45 分钟

## 学习目标

- 说出三个顶级开源智能体可观测性平台及其许可证。
- 区分每个平台最强的领域：Langfuse（提示管理 + 会话）、Phoenix（RAG + 自动仪表化）、Opik（优化 + 护栏）。
- 解释为什么到 2026 年 89% 的组织报告已具备智能体可观测性。
- 实现一个带有 LLM 裁判评估的标准库追踪到仪表板的管道。

## 问题

OTel GenAI（第 23 课）为你提供了模式。你仍然需要摄取跨度、运行评估、存储提示版本和显示回归的平台。三个竞争者各自强调生命周期的不同部分。

## 概念

### Langfuse（MIT）

- 每月 600 万+ SDK 安装量，19k+ GitHub 星标。
- 功能：追踪、带版本控制和游乐场的提示管理、评估（LLM-as-judge、用户反馈、自定义）、会话回放。
- 2025 年 6 月：以前商业模块（LLM-as-a-judge、注释队列、提示实验、Playground）在 MIT 下开源。
- 最强领域：端到端可观测性，配合紧密的提示管理循环。

### Arize Phoenix（Elastic License 2.0）

- 更深入的智能体特定评估：轨迹聚类、异常检测、RAG 检索相关性。
- 原生 OpenInference 自动仪表化。
- 与托管 Arize AX 配对用于生产。
- 没有提示版本控制——定位于漂移/行为回归工具，配合更广泛的平台。
- 最强领域：RAG 相关性、行为漂移、异常检测。

### Comet Opik（Apache 2.0）

- 通过 A/B 实验进行自动提示优化。
- 护栏（PII 编辑、主题约束）。
- LLM 裁判幻觉检测。
- Comet 自己的测量基准：Opik 记录 + 评估在 23.44 秒内 vs Langfuse 327.15 秒（约 14 倍差距）——将供应商基准仅视为方向性参考。
- 最强领域：优化循环、自动实验、护栏执行。

### 行业数据

根据 Maxim（2026 年实地分析）：89% 的组织已具备智能体可观测性；质量问题是最主要的生产障碍（32% 的受访者引用）。

### 选择一个

| 需求 | 选择 |
|------|------|
| 全能型带提示管理 | Langfuse |
| 深度 RAG 评估 + 漂移 | Phoenix |
| 自动优化 + 护栏 | Opik |
| 开放许可，无 ELv2 | Langfuse（MIT）或 Opik（Apache 2.0） |
| Datadog / New Relic 集成 | 任意——它们都导出 OTel |

### 这种模式出错的地方

- **没有评估策略。** 没有评估的追踪只是昂贵的日志记录。
- **自制的无根据 LLM 裁判。** CRITIC 模式（第 05 课）适用——裁判需要用于事实验证的外部工具。
- **提示版本未关联到追踪。** 当生产出现回归时，你无法二分到引起问题的提示。

## 构建

`code/main.py` 实现一个标准库追踪收集器 + LLM 裁判评估器：

- 摄取 GenAI 形状的跨度。
- 按会话分组，标记失败运行（护栏触发、低置信度评估）。
- 一个根据评分标准对智能体响应评分的脚本化 LLM 裁判。
- 一个类似仪表板的摘要：失败率、主要失败原因、评估分数分布。

运行：

```
python3 code/main.py
```

输出：每个会话的评估分数和失败分类，匹配 Langfuse/Phoenix/Opik 会显示的内容。

## 使用

- **Langfuse** 自托管或云；通过 OTel 或其 SDK 连接。
- **Arize Phoenix** 自托管；自动仪表化 OpenInference。
- **Comet Opik** 自托管或云；自动优化循环。
- **Datadog LLM 可观测性** 用于已经运行 Datadog 的混合运维+ML 团队。

## 交付

`outputs/skill-obs-platform-wiring.md` 选择一个平台，并将追踪 + 评估 + 提示版本连接到现有智能体。

## 练习

1. 导出一周的 OTel 追踪到 Langfuse 云（免费层）。哪些会话失败了？为什么？
2. 为你的领域编写 LLM 裁判评分标准（事实正确性、语气、范围遵守）。在 50 个追踪上测试。
3. 比较 Langfuse 提示版本控制与 Phoenix 的轨迹聚类。哪个更快告诉你什么坏了？
4. 阅读 Opik 的护栏文档。将 PII 编辑护栏连接到你的一个智能体运行。
5. 在你的语料库上对三者进行基准测试。忽略供应商发布的数字；测量你自己的。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 追踪 | "跨度收集器" | 摄取 OTel / SDK 跨度；按会话索引 |
| 提示管理 | "提示 CMS" | 与追踪关联的版本化提示 |
| LLM-as-judge | "自动评估" | 单独的 LLM 根据评分标准对智能体输出评分 |
| 会话回放 | "轨迹回放" | 逐步遍历过去的运行以进行调试 |
| RAG 相关性 | "检索质量" | 检索到的上下文是否匹配查询 |
| 轨迹聚类 | "行为分组" | 聚类类似运行以检测漂移 |
| 护栏执行 | "日志时的策略" | 对记录内容的 PII/毒性/范围检查 |
