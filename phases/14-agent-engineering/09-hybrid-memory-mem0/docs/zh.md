# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikara 等人，2025）将记忆视为三个并行存储——向量用于语义相似度，KV 用于快速事实查找，图用于实体关系推理。一个评分层在检索时融合三者。这是 2026 年外部记忆的生产标准。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT），Phase 14 · 08（Letta 块）
**时间：** ~75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅 KV）对于智能体记忆是不够的。
- 说出 Mem0 的三个并行存储以及每个存储优化的目标。
- 描述 Mem0 的融合评分——相关性、重要性、时效性——以及为什么它是加权和而非层次结构。
- 在标准库中实现一个玩具三存储记忆，带有写入所有三个存储的 `add()` 和融合结果的 `search()`。

## 问题

单一存储对于三种查询类型中的至少两种是错误的：

- **语义相似度**——"我们上周讨论了关于智能体漂移的什么？"向量胜出；KV 和图无法匹配。
- **事实查找**——"用户的电话号码是什么？"KV 胜出；向量是浪费，图是过度设计。
- **关系推理**——"哪些客户共享同一计费实体？"图胜出；向量和 KV 无法回答。

生产智能体在一个会话中发出所有三种查询。单一存储记忆总是对其中两种是错误的。Mem0 的贡献是在单个 `add`/`search` 接口后连接三者，并使用评分函数进行融合。

## 概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 上：

1. 从文本中提取候选事实（一个 LLM 驱动的步骤）。
2. 将每个事实写入向量存储（嵌入）用于语义搜索。
3. 将每个事实写入以 `(user_id, fact_type, entity)` 为键的 KV 存储，用于 O(1) 查找。
4. 将每个事实写入图存储（Mem0g）作为类型化边，用于关系查询。

在 `search(query, user_id)` 上：

1. 向量存储通过嵌入余弦相似度返回 top-k。
2. KV 存储返回基于查询派生的 `(user_id, type, entity)` 键的直接命中。
3. 图存储返回从查询实体可达的子图。
4. 一个评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性**——向量余弦、KV 精确匹配、图路径权重。
- **重要性**——在写入时标记或学习（某些事实更重要：姓名、ID、策略）。
- **时效性**——自上次写入或读取以来的指数衰减。

权重按产品调整。聊天智能体使用更高的 `w_recency`；合规智能体使用更高的 `w_importance`；检索智能体使用更高的 `w_relevance`。

### Mem0g 与时序推理

Mem0g 添加了一个冲突检测器。当新事实与现有边矛盾时，现有边被标记为无效但不删除。时序查询（"用户在 3 月的城市是什么？"）遍历有效时间子图。

这是 Letta 的失效模式泛化后的合规级行为。

### 基准数字

Mem0 论文报告（2025）：

- **LoCoMo**（长形式对话记忆）：91.6
- **LongMemEval**（长视界情景记忆）：93.4
- **BEAM 1M**（100 万 token 记忆基准）：64.1

比较基线（128k 全上下文 LLM、平面向量存储、平面 KV）都输 10+ 分。仅凭基准不足以证明选择合理性——操作形态才是关键——但数字表明融合设计不是四舍五入误差。

### 范围分类

Mem0 按范围拆分记忆：

- **用户记忆**——跨会话持久化，以 `user_id` 为键。
- **会话记忆**——在一个线程内持久化。
- **智能体记忆**——每个智能体实例的状态。

每次写入选择一个范围。检索可以按范围权重查询跨范围的内容。不加思考地混合范围是导致"助手告诉 Alice 关于 Bob 的项目"事件的原因。

### 这种模式出错的地方

- **嵌入漂移。** 在前一百次查询中看起来正确的向量结果随着语料库增长而退化。添加对使用最多的 N 条记录的定期重新嵌入。
- **KV 模式蔓延。** `(user_id, type, entity)` 看起来很简单，直到每个团队添加自己的 `type`。每季度审计类型集。
- **图爆炸。** 一个嘈杂的提取器每条消息添加 50 条边。限制每次 `add` 调用的图写入；丢弃低置信度边。

## 构建

`code/main.py` 在标准库中实现三存储模式：

- `VectorStore`——作为嵌入替代的朴素 token 重叠相似度。
- `KVStore`——以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore`——类型化边（subject, relation, object, valid）。
- `Mem0`——顶层外观，带有 `add()`、`search()`、融合评分和范围感知检索。
- 一个在多用户、多会话对话上的工作轨迹。

运行：

```
python3 code/main.py
```

输出显示三个独立的召回路径加上融合的 top-k。在 `main()` 顶部翻转评分权重，观察排序的变化。

## 使用

- **Mem0（Apache 2.0）**——生产就绪。使用 Postgres + Qdrant + Neo4j 自托管，或使用托管云。
- **Letta**——三级核心/回忆/档案；自带向量和图后端。
- **Zep**——具有时序 KG 和事实提取的商业替代品。
- **自定义构建**——当你需要精确控制提取器（合规）或融合权重（时效性占主导的语音智能体）时。

## 交付

`outputs/skill-hybrid-memory.md` 生成一个三存储记忆框架，包含融合评分器、范围分类和时序失效功能。

## 练习

1. 将玩具向量相似度替换为真实嵌入模型（sentence-transformers、Ollama、OpenAI embeddings）。在合成长对话上测量 recall@10。经过 1000 次写入后排序漂移吗？
2. 添加时序查询：`search(query, as_of=timestamp)`。仅返回在该时间或之前有效的记录。哪个存储需要最多工作？
3. 实现冲突检测器：如果传入事实与图边矛盾，使旧边失效并记录两者。在"用户住在柏林" -> "用户住在里斯本"上测试。
4. 将融合评分器扩展为包含 `user_feedback` 维度（对检索记录的点赞）。如何防止作弊（智能体只返回它已经喜欢的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。将玩具移植到 `mem0` 客户端调用。在相同的 20 个测试查询上比较检索质量。
