# 故障模式：智能体为何崩溃

> MASFT（伯克利，2025）将 14 种多智能体故障模式归类为 3 个类别。微软的分类法记录了现有 AI 故障如何在智能体环境中被放大。行业现场数据收敛到五种反复出现的模式：幻觉行动、范围蔓延、级联错误、上下文丢失、工具误用。

**类型：** Learn + Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 05（自我精炼与 CRITIC），Phase 14 · 24（可观测性）
**时间：** ~60 分钟

## 学习目标

- 说出 MASFT 的三个故障类别以及每个类别中至少四种具体模式。
- 解释为什么智能体故障会放大现有的 AI 故障模式（偏见、幻觉）。
- 描述五种行业反复出现的模式及其缓解措施。
- 实现一个标准库检测器，用故障模式标签标记智能体轨迹。

## 问题

团队交付的智能体在 90% 的轨迹上工作正常。那 10% 的失败并非随机噪声——它们属于少数反复出现的类别。一旦你能命名它们，你就可以监控它们并修复它们。

## 概念

### MASFT（伯克利，arXiv:2503.13657）

多智能体系统故障分类法。14 种故障模式聚类为 3 个类别。标注者间 Cohen's Kappa 0.88——这些类别是可靠可区分的。

核心主张：故障是多智能体系统中的根本设计缺陷，而不是可以通过更好的基础模型修复的 LLM 限制。

### 微软的智能体 AI 系统故障模式分类法

- 现有的 AI 故障（偏见、幻觉、数据泄露）在智能体环境中被放大。
- 新的故障从自主性中涌现：大规模意外行动、工具误用、任务漂移。
- 该白皮书是智能体产品的风险登记册。

### 智能体 AI 中的故障特征描述（arXiv:2603.06847）

- 故障源于编排、内部状态演化和环境交互。
- 不仅仅是"糟糕的代码"或"糟糕的模型输出"。

### LLM 智能体幻觉综述（arXiv:2509.18970）

两种主要表现：

1. **指令遵循偏差**——智能体不遵循系统提示。
2. **长距离上下文误用**——智能体忘记或误用了早期轮次的上下文。

子意图错误：遗漏（错过步骤）、冗余（重复步骤）、无序（步骤顺序错误）。

### 五种行业反复出现的模式

Arize、Galileo、NimbleBrain 2024-2026 年的现场分析收敛到：

1. **幻觉行动。** 智能体调用不存在的工具或编造参数。
2. **范围蔓延。** 智能体将任务扩展到用户要求之外（创建额外的 PR、发送额外的邮件）。
3. **级联错误。** 一个错误的调用触发下游效应。一个幻影 SKU 幻觉触发四个 API 调用——一个多系统事故。
4. **上下文丢失。** 长周期任务忘记早期轮次的约束。
5. **工具误用。** 用错误的参数调用正确的工具，或者完全调用错误的工具。

级联是最致命的。智能体无法区分"我失败了"和"任务不可能完成"，并且经常在 400 错误上幻觉成功消息以关闭循环。

### 缓解措施：每一步都设置关卡

在推理链的每一步设置自动验证关卡，对照环境状态检查事实基础。具体来说：

- 每步安全分类器（第 21 课）。
- 工具调用参数验证（第 06 课）。
- 将检索到的内容与已知事实进行交叉检查（第 05 课，CRITIC）。
- 通过重新探测状态来检测成功幻觉（文件是否真的被创建了？）。

### 故障监控在何处出错

- **仅标记崩溃。** 大多数智能体故障产生看似有效的输出。需要内容级别的检查。
- **没有基线。** 漂移检测需要一个已知的良好状态；没有它，你无法说"情况正在恶化"。
- **过度告警。** 每个故障都产生一个页面。需要聚类和限速。

## 构建

`code/main.py` 实现了一个标准库故障模式标记器：

- 一个覆盖五种模式的合成轨迹数据集。
- 每种模式的检测器函数（工具调用、输出、重复动作上的签名模式）。
- 一个标记器，为每个轨迹打标签并报告模式分布。

运行：

```
python3 code/main.py
```

输出：每个轨迹的标签 + 聚合分布，这是 Phoenix 轨迹聚类功能的廉价复现。

## 使用

- **Phoenix**用于生产环境中的漂移聚类（第 24 课）。
- **Langfuse**用于会话回放 + 标注。
- **自定义**用于你的可观测性平台无法检测的领域特定签名。

## 交付

`outputs/skill-failure-detector.md` 生成针对你的领域定制的故障模式检测器，并连接到轨迹存储。

## 练习

1. 添加一个"成功幻觉"检测器：智能体返回成功但目标状态未改变。
2. 标记你构建过的产品的 100 条真实轨迹。哪种模式占主导？修复它的成本是多少？
3. 实现一个"级联半径"指标：给定第 N 步的一个故障，它影响了多少下游步骤？
4. 阅读 MASFT 的 14 种故障模式。选择三种适用于你产品的模式。编写检测器。
5. 将一个检测器接入 CI 任务：如果 >=5% 的轨迹标记了某种模式，则构建失败。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| MASFT | "多智能体故障分类法" | 伯克利 14 模式分类 |
| Cascading error | "涟漪故障" | 一个早期错误通过 N 步传播 |
| Context loss | "忘记了约束" | 长周期轮次丢失早期轮次的事实 |
| Tool misuse | "错误的工具/错误的参数" | 有效调用，错误的调用方式 |
| Success hallucination | "伪造完成" | 智能体在 400 错误上声称成功；状态未改变 |
| Scope creep | "越界" | 智能体做了超出要求的事情 |
| Instruction-following deviation | "不服从" | 忽略系统提示或用户约束 |
| Sub-intention errors | "计划缺陷" | 计划执行中的遗漏、冗余、无序 |
