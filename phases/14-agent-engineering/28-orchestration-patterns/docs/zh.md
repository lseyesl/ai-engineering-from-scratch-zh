# 编排模式：监督者、群体、分层 (Orchestration Patterns: Supervisor, Swarm, Hierarchical)

> 四种编排模式在 2026 年的框架中反复出现：监督者-工作者、群体/点对点、分层、辩论。Anthropic 的指导原则："关键在于为你的需求构建正确的系统。"从简单开始；只有当单个智能体加上五种工作流模式不够用时，才增加拓扑结构。

**类型：** Learn + Build
**语言：** Python (stdlib)
**前置要求：** Phase 14 · 12（工作流模式），Phase 14 · 25（多智能体辩论）
**时间：** ~60 分钟

## 学习目标

- 列举四种反复出现的编排模式以及各自适用的场景。
- 描述 2026 年 LangChain 的建议：基于工具调用的监督 vs 监督者库。
- 解释 Anthropic 的"构建正确的系统"原则以及它如何决定拓扑选择。
- 在 stdlib 中针对一个脚本化的 LLM 实现所有四种模式。

## 问题

团队在真正需要之前就急于采用"多智能体"。四种模式在不同框架中反复出现；一旦你能命名它们，就能选择正确的模式——或者完全跳过拓扑。

## 概念

### 监督者-工作者

- 一个中央路由 LLM 将任务分派给专业智能体。
- 决策：回环到自身、移交给专业智能体、终止。
- 专业智能体之间不直接通信；所有路由都通过监督者。

框架：LangGraph `create_supervisor`、Anthropic 编排者-工作者、CrewAI 分层流程。

**2026 年 LangChain 建议：** 通过直接工具调用来实现监督，而不是使用 `create_supervisor`。这样可以实现更精细的上下文工程控制——你决定每个专业智能体看到什么。

### 群体/点对点

- 智能体通过共享的工具表面直接移交。
- 没有中央路由器。
- 延迟低于监督者模式（跳数更少）。
- 更难推理（没有单一控制点）。

框架：LangGraph 群体拓扑、OpenAI Agents SDK 移交（当所有智能体都可以相互移交时）。

### 分层

- 监督者管理子监督者，子监督者管理工作者。
- 在 LangGraph 中实现为嵌套子图；在 CrewAI 中实现为嵌套团队。
- 可扩展到大量智能体群体，代价是操作复杂性增加。

何时需要：当单个监督者的上下文预算无法容纳所有专业智能体的描述时。

### 辩论

- 并行提议者 + 迭代交叉评审（第 25 课）。
- 严格来说不是编排——更像是验证——但在框架中作为拓扑选择出现。

### CrewAI 团队 vs 流程

CrewAI 形式化了两种部署模式：

- **流程**用于确定性事件驱动自动化（推荐的生产起点）。
- **团队**用于自主的基于角色的协作。

这与上述四种模式正交，但映射到拓扑：流程通常是监督者或分层；团队通常是带有 LLM 路由器的监督者。

### Anthropic 的指导原则

"在 LLM 领域的成功不在于构建最复杂的系统。而在于为你的需求构建正确的系统。"

决策顺序：

1. 单个智能体 + 工作流模式（第 12 课）——从这里开始。
2. 监督者-工作者——当你拥有 2-4 个专业智能体时。
3. 群体——当延迟比推理清晰度更重要时。
4. 分层——仅当监督者上下文预算不足时。
5. 辩论——当准确性比成本更重要时。

### 这种模式出错的地方

- **拓扑优先思维。** "我们需要多智能体"——在确定多智能体解决什么问题之前。
- **群体中的来回移交。** A -> B -> A -> B。使用跳数计数器。
- **虚假分层。** 因为"企业级"而建三层；实际只有两个团队。合并。

## 构建它

`code/main.py` 在 stdlib 中针对脚本化的 LLM 实现了所有四种模式：

- `Supervisor`——中央路由器。
- `Swarm`——点对点直接移交。
- `Hierarchical`——监督者的监督者。
- `Debate`——并行提议者 + 评审。

每种模式处理相同的三意图任务（退款/缺陷/销售）。追踪形状各不相同。

运行它：

```
python3 code/main.py
```

输出：每种模式的追踪 + 操作计数。监督者最清晰；群体最短；分层最深；辩论最昂贵。

## 使用它

- **LangGraph**用于监督者和分层（嵌套子图）。
- **OpenAI Agents SDK**用于作为工具的移交（监督者形态）。
- **CrewAI Flow**用于生产环境确定性流程。
- **自定义**用于辩论或当你需要精确控制时。

`outputs/skill-orchestration-picker.md` 选择一种拓扑并实现它。

## 练习

1. 通过移除路由器将监督者-工作者转换为群体。什么被破坏了？什么改进了？
2. 为群体添加跳数计数器：3 次移交后拒绝。它能捕获 A->B->A 的来回弹跳吗？
3. 为一个有 12 个专业智能体的领域构建一个两层分层系统。如果不使用嵌套，上下文预算在哪里失败？
4. 对四种模式在生产形态的工作负载上进行性能分析。哪种模式在哪个指标上胜出（延迟、成本、准确性、可调试性）？
5. 阅读 Anthropic 的"构建有效智能体"文章。将你的每个生产流程映射到四种模式之一。有没有无法清晰映射的？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| Supervisor-worker | "路由器 + 专业智能体" | 中央 LLM 分派给专业智能体；它们之间不直接通信 |
| Swarm | "点对点" | 通过共享工具直接移交；没有中央路由器 |
| Hierarchical | "监督者的监督者" | 用于大规模群体的嵌套子图 |
| Debate | "提议者 + 评审" | 并行提议者，交叉评审（第 25 课） |
| Tool-call-based supervision | "无需库的监督者" | 通过直接工具调用实现监督者，以获得上下文控制 |
| Crew | "自主团队" | CrewAI 的基于角色的协作模式 |
| Flow | "确定性工作流" | CrewAI 的事件驱动生产模式 |
