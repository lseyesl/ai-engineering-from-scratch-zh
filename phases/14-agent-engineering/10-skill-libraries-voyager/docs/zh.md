# 技能库与终身学习（Voyager）

> Voyager（Wang 等人，TMLR 2024）将可执行代码视为技能。技能是可命名、可检索、可组合、并通过环境反馈进行优化的。这是 Claude Agent SDK 技能、skillkit 和 2026 年技能库模式的参考架构。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT），Phase 14 · 08（Letta 块）
**时间：** ~75 分钟

## 学习目标

- 说出 Voyager 的三个组件——自动课程、技能库、迭代提示——以及每个组件的角色。
- 解释为什么 Voyager 将行动空间设为代码，而不是原始命令。
- 在标准库中实现一个包含注册、检索、组合和失败驱动优化的技能库。
- 将 Voyager 的模式映射到 2026 年 Claude Agent SDK 技能和 skillkit 生态系统。

## 问题

在每个会话中从头重建每个能力的智能体做了三件错事：

1. **浪费 token。** 每个任务重新引出相同的推理。
2. **丢失进度。** 在会话 A 中学到的修正不会转移到会话 B。
3. **在长视界组合上失败。** 复杂任务需要能力层次结构；一次性提示无法表达它们。

Voyager 的回答：将每个可复用能力视为存储在库中的命名代码块，通过相似性检索，可与其他技能组合，并通过执行反馈进行优化。

## 概念

### 三个组件

Voyager（arXiv:2305.16291）围绕以下组件构建智能体：

1. **自动课程。** 一个好奇心驱动的提议者基于智能体当前的技能集和环境状态选择下一个任务。探索是自底向上的。
2. **技能库。** 每个技能是可执行代码。任务成功时添加新技能。通过查询到描述的相似性检索技能。
3. **迭代提示机制。** 失败时，智能体接收执行错误、环境反馈和自我验证输出，然后优化技能。

Minecraft 评估（Wang 等人，2024）：与基线相比，3.3 倍更多独特物品、8.5 倍更快石制工具、6.4 倍更快铁制工具、2.3 倍更长的地图遍历。数字是 Minecraft 特定的，但模式是通用的。

### 行动空间 = 代码

大多数智能体发出原始命令。Voyager 发出 JavaScript 函数。一个技能是：

```
async function craftIronPickaxe(bot) {
  await mineIron(bot, 3);
  await mineStick(bot, 2);
  await placeCraftingTable(bot);
  await craft(bot, 'iron_pickaxe');
}
```

由子技能组合而成。以描述和嵌入为键存储。作为程序而非提示进行检索。

这就是 2026 年 Claude Agent SDK 技能：一个命名、可检索的代码块加指令，智能体按需加载。

### 技能检索

新任务"制作钻石镐。"智能体：

1. 嵌入任务描述。
2. 查询技能库获取 top-k 相似技能。
3. 检索 `craftIronPickaxe`、`mineDiamond`、`placeCraftingTable` 等。
4. 从检索到的原语和新逻辑中组合新技能。

这就是 MCP 资源（Phase 13）和 Agent SDK 技能实现的模式：对知识/代码表面的检索，限定在当前任务范围内。

### 迭代优化

Voyager 的反馈循环：

1. 智能体编写技能。
2. 技能对环境运行。
3. 三种信号之一返回：`success`、`error`（带堆栈跟踪）、`self-verification failure`。
4. 智能体使用信号作为上下文重写技能。
5. 循环直到成功或达到最大轮次。

这是 Self-Refine（第 05 课）应用于代码生成，具有基于环境的验证。CRITIC（第 05 课）是相同模式，使用外部工具作为验证器。

### 课程与探索

Voyager 的课程模块基于智能体拥有什么和尚未做什么提出类似"在湖边建个庇护所"的任务。提议者使用环境状态和技能清单来选择略高于当前能力的任务——这是探索的最佳点。

对于生产智能体，这转化为一个"缺少什么"的操作符：给定当前技能库和领域，我们尚未覆盖哪些技能？团队通常作为课程审查来手动实现此功能。

### 这种模式出错的地方

- **技能库腐烂。** 相同技能使用略有不同的描述添加 10 次。在写入时添加去重；检索只返回一个。
- **组合技能漂移。** 父技能依赖于已被优化的子技能。对技能进行版本控制；固定在 v1 的父技能不会神奇地获取 v3。
- **检索质量。** 当技能库增长到几百个以上时，基于向量的技能描述检索会退化。补充标签过滤器和硬约束（"仅 `category=tooling` 的技能"）。

## 构建

`code/main.py` 实现一个标准库技能库：

- `Skill`——名称、描述、代码（作为字符串）、版本、标签、依赖关系。
- `SkillLibrary`——注册、搜索（token 重叠）、组合（依赖关系的拓扑排序）和优化（更新时版本增加）。
- 一个脚本化智能体注册三个原语技能，组合第四个，遇到失败，并进行优化。

运行：

```
python3 code/main.py
```

轨迹显示库写入、检索、组合、失败执行和 v2 优化——端到端的 Voyager 循环。

## 使用

- **Claude Agent SDK 技能**（Anthropic）——2026 年参考：每个技能有描述、代码和指令；在智能体会话期间按需加载。
- **skillkit**（npm: skillkit）——跨智能体技能管理，支持 32+ AI 编码智能体。
- **自定义技能库**——领域特定（数据智能体的 SQL 技能、基础设施智能体的 Terraform 技能）。Voyager 模式可向下扩展。
- **OpenAI Agents SDK `tools`**——在低端；每个工具是一个轻量级技能。

## 交付

`outputs/skill-skill-library.md` 生成一个 Voyager 形状的技能库，包含注册、检索、版本控制和优化功能，适用于任何目标运行时。

## 练习

1. 在 `compose()` 中添加依赖循环检测器。当技能 A 依赖 B，B 又依赖 A 时会发生什么？错误 vs 警告？
2. 实现每技能版本固定。当父技能组合子 `crafting@1` 时，`crafting@2` 的优化不得静默升级父技能。
3. 将 token 重叠检索替换为 sentence-transformers 嵌入（或 BM25 标准库实现）。在 50 技能玩具库上测量 retrieval@5。
4. 添加一个"课程"智能体：给定当前库和领域描述，提出 5 个缺失技能。每周调用一次。
5. 阅读 Anthropic 的 Claude Agent SDK 技能文档。将玩具库移植到 SDK 的技能模式。关于可发现性有什么变化？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 技能 | "可复用能力" | 命名的代码块 + 描述，通过相似性可检索 |
| 技能库 | "智能体"如何做"的记忆" | 技能的持久存储，可搜索和可组合 |
| 课程 | "任务提议者" | 由当前能力差距驱动的自底向上目标生成器 |
| 组合 | "技能 DAG" | 技能调用技能；执行时按拓扑排序 |
| 迭代优化 | "自我修正循环" | 环境反馈 + 错误 + 自我验证折叠回下一版本 |
| 行动空间即代码 | "程序化行动" | 发出函数而非原始命令，实现时间扩展行为 |
| 写入时去重 | "技能合并" | 近似重复的描述合并为一个规范技能 |
