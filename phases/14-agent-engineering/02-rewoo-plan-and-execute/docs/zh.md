# ReWOO 与计划-执行：解耦规划

> ReAct 将思考和行动交错在一个流中。ReWOO 将它们分离：首先制定一个大计划，然后执行。token 减少 5 倍，HotpotQA 准确率提升 +4%，并且你可以将规划器蒸馏到 7B 模型中。计划-执行将其泛化；计划-行动将其扩展到网络导航。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）
**时间：** ~60 分钟

## 学习目标

- 解释为什么 ReWOO 的规划器/工作器/求解器拆分相比 ReAct 的交错循环节省 token 并提高鲁棒性。
- 实现一个计划 DAG、依赖顺序执行器以及组合工作器输出的求解器——全部使用标准库。
- 使用 2026 年的"五种工作流模式"框架（Anthropic），判断任务应该以计划-执行还是交错 ReAct 的方式运行。
- 识别何时需要计划-行动的合成计划数据来处理长视界网络或移动任务。

## 问题

ReAct 的交错思考-行动-观察循环简单灵活，但每次工具调用都必须携带完整的先前上下文——包括每个之前的思考。Token 使用随深度呈二次增长。更糟的是：当工具在循环中途失败时，模型必须从错误观察中重新推导整个计划。

ReWOO（Xu 等人，arXiv:2305.18323，2023 年 5 月）注意到了这一点并做出了一个选择：提前规划好一切，并行获取证据，最后组合答案。一次 LLM 调用来规划，N 次工具调用来获取证据（可以并行），一次 LLM 调用来求解。代价是灵活性降低（计划是静态的），但获得了更好的 token 效率和更清晰的失败模式。

## 概念

### 三个角色

```
Planner:  user_question -> [plan_dag]
Workers:  [plan_dag]     -> [evidence]        (工具调用，可能并行)
Solver:   user_question, plan_dag, evidence -> final_answer
```

规划器产生一个 DAG。每个节点命名一个工具、其参数以及它依赖的早期节点（如 `#E1`、`#E2` 的引用）。工作器按拓扑顺序执行节点。求解器将所有内容拼接在一起。

### 为什么 token 减少 5 倍

ReAct 的提示长度随步骤数线性增长。在第 10 步，提示包含思考 1 加行动 1 加观察 1 加思考 2 加行动 2 加观察 2，依此类推。每个中间步骤还冗余包含原始提示。

ReWOO 支付一个规划器提示（大），N 个小工作器提示（每个只是工具调用，无链），和一个求解器提示。在 HotpotQA 上，论文测量的 token 减少了约 5 倍，同时准确率绝对提升了 +4。

### 为什么更鲁棒

如果工作器 3 在 ReAct 中失败，循环必须在流中推理出错误。在 ReWOO 中，工作器 3 返回一个错误字符串；求解器在上下文中看到它，与原始计划一起，可以优雅地降级。失败定位是按节点而非按步骤的。

### 规划器蒸馏

论文的第二个结果：因为规划器看不到观察，你可以在来自 175B 教师模型的规划器输出上微调一个 7B 模型。小模型处理规划；大模型在推理时不需要。这现在已成为标准——许多 2026 年生产智能体使用小规划器和执行器，或者反过来。

### 计划-执行（LangChain，2023）

LangChain 团队 2023 年 8 月的帖子将 ReWOO 泛化为一个模式名称：计划-执行。前置规划器发出步骤列表，执行器运行每个步骤，可选的重新规划器可以在观察结果后进行修订。这比 ReWOO 更接近 ReAct（重新规划器将观察带回规划），但保留了 token 节省。

### 计划-行动（Erdogan 等人，arXiv:2503.09572，ICML 2025）

计划-行动将模式扩展到长视界网络和移动智能体。关键贡献是合成计划数据：一个带标签的轨迹生成器产生计划明确可见的训练数据。用于微调规划器模型，使其在 WebArena 类任务中能在 30-50 步以上保持有效，而单个 ReAct 轨迹会失去连贯性。

### 何时选择哪个

| 模式 | 适用场景 |
|---------|------|
| ReAct | 短任务、未知环境、需要反应式异常处理 |
| ReWOO | 具有已知工具的结构化任务、对 token 敏感、可并行获取证据 |
| 计划-执行 | 类似 ReWOO，但在部分执行后具有重新规划能力 |
| 计划-行动 | 长视界（>30 步）、网络/移动/计算机使用 |
| 思维树 | 搜索值得付出代价时（第 04 课） |

Anthropic 2024 年 12 月的指南：从最简单的开始。如果任务是一个工具调用加一个摘要，不要构建 ReWOO。如果任务是 40 步的研究任务，不要单独使用 ReAct。

## 构建

`code/main.py` 实现了一个玩具 ReWOO：

- `Planner`——一个脚本化策略，从提示中发出计划 DAG。
- `Worker`——通过注册表分派每个节点的工具调用。
- `Solver`——脚本化组合，读取证据并生成最终答案。
- 依赖解析——像 `#E1` 这样的引用被替换为早期工作器的输出。

演示回答"法国的首都的人口是多少（四舍五入到百万）？"使用两步计划：(1) 查找首都，(2) 查找人口，然后求解。

运行：

```
python3 code/main.py
```

轨迹首先显示完整的计划，然后显示工作器结果，最后显示求解器组合。将 token 计数（我们打印粗略的字符计数）与 ReAct 风格的交错运行进行比较——ReWOO 在这种结构化任务上胜出。

## 使用

LangGraph 以配方形式提供计划-执行（`create_react_agent` 用于 ReAct，自定义图用于计划-执行）。CrewAI 的 Flows 直接编码该模式：你预先定义任务，Flow DAG 执行它们。计划-行动的合成数据方法仍主要是研究；运行时模式（显式计划 DAG）通过 LangGraph 和 CrewAI Flows 进入生产。

## 交付

`outputs/skill-rewoo-planner.md` 根据用户请求生成 ReWOO 计划 DAG，给定工具目录。它在交给执行器之前验证计划（无环、每个引用已解析、每个工具存在）。

## 练习

1. 并行化独立计划节点的工作器执行。在具有 2 个并行组的 6 节点 DAG 上，它能带来什么好处？
2. 如果任何工作器返回错误，添加一个触发的重新规划器节点。对 ReWOO 的最小更改是什么使其成为计划-执行？
3. 将 `Planner` 替换为小模型（7B 类），将 `Solver` 保留在前沿模型上。比较端到端质量——拆分在哪里失败？
4. 阅读 ReWOO 论文第 4 节关于规划器蒸馏的内容。从概念上复现 175B -> 7B 的结果：你需要什么训练数据，以及如何评分计划质量？
5. 将玩具移植到计划-行动的轨迹形状：计划是一个序列而不是 DAG。哪些权衡发生了变化？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| ReWOO | "无观察的推理" | 规划，然后并行获取证据，然后求解——规划提示中没有观察 |
| 计划-执行 | "LangChain 的计划-执行模式" | ReWOO 加上执行后可选的重新规划器节点 |
| 计划-行动 | "扩展的计划-执行" | 显式规划器/执行器拆分，用于长视界任务的合成计划训练数据 |
| 证据引用 | "#E1, #E2, ..." | 在分派时用先前工作器输出替换的计划节点占位符 |
| 规划器蒸馏 | "小规划器，大执行器" | 在大教师模型的规划器轨迹上微调小模型 |
| Token 效率 | "更少的往返次数" | 在 HotpotQA 上比 ReAct 减少 5 倍 token |
| DAG 执行器 | "拓扑分派器" | 按依赖顺序运行计划节点；每层并行 |
