# 工具使用与函数调用

> Toolformer（Schick 等人，2023）开创了自我监督工具注释。Berkeley 函数调用排行榜 V4（Patil 等人，2025）设定了 2026 年的基准：40% 智能体型、30% 多轮、10% 实时、10% 非实时、10% 幻觉检测。单轮调用已基本解决。记忆、动态决策和长视界工具链尚未解决。

**类型：** Build
**语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环），Phase 13 · 01（函数调用深入）
**时间：** ~60 分钟

## 学习目标

- 解释 Toolformer 的自我监督训练信号：仅当工具执行减少下一个 token 的损失时才保留工具注释。
- 说出 BFCL V4 的五个评估类别以及每个类别衡量什么。
- 实现一个标准库工具注册表，包含模式验证、参数强制类型转换和执行沙箱。
- 诊断 2026 年的三个未解决问题：长视界工具链、动态决策和记忆。

## 问题

早期的工具使用问的是：模型能预测正确的函数调用吗？现代的工具使用问的是：模型能跨 40 步链式使用工具吗？能使用记忆吗？能在部分可观测性下工作吗？能从工具故障中恢复吗？能不幻觉出不存在的工具吗？

Toolformer 建立了基线：模型可以通过自我监督学习何时调用工具。BFCL V4 定义了 2026 年的评估目标。两者之间的差距是生产智能体生存的空间。

## 概念

### Toolformer（Schick 等人，NeurIPS 2023）

思路：让模型用自己的预训练语料库注释候选 API 调用。对每个候选执行它。仅当包含工具结果减少下一个 token 的损失时保留注释。在过滤后的语料库上微调。

涵盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。自我监督信号纯粹是关于工具是否有助于预测文本——无需人工标签。

规模化结果：工具使用在规模化时涌现。较小的模型因工具注释而受损；较大的模型受益。这就是为什么 2026 年的前沿模型拥有强大的内置工具使用能力，而大多数 7B 模型需要显式的工具使用微调才能可靠。

### Berkeley 函数调用排行榜 V4（Patil 等人，ICML 2025）

BFCL 是 2026 年事实上的评估标准。V4 组成：

- **智能体型（40%）**——完整的智能体轨迹：记忆、多轮、动态决策。
- **多轮（30%）**——带工具链的交互式对话。
- **实时（10%）**——用户提交的真实提示（更难的分布）。
- **非实时（10%）**——合成测试用例。
- **幻觉检测（10%）**——检测何时不应调用任何工具。

V3 引入了基于状态的评估：在工具序列之后，检查 API 的实际状态（例如"文件是否已创建？"）而不是匹配工具调用的 AST。V4 增加了网络搜索、记忆和格式敏感性类别。

2026 年的关键发现：单轮函数调用已接近解决。失败集中在记忆（跨轮次携带上下文）、动态决策（基于先前结果选择工具）、长视界链（超过 20 步后漂移）和幻觉检测（当没有工具适合时拒绝调用）。

### 工具模式

每个提供商都有自己的模式。它们在细节上有所不同，但共享相同的形状：

```
name: string
description: string (它做什么，何时使用它)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述是承重的——模型读取它们来挑选正确的工具。糟糕的工具描述是选错工具失败的首要原因。

### 参数验证

不信任任何工具调用。验证：

1. **类型强制转换。** 模式说 int 时模型可能返回字符串 "5"。如果明确则强制转换；如果不明确则拒绝。
2. **枚举验证。** 如果模式说 `status in {"open", "closed"}` 而模型发出 `"in_progress"`，用描述性错误拒绝。
3. **必填字段。** 缺少必填字段 -> 立即返回错误观察给模型，而不是崩溃。
4. **格式验证。** 日期、电子邮件、URL——使用具体解析器验证，而不是正则表达式。

每个验证失败应返回结构化观察，以便模型可以用正确的形状重试。

### 并行工具调用

现代提供商支持单个助手轮次中的并行工具调用。循环：

1. 模型发出带有不同 `tool_use_id` 的 3 个工具调用。
2. 运行时执行它们（如果独立则并行）。
3. 每个结果通过 `tool_use_id` 关联作为 `tool_result` 块返回。

工程规则：将关联 ID 视为承重的。交换它们会导致工具到结果的路由错误。

### 沙箱

工具执行是沙箱边界。详见第 09 课。简而言之：每个工具应指定读/写面、网络访问、超时、内存上限。通用的 `run_shell(cmd)` 是危险信号；特定的 `git_status()` 更安全。

## 构建

`code/main.py` 实现一个生产形态的工具注册表：

- JSON Schema 子集验证器（仅标准库）。
- 带有描述、输入模式、超时和执行器的工具注册。
- 参数强制转换和枚举验证。
- 带有关联 ID 的并行工具分派。
- 作为结构化字符串的错误观察。

运行：

```
python3 code/main.py
```

轨迹显示一个小型智能体在一轮中调用三个工具，其中一个故意格式错误的调用被拒绝，并返回模型可以处理的描述性错误。

## 使用

每个提供商都有自己的工具模式——Anthropic、OpenAI、Gemini、Bedrock。如果需要多提供商支持，使用转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。BFCL 是参考基准——如果工具使用是产品的核心，在交付前针对你的智能体运行它。

## 交付

`outputs/skill-tool-registry.md` 为给定任务域生成工具目录、模式和注册表。包括描述质量检查（每个工具的描述是否告诉模型何时使用它？）。

## 练习

1. 添加一个"无操作"工具，让模型明确拒绝使用任何其他工具。在类似 BFCL 的幻觉测试上测量。
2. 为字符串转整数和字符串转浮点数实现参数强制转换。强制转换在哪里开始隐藏真正的 bug？
3. 添加每工具超时和断路器（连续 3 次失败后 60 秒内拒绝该工具）。这对模型恢复方式有什么影响？
4. 阅读 BFCL V4 描述。选择一个类别（例如"多轮"）并通过你的智能体运行 10 个示例提示。报告通过率。
5. 将标准库验证器移植到 Pydantic 或 Zod。Pydantic/Zod 捕获了玩具遗漏的什么？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 函数调用 | "工具使用" | 带有已验证模式的结构化输出工具调用 |
| Toolformer | "自我监督工具注释" | Schick 2023——保留其结果减少下一个 token 损失的工具调用 |
| BFCL | "Berkeley 函数调用排行榜" | 2026 年基准：40% 智能体型、30% 多轮、10% 实时、10% 非实时、10% 幻觉 |
| 工具模式 | "模型的函数签名" | 名称、描述、参数的 JSON Schema |
| tool_use_id | "关联 ID" | 将工具调用与其结果关联；对并行分派至关重要 |
| 幻觉检测 | "知道何时不调用" | V4 类别：当没有工具适合时拒绝调用 |
| 参数强制转换 | "字符串转整数修复" | 针对可预测的模式不匹配的狭窄修复；不明确时拒绝 |
| 沙箱 | "工具执行边界" | 每工具读/写面、网络、超时、内存上限 |
