# 带模式验证的工具注册中心

> 智能体无法验证的工具就是智能体无法调用的工具。在构建工具之前，先构建注册中心和模式检查器。

**类型:** Build
**语言:** Python
**前置要求:** Phase 13 课程 01-07、Phase 14 课程 01
**时间:** ~90 分钟

## 学习目标
- 持有一个类型化的工具名称 → 模式 → 处理程序注册中心，调度器可以一次查询然后信任。
- 实现一个 JSON Schema 2020-12 子集，覆盖 90% 工具调用实际使用的关键字。
- 返回精确的、json-pointer 形状的错误路径，以便模型可以在一个往返中自我纠正。
- 在没有显式覆盖的情况下拒绝重新注册，因为静默覆盖是生产工具目录漂移的方式。
- 保持验证器纯净（无 I/O、无时间、无全局变量），以便可以在重放日志上重新运行。

## 为什么注册中心在工具之前

2026 年的编码智能体拥有的已注册工具比模型在单个上下文窗口中能容纳的还要多。一个非平凡的框架会注册两百个工具，并在任何给定轮次中展示十个到四十个。注册中心是"存在哪些工具"、"它们的参数是什么形态"以及"调用哪个处理程序"的真相来源。一旦这三个答案确定下来，框架的其余部分就可以停止猜测了。

我们要避免的错误是发布没有模式的处理程序，或发布没有验证的模式。两者都很常见。两者都将下一层（第二十三课中的调度器）变成一个猜谜游戏，唯一的故障模式是来自处理程序的堆栈跟踪。

## 工具记录的样子

```text
ToolRecord
  name        : str          (唯一，小写字母数字和下划线段，以点分隔，例如 snake_case.segment.case)
  description : str          (一行，展示给模型)
  schema      : dict         (JSON Schema 2020-12 子集)
  handler     : Callable     (异步或同步，返回 Any)
  idempotent  : bool         (调度器使用此标志进行重试决策)
  timeout_ms  : int          (覆盖每工具调度器默认值)
```

模式是验证器唯一接触的字段。处理程序对验证器是不透明的。我们有意将它们分开。模式是数据。处理程序是代码。将它们混合会诱使你把验证逻辑放在处理程序内部，这正是我们要阻止的 bug。

## JSON Schema 2020-12 子集

完整的 2020-12 规范是一篇论文。我们需要八个关键字。

```text
type           string / number / integer / boolean / object / array / null
properties     属性名 → 模式的映射
required       属性名列表
enum           允许的原语值列表
minLength      整数，应用于字符串
maxLength      整数，应用于字符串
pattern        ECMA-262 兼容的正则表达式，应用于字符串
items          应用于每个数组元素的模式
```

这足以覆盖工具 API 实际需要的内容。我们不添加的关键字（oneOf、anyOf、allOf、$ref、条件式）在生产模式中是有效的，但会将验证器变成一个带循环的树遍历器。我们在构建注册中心，而不是 JSON Schema 引擎。

## Json pointer 错误路径

当验证失败时，验证器返回一个错误列表。每个错误携带一个指向输入的 json-pointer 路径。指针是一个斜杠前缀的属性名和数组索引序列。

```text
{"a": {"b": [1, 2, "x"]}}
                    ^
                    /a/b/2
```

模型读取错误路径比阅读句子更好。如果一个模式要求 `args.user.email` 而模型传入了一个整数，错误应该是 `/user/email` 加上 `expected_type: string`。模型在下一个调用中修复这个问题，而无需经过一轮自然语言。

## 注册和覆盖

`register(name, schema, handler, **opts)` 默认拒绝重新注册。调用者必须传递 `override=True` 才能替换。这是操作卫生规范。代码库的两个部分静默注册相同的工具名称是需要一周才能在生产中找到的那种 bug。

注册中心暴露三个读取方法。`get(name)` 返回记录或引发异常。`validate(name, args)` 返回 `Ok` 或错误列表。`names()` 按注册顺序返回工具名称。

## 验证器是什么和不是什么

它是一个对模式树的单次遍历，递归的。它是纯粹的。它不调用处理程序。它不强制类型转换（字符串 `"42"` 不会通过数字模式）。它不静默截断。

它不是安全边界。一个恶意的处理程序仍然可以在验证通过后行为不端。第二十三课的调度器添加了超时和沙箱层。注册中心添加了形态。

## 如何阅读代码

`code/main.py` 定义了 `ToolRegistry`、`ToolRecord`、`ValidationError` 和八个验证器函数。验证器根据 `schema["type"]` 进行分发（或将带有 `enum` 的模式视为无类型枚举检查）。每个类型验证器返回一个空列表或一个 `ValidationError` 列表。顶层遍历器连接错误并在下降时预置路径段。

`code/tests/test_registry.py` 涵盖了注册、覆盖、验证成功、带路径的验证失败以及子集中的每个关键字。
