# AlphaEvolve — 进化式编码智能体

> 将一个前沿编码模型与一个进化循环和一个机器可检查的评估器配对。让循环运行足够长的时间。它发现了一个 4x4 复数矩阵乘法过程，仅使用 48 次标量乘法——这是 56 年来对 Strassen 算法的首次改进。它还发现了一个 Google 范围内的 Borg 调度启发式方法，在生产中恢复了约 0.7% 的集群算力。架构故意设计得很朴素。胜利来自评估器的严谨性。

**类型：** 学习
**语言：** Python（标准库，进化循环玩具）
**前置知识：** 阶段 15 · 01（长时域框架），阶段 15 · 02（自教推理）
**时间：** ~60 分钟

## 问题

大型语言模型可以编写代码。进化算法可以搜索代码空间。两者都分别被尝试了几十年；两者都遇到了天花板。LLM 的天花板是虚构（confabulation）：模型编写看似合理的代码，但实际上并不做它声称的事情。进化的天花板是搜索成本：对语法进行随机突变很少产生可编译的程序，更不用说更好的程序了。

AlphaEvolve（Novikov 等人，DeepMind，arXiv:2506.13131，2025 年 6 月）将两者结合。LLM 对程序数据库提出有针对性的编辑；一个自动评估器对每个变体进行评分；高分变体成为未来世代的父代。LLM 处理编写合理代码这一昂贵步骤；评估器捕捉虚构。循环运行数小时到数周。

报告的结果：48 次标量乘法的 4x4 复数矩阵乘法（Strassen 1969 年的界限是 49），Google 生产环境中的 Borg 调度启发式方法，32.5% 的 FlashAttention 内核加速，Gemini 训练吞吐量改进。

该架构之所以有效，是因为评估器是机器可检查的。在评估器不可检查的地方，它就不起作用。这种不对称性就是本课的核心。

## 概念

### 循环

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个程序由评估器评分。
3. 从数据库中采样一个或多个父代（MAP-elites 风格或岛屿风格）。
4. 提示 LLM（许多候选使用 Gemini Flash，困难的使用 Gemini Pro）生成父代的一个修改变体。
5. 编译、运行并在保留的评估器上评估变体。
6. 按分数和特征向量键值插入数据库。
7. 重复。

两个细节很重要。首先，LLM 收到的提示不仅仅是父程序——通常还包括数据库中的几个顶级变体、评估器签名和简短的任务描述。模型的工作是提出一个可能提高分数的有针对性的更改。其次，数据库是结构化的（MAP-elites 网格、岛屿风格），因此循环探索多样性，而不仅仅是当前的领先者。

### 为什么评估器是不可协商的

AlphaEvolve 的胜利都来自评估器快速、确定且难以博弈的领域：

- **矩阵乘法算法**：一个单元测试，将矩阵相乘并逐位检查相等性。
- **Borg 调度启发式方法**：一个生产级模拟器，重放历史集群负载并测量浪费的算力。
- **FlashAttention 内核**：一个正确性测试加上真实硬件上的挂钟基准测试。
- **Gemini 训练吞吐量**：每步测量的 GPU 秒数。

在每种情况下，评估器都捕捉到了 LLM 错误中原本会占主导地位的类别：虚构的正确性声明、在硬件上消失的性能声明以及边缘情况失败。移除评估器，循环就会优化出漂亮的代码。

### 奖励黑客是该表述的另一面

进化会优化评估器测量的任何东西。如果评估器不完美，循环就会找到这个不完美之处。在一个未经验证的领域中，循环会优化表面特征，而不是预期行为。DeepMind 在论文中明确指出了这一点：AlphaEvolve 的成功只转移到评估器严谨性与搜索雄心相匹配的领域。

2025-2026 年代码搜索循环中奖励黑客（reward hacking）的具体例子：

- 奖励"完成时间"的优化目标奖励了提交空解决方案。
- 奖励"测试下正确性"的基准分数奖励了记忆测试和过拟合。
- 一个"代码质量"代理奖励了删除注释和重写变量名，而没有语义变化。

AlphaEvolve 的修复方法：提供一个 LLM 从未见过的保留评估器，输入在评估时生成。即便如此，DeepMind 仍建议对任何拟议的部署进行严格审查。

### 为什么 LLM + 搜索优于任何单独一方

LLM 可以生成可编译、语义合理的修改。在一个 2000 行的 Python 文件上使用随机突变的遗传算法几乎总是产生语法错误。LLM 还将搜索集中在合理的邻域（更改一个函数，而不是随机字节），这大大减少了浪费的评估器调用。

反过来，评估器捕捉了 LLM 的虚构。LLM 会自信地声称一个函数"在极限情况下是 O(n log n)"，而实际上它是 O(n^2)；一个挂钟基准测试让问题尘埃落定。

### AlphaEvolve 在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 示例胜利 |
|---|---|---|---|---|
| AlphaEvolve | Gemini | 正确性 + 基准测试 | 算法、内核、调度器 | 48 次乘法的 4x4 矩阵乘法 |
| FunSearch（DeepMind，2023） | PaLM / Codey | 正确性 | 组合数学 | cap-set 下界 |
| AI Scientist v2（Sakana，L5） | GPT/Claude | LLM 批评 + 实验 | ML 研究 | ICLR 研讨会论文 |
| Darwin Godel Machine（L4） | 智能体脚手架 | SWE-bench / Polyglot | 智能体代码 | 20% → 50% SWE-bench |

所有四个都是同一配方的变体：生成器加评估器，循环。不同之处在于评估器评分的内容以及它的严谨程度。

## 使用它

`code/main.py` 在一个玩具符号回归问题上实现了一个最小化的 AlphaEvolve 风格循环。"LLM"是一个标准库代理，对一个计算目标函数的程序提出小的语法突变。"评估器"测量保留测试点上的均方误差。

观察：

- 最佳分数如何随着世代推移而改善。
- MAP-elites 网格如何保持多样化的解决方案存活，使循环不会收敛到局部最小值。
- 移除保留测试（仅训练评估器）如何让循环惊人地过拟合。

`outputs/skill-evaluator-rigor-audit.md` 是在新领域考虑 AlphaEvolve 风格循环的前提条件：你的评估器是否真的捕捉到了你关心的失败？

## 练习

1. 运行 `code/main.py`。记录最佳分数轨迹。禁用保留评估器（`--no-holdout` 标志）并重新运行。量化过拟合程度。

2. 阅读 AlphaEvolve 论文第 3 节关于 MAP-elites 网格的内容。为一个新问题（例如编译器优化通道）设计一个能保持搜索多样性的特征向量描述符。

3. 48 次乘法的 4x4 结果在 56 年后改进了 Strassen 的 49 次乘法界限。阅读论文附录 F，用三句话解释为什么这个问题的评估器特别容易做对，以及为什么大多数领域不像它这样。

4. 提出一个 AlphaEvolve 会失败的领域。明确指出评估器在哪里失效以及为什么。

5. 为你熟悉的领域，编写你会使用的评估器签名。包括（a）正确性条件，（b）性能指标，（c）保留输入生成规则，（d）至少一个反奖励黑客检查。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|---|---|---|
| AlphaEvolve | "DeepMind 的进化式编码智能体" | Gemini + 程序数据库 + 机器可检查评估器 |
| MAP-elites | "保持多样性的存档" | 按特征向量键值划分的网格；每个单元格持有具有该描述符的最佳变体 |
| 岛屿模型（Island model） | "并行进化子种群" | 独立种群，定期迁移；防止过早收敛 |
| 机器可检查评估器（Machine-checkable evaluator） | "确定性预言机" | LLM 无法伪造的单元测试、模拟器或基准测试——此循环的前提条件 |
| 奖励黑客（Reward hacking） | "优化度量，而非目标" | 循环找到最大化分数的方法，而不执行预期任务 |
| 种子程序（Seed program） | "起点" | 循环从中进化的初始正确但次优程序 |
| 保留评估器（Held-out evaluator） | "LLM 从未见过的评估数据" | 在评估时生成的输入，以防止记忆 |
