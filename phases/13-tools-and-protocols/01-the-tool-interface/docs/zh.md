# 工具接口——为什么智能体需要结构化 I/O

> 语言模型产生 token。程序执行动作。两者之间的差距就是工具接口：一种让模型请求动作、主机执行动作的契约。每一套 2026 年的技术栈——OpenAI、Anthropic 和 Gemini 的函数调用；MCP 的 `tools/call`；A2A 的任务分块——都是同一四步循环的不同编码形式。本课程命名这个循环，并展示运行它的最小机制。

**类型：** 学习
**语言：** Python（标准库，无 LLM）
**前置要求：** Phase 11（LLM 补全 API）
**时间：** ~45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法自行对现实世界执行动作。
- 画出四步工具调用循环（描述 → 决策 → 执行 → 观察）并命名每一步的归属方。
- 将工具描述写为三个部分：名称、JSON Schema 输入和确定性执行函数。
- 区分纯工具和副作用工具，并说明这种区分为何对安全性至关重要。

## 问题

LLM 输出的是下一个 token 的概率分布。这就是完整的输出面。如果你问一个聊天模型"班加罗尔现在天气如何"，它可以写一个看似合理的句子，但无法接入天气 API。这个句子可能碰巧正确，也可能已经过时三天。

弥合这一差距是工具接口的目的。主机程序——你的智能体运行时、Claude Desktop、ChatGPT、Cursor 或自定义脚本——向模型通告一个可调用工具的列表。模型在决定需要执行动作时，输出一个结构化载荷，指明工具名称及其参数。主机解析该载荷，实际运行工具，并将结果反馈回去。循环持续进行，直到模型判定不再需要调用。

这个契约的第一个版本于 2023 年 6 月作为 OpenAI 的"functions"参数发布。Anthropic 在 Claude 2.1 中跟随引入了 `tool_use` 块。Gemini 在几个月后添加了 `functionDeclarations`。现在每家提供商都暴露相同的结构：输入是一个 JSON Schema 类型的工具列表，输出是一个 JSON 载荷的工具调用。模型上下文协议（2024 年 11 月）推广了这个契约，使一个工具注册中心可以为任何模型服务。A2A（2026 年 4 月，v1.0）将相同的原语用于智能体到智能体的委托。

四步循环是所有这些实现之下的不变式。Phase 13 中的其他内容都是它的延伸。

## 概念

### 第一步：描述

主机用三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。`get_weather`，而不是"天气的东西"。
- **描述。** 一段自然语言的简短说明。"当用户询问某个城市的当前天气状况时使用。不应用于历史数据。"
- **输入 schema。** 一个 JSON Schema 对象（draft 2020-12），描述工具的参数。

模型接收这个列表。现代提供商使用提供商特定的模板将这些声明序列化到系统提示中，因此你作为调用者只需要处理结构化形式。

### 第二步：决策

给定用户的消息和可用的工具，模型选择三种行为之一。

1. **直接回答** 文本。不调用工具。
2. **调用一个或多个工具。** 输出结构化的调用对象。在 `parallel_tool_calls: true`（OpenAI 和 Gemini 默认，Anthropic 可选）下，模型可以在一次回复中输出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生一个类型化的 `refusal` 块，而不是调用。

工具调用载荷包含三个稳定字段：调用 `id`、工具 `name` 和 JSON `arguments` 对象。id 的存在是为了让主机能够将后续结果与特定的调用关联起来，这在并行调用乱序返回时很重要。

### 第三步：执行

主机接收调用，根据声明的 schema 验证参数，并运行执行器。参数无效意味着模型幻觉了一个字段或使用了错误的类型——这是较弱模型上非常常见的失败模式。生产主机在参数无效时会做三件事之一：快速失败并将错误返回给模型，使用受限解析器修复 JSON，或使用包含验证错误的提示重试模型。

执行器本身是普通代码：Python、TypeScript、shell 命令、数据库查询。它产生一个结果，通常是字符串，但也可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

主机将工具结果附加到对话中（作为带有匹配 `id` 的 `tool` 角色消息），并重新调用模型。模型现在有了上下文中的工具输出，可以生成最终答案或请求更多调用。循环继续，直到模型停止发出调用或主机达到迭代次数的安全限制。

### 信任分离

工具根据安全性可分为两种类型：

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地推测性调用。
- **后果性工具。** 改变状态、花钱、触及用户数据。`send_email`、`delete_file`、`execute_trade`。必须加门控。

Meta 2026 年的智能体安全"双重规则"指出，单次调用最多可以组合以下三项中的两项：不受信任的输入、敏感数据、后果性动作。工具接口是你执行该规则的地方——通过拒绝调用、要求用户确认或升级权限。参见 Phase 13 · 15 获取完整安全章节，以及 Phase 14 · 09 获取智能体级别权限策略。

### 循环的归属

| 上下文 | 谁描述 | 谁决策 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单次函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM 通过 MCP 客户端 | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方智能体 | 被调用智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无处不在，相同的四步。列名在变；结构不变。

### 为什么不直接提示模型输出 JSON？

"要求模型以 JSON 回复"是函数调用之前的模式。在前沿模型上失败率约为 5% 到 15%，在较小的模型上则更高。失败模式包括：缺少花括号、尾随逗号、幻觉字段和错误的类型。然后你需要 JSON 修复、重试或受限解码器。

原生函数调用更好的三个原因。首先，提供商端到端地训练模型执行确切的调用形状，因此在严格模式下有效 JSON 率上升到 98% 到 99%。其次，调用载荷位于自己的协议槽中，而不是自由文本内——因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商通过受限解码（OpenAI 的 strict 模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）强制 schema 合规。输出保证通过验证。

Phase 13 · 02 并排讲解了三个提供商的 API。Phase 13 · 04 深入结构化输出。

### 断路器

循环在模型停止发出调用或主机达到最大轮次时终止。生产主机通常设置为 5 到 20 轮。超过这个数目，你几乎肯定陷入了一个模型无法退出的循环。Claude Code 默认为 20；OpenAI Assistants 为 10；Cursor 的智能体模式为 25。

另一种选择——无界循环——每六个月以"智能体一夜之间在 API 调用上花了 400 美元"的事后总结形式出现。没有限制就不要发布。

Phase 14 · 12 深入介绍错误恢复和自愈；Phase 17 涵盖生产速率限制。

## 使用它

`code/main.py` 在没有 LLM 的情况下运行四步循环。一个假的"决策器"函数通过模式匹配用户消息来模拟模型；执行器、schema 验证器和观察步骤工具是真实的。运行它以查看带有可打印中间状态的完整请求/响应编排，然后在后面的课程中将假的决策器替换为任何真实的提供商。

关注点：

- 工具注册中心为每个工具保存三个字段：名称、描述、schema 和执行器引用。
- 验证器是一个最小的 JSON Schema 子集（类型、required、enum、min/max），仅使用标准库编写。Phase 13 · 04 提供一个更完整的版本。
- 循环将迭代次数限制在 5 次。生产智能体确实需要这种断路器。

## 交付物

本课程产生 `outputs/skill-tool-interface-reviewer.md`。给定一个工具定义草案（名称 + 描述 + schema + 执行器大纲），该技能审计其循环适配性：名称是否机器稳定、描述是否完整的使用说明、schema 是否正确使用 JSON Schema 2020-12、纯工具与后果性工具分类是否明确。

## 练习

1. 在 `code/main.py` 中添加第四个工具，名为 `get_stock_price(ticker)`。将其描述写为"当用户按代码询问当前股票价格时使用。不应用于历史价格或市场摘要。"运行测试工具并确认假的决策器将提及代码的查询路由到新工具。

2. 破坏 schema 验证器。传入一个缺少必需字段的 `arguments` 对象的调用，并确认主机在执行前拒绝了它。然后传入一个包含额外未知字段的调用。决定：主机应该拒绝还是忽略？用安全论证证明你的选择。

3. 将测试工具中的每个工具分类为纯工具或后果性工具。向需要它的注册中心条目添加 `consequential: true` 标志，并修改循环，使其在选择后果性工具时打印一个"将向用户确认"行。这是每个生产主机所需的确认门控的形状。

4. 在纸上画出四步循环，并填入上面提供者列对应的你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义技术栈）。与 Phase 13 · 06 中 MCP 特定的变体进行交叉引用。

5. 从头到尾阅读 OpenAI 的函数调用指南。确定位于请求中但不在本文所述的四步循环中的一个字段。解释它增加了什么以及为什么它是便利性的而不是必不可少的。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 工具 | "模型可以调用的东西" | 名称 + JSON Schema 类型输入 + 执行器函数的三元组 |
| 函数调用 | "原生工具使用" | 提供商级别 API 支持，用于输出结构化工具调用而不是散文 |
| 工具调用 | "模型的动作请求" | 模型输出的带有 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果 | "工具返回的内容" | 执行器的输出，包装在带有匹配 id 的 `tool` 角色消息中 |
| 并行工具调用 | "一次多个调用" | 一次模型回复中的多个调用对象，独立且可通过 id 排序 |
| 严格模式 | "保证的 JSON" | 受限解码，强制模型输出符合声明的 schema |
| 纯工具 | "只读工具" | 无副作用；可以安全地重新运行 |
| 后果性工具 | "动作工具" | 改变外部状态；需要门控、审计或用户确认 |
| 四步循环 | "工具调用周期" | 描述 → 决策 → 执行 → 观察 |
| 主机 | "智能体运行时" | 持有工具注册中心、调用模型和运行执行器的程序 |
